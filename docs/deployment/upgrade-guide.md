# Upgrade Guide

Procedures for upgrading Cilex Vision components, including application services, database schema, inference models, and infrastructure.

## Pre-Upgrade Checklist

Before any upgrade:

- [ ] **Backup**: Run backups per `docs/runbooks/backup-restore.md`
  ```bash
  # TimescaleDB
  pg_dump -h timescaledb-1 -U cilex -d vidanalytics -Fc -f backup-pre-upgrade.dump

  # MinIO (critical buckets)
  mc mirror local/event-clips /backup/minio/event-clips/
  ```
- [ ] **Maintenance window**: Schedule with operations team; notify stakeholders
- [ ] **Health check**: Confirm current deployment is healthy
  ```bash
  bash scripts/deploy/health-check-all.sh --inventory infra/ansible/inventory/mysite.yml
  ```
- [ ] **Git state**: Ensure current deployment matches a known commit
  ```bash
  git log --oneline -1   # Record the current commit hash
  git status             # Should be clean
  ```
- [ ] **Document current state**: Save the running container versions
  ```bash
  docker ps --format "{{.Names}}\t{{.Image}}" > pre-upgrade-versions.txt
  ```

## Version Compatibility

| Component | Upgrade Path | Notes |
|-----------|-------------|-------|
| Application services | Any version -> latest | Stateless; rolling restart safe |
| Database schema | Sequential Alembic migrations | Run `alembic upgrade head`; never skip migrations |
| Triton models | Shadow deploy -> cutover (ADR-005) | 24-48 hour comparison window |
| Kafka | Minor version in-place | Rolling restart with `controlled.shutdown.enable=true` |
| TimescaleDB | Follow official upgrade docs | Extension upgrade may require `ALTER EXTENSION` |
| Protobuf schemas | Backward-compatible only | Enforced by `buf breaking` in CI |

## Application Service Upgrades

Application services are stateless containers. Upgrade by rebuilding and restarting.

### Rolling Upgrade (Zero Downtime)

For services with multiple instances (e.g., query-api on app-1 and app-2):

```bash
# 1. Pull latest code
git pull origin main

# 2. Rebuild images
cd infra/ansible
ansible-playbook -i inventory/mysite.yml playbooks/deploy-services.yml \
    --limit app-1

# 3. Verify app-1 is healthy
curl -s http://app-1:8000/health

# 4. Upgrade app-2
ansible-playbook -i inventory/mysite.yml playbooks/deploy-services.yml \
    --limit app-2

# 5. Verify app-2 is healthy
curl -s http://app-2:8000/health
```

### Single-Instance Upgrade (Brief Downtime)

For services with one instance:

```bash
# Pull latest code and rebuild
git pull origin main
cd infra/ansible
ansible-playbook -i inventory/mysite.yml playbooks/deploy-services.yml
```

### Pilot Upgrade

```bash
cd cilex-vision
git pull origin main

# Rebuild and restart all services
cd infra
docker compose -f docker-compose.pilot.yml build
docker compose -f docker-compose.pilot.yml up -d
```

## Database Schema Migration

All schema changes are managed through Alembic migrations in `services/db/alembic/`.

### Running Migrations

```bash
# From operator workstation or service node with DB access
cd services/db

# Check current migration state
DATABASE_URL="postgresql+asyncpg://cilex:<password>@timescaledb-1:5432/vidanalytics" \
    alembic current

# Review pending migrations
DATABASE_URL="postgresql+asyncpg://cilex:<password>@timescaledb-1:5432/vidanalytics" \
    alembic history --verbose

# Apply all pending migrations
DATABASE_URL="postgresql+asyncpg://cilex:<password>@timescaledb-1:5432/vidanalytics" \
    alembic upgrade head
```

### Migration Safety Notes

- Migrations are **hand-crafted** (not autogenerated) because TimescaleDB operations require raw SQL
- TimescaleDB operations (hypertable creation, retention/compression policies) use `op.execute()`
- Always run migrations during a maintenance window for large schema changes
- Test migrations on a staging database first if available:
  ```bash
  # Restore backup to staging
  pg_restore -h staging-db -U cilex -d vidanalytics_staging backup-pre-upgrade.dump
  # Run migration on staging
  DATABASE_URL="...staging..." alembic upgrade head
  ```

### Migration Rollback

```bash
# Roll back one migration
DATABASE_URL="postgresql+asyncpg://cilex:<password>@timescaledb-1:5432/vidanalytics" \
    alembic downgrade -1

# Roll back to a specific revision
DATABASE_URL="postgresql+asyncpg://cilex:<password>@timescaledb-1:5432/vidanalytics" \
    alembic downgrade <revision_id>
```

**Warning:** Downgrade scripts for TimescaleDB hypertable operations may not be fully reversible (e.g., dropping a hypertable loses data). Always have a database backup before migrating.

## Model Upgrades

Model upgrades follow the shadow deployment lifecycle defined in ADR-005 and `docs/runbooks/model-rollout-sop.md`.

### Shadow Deployment Process

```
1. Offline qualification (metrics meet minimum thresholds)
      |
      v
2. Shadow deployment (load new model alongside current)
      |
      v
3. 24-48 hour comparison window (shadow vs production)
      |
      v
4. Cutover (if metrics pass) or rollback (if regression detected)
      |
      v
5. Post-cutover watch (monitor for 24 hours, rollback trigger: >15% regression)
```

### Step-by-Step Model Upgrade

```bash
# 1. Deploy shadow model
python scripts/shadow/deploy_shadow_model.py \
    --model-name yolov8l \
    --version 2 \
    --model-path /path/to/new/model.plan \
    --triton-url http://triton-1:8000

# 2. Start shadow inference worker
python scripts/shadow/shadow_inference_worker.py \
    --kafka-bootstrap kafka-1:9093 \
    --triton-url triton-1:8001 \
    --model-version 2 \
    --metrics-port 9095

# 3. Wait 24-48 hours, then compare
python scripts/shadow/compare_shadow.py \
    --prometheus-url http://monitoring-1:9090 \
    --window-hours 24 \
    --output-file shadow-comparison-report.md

# 4a. If passed: cutover
#     Update service config to use new model version
#     Unload old model version

# 4b. If failed: rollback
python scripts/shadow/deploy_shadow_model.py \
    --model-name yolov8l \
    --version 2 \
    --unload \
    --triton-url http://triton-1:8000
```

### Pilot Model Upgrade (CPU -> GPU)

To upgrade the pilot from YOLOv8n (CPU) to YOLOv8-L (GPU):

1. Install NVIDIA Container Toolkit on the host
2. Export the retrained model to TensorRT:
   ```bash
   python scripts/training/export.py --format tensorrt --precision fp16
   ```
3. Place the engine at `infra/triton/model-repo/yolov8l/1/model.plan`
4. Update `config.pbtxt` for GPU inference
5. Add GPU resources to the Triton service in docker-compose
6. Restart Triton:
   ```bash
   docker restart pilot-triton pilot-inference-worker
   ```

See `docs/deployment-guide-pilot.md` section "Upgrading to GPU" for details.

### Re-ID Model Upgrade (ADR-008)

When upgrading the OSNet Re-ID model:

- **FAISS index must be flushed** -- embeddings from different model versions cannot be compared
- **All trackers reset** -- ~30 second matching blackout is expected
- The MTMC service handles this automatically when it detects a model version change

## Kafka Upgrades

### Minor Version Rolling Upgrade

```bash
# 1. Set controlled shutdown (already default in production)
# In server.properties: controlled.shutdown.enable=true

# 2. Upgrade one broker at a time
# On kafka-1:
docker pull bitnami/kafka:<new-version>
docker stop kafka-1
docker run ... bitnami/kafka:<new-version>

# Wait for kafka-1 to rejoin the cluster and ISR to be in sync
# Verify:
kafka-metadata.sh --snapshot /var/kafka-logs/__cluster_metadata-0/00000000000000000000.log \
    --cluster-id <cluster-id>

# 3. Repeat for kafka-2, then kafka-3
```

### Topic Schema Migration

If a Protobuf schema changes (backward-compatible only):

1. Update the `.proto` files in `proto/vidanalytics/v1/`
2. Run `buf lint` and `buf breaking` to verify compatibility
3. Register the new schema version in Schema Registry
4. Deploy updated producer services first, then consumers

## Monitoring Upgrades

### Prometheus

```bash
# Pull new version and restart
ansible-playbook -i inventory/mysite.yml playbooks/deploy-monitoring.yml
```

Prometheus retains 7 days of data by default. After upgrade, verify scrape targets:

```bash
curl -s http://monitoring-1:9090/api/v1/targets | python3 -m json.tool | grep health
```

### Grafana

Dashboards are provisioned from `infra/grafana/dashboards/`. After upgrade:

1. Verify dashboards load correctly
2. Check datasource connectivity (Prometheus)
3. Re-import any manually-created dashboards from backup

## Rollback Procedure

If an upgrade causes issues:

### Application Service Rollback

```bash
# 1. Check out the previous known-good commit
git checkout <previous-commit-hash>

# 2. Rebuild and deploy
cd infra/ansible
ansible-playbook -i inventory/mysite.yml playbooks/deploy-services.yml
```

### Database Rollback

```bash
# 1. Roll back migration
cd services/db
DATABASE_URL="..." alembic downgrade -1

# 2. If migration rollback is not possible, restore from backup
pg_restore -h timescaledb-1 -U cilex -d vidanalytics -c backup-pre-upgrade.dump
```

### Model Rollback

```bash
# Unload the new model version
python scripts/shadow/deploy_shadow_model.py \
    --model-name yolov8l \
    --version 2 \
    --unload \
    --triton-url http://triton-1:8000

# The previous version remains loaded and active
```

### Full Rollback (Last Resort)

If multiple components need rollback:

1. Stop all application services
2. Restore database from backup
3. Check out the previous commit
4. Re-deploy everything:
   ```bash
   cd infra/ansible
   ansible-playbook -i inventory/mysite.yml playbooks/deploy-multi-node.yml
   ```

## Post-Upgrade Verification

After any upgrade:

```bash
# 1. Run health checks
bash scripts/deploy/health-check-all.sh --inventory infra/ansible/inventory/mysite.yml

# 2. Check Kafka consumer lag (should return to zero within minutes)
kafka-consumer-groups.sh --bootstrap-server kafka-1:9093 \
    --describe --all-groups

# 3. Verify data flow (detections appearing in DB)
psql -h timescaledb-1 -U cilex -d vidanalytics \
    -c "SELECT COUNT(*) FROM detections WHERE time > NOW() - INTERVAL '5 minutes'"

# 4. Check Grafana dashboards for anomalies
# - Stream Health: all cameras connected
# - Inference Performance: latency within expected range
# - Bus Health: consumer lag near zero
# - Storage: no unexpected growth

# 5. Monitor for 1 hour before closing the maintenance window
```
