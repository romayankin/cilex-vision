# P4-V01: Zone Sharding for Large Sites

You are working in /repo/services/mtmc-service/. Extend the MTMC service for zone-based sharding.

## Context

For sites with 50+ cameras, full pairwise MTMC matching becomes computationally infeasible. Zone sharding partitions cameras into zones with full matching within zones and lighter cross-zone matching at boundaries.

## Reading Instructions

Before writing any code, read:
- .agents/handoff/P2-V02.md — MTMC service architecture, FAISS index, checkpoint
- .agents/handoff/P0-D05.md — topology graph, zone_id field on cameras
- .agents/handoff/P3-V04.md — adaptive transit times (once available)
- services/mtmc-service/ — current MTMC implementation
- CONVENTIONS.md

## Implementation

1. Add zone_id to topology graph cameras and edges
2. When zone config is present, shard MTMC by zone_id instead of site_id — each MTMC instance owns one zone
3. Build cross-zone-associator.py: a lighter service consuming only zone-boundary track closures, running matching across adjacent zones with lower frequency
4. Same data model: local_track → zone_global_track → site_global_link
5. Add zone boundary detection: when a track exits near a zone-boundary camera, publish to cross-zone topic
6. Tests: synthetic 100-camera multi-zone dataset with known cross-zone identities

## Constraints

- Must be backward-compatible: single-zone sites work without zone config
- Do NOT break existing MTMC tests
- Must pass `ruff check` and `mypy --strict`
