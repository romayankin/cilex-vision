# P3-X01: Deployment Guide — Handoff

## What was built

Six customer-facing Markdown documents in `docs/deployment/`:

1. **`README.md`** — Overview page with deployment scenarios (pilot/small/medium/large), prerequisites table, quick-start commands for pilot and production, deployment order, links to all sub-docs and operational runbooks, ASCII architecture diagram.

2. **`hardware-requirements.md`** — Per-scenario hardware specs:
   - Pilot: 1 host, 12 threads, 16 GB RAM, no GPU, with container memory budget breakdown
   - Small: 1 GPU host + edge gateways, with GPU memory budget (410 MB for 3 models)
   - Medium: 11+ core nodes (3 Kafka, 1 TSDB, 1 MinIO, 2 GPU, 2 service, 1 monitoring, 1 MLflow) + N edge gateways
   - Large: horizontal scaling guidance with formulas for when to add GPU nodes
   - Disk IOPS guidance per service, network bandwidth per camera (5 Mbps planning estimate), edge-to-core bandwidth after filtering (~250 Kbps/camera)

3. **`installation-guide.md`** — Two deployment paths:
   - Pilot: 8-step guide referencing existing `setup-pilot.sh` with camera config, credentials, verification
   - Production: 10-step guide covering inventory, PKI bootstrap, Terraform (optional), Ansible `deploy-multi-node.yml`, Triton model loading (EXPLICIT mode), DB migrations, health check, camera onboarding, dashboard access, post-install verification

4. **`network-guide.md`** — Network architecture with 5-zone diagram, camera VLAN setup with generic switch config example, firewall rules tables (edge/core/operator zones), full service port reference (18 services), mTLS certificate enrollment flow (5-step diagram), DNS requirements with example entries, Chrony NTP configuration, bandwidth planning tables by scenario.

5. **`upgrade-guide.md`** — Pre-upgrade checklist, version compatibility matrix, rolling upgrade procedures for stateless services, Alembic DB migration with rollback, model shadow deployment lifecycle (ADR-005), Kafka rolling upgrade, Protobuf schema migration, rollback procedures (per-component and full), post-upgrade verification checklist.

6. **`troubleshooting.md`** — 18 issue entries across 6 categories:
   - Installation: Docker, Ansible, Terraform, setup script failures
   - Cameras: RTSP unreachable, decode errors, no motion detected
   - Inference: Triton not ready, VRAM exhaustion, high latency
   - Data flow: Kafka lag, bridge spool fill, bulk collector stalls
   - Storage: TimescaleDB disk full, MinIO full, FAISS checkpoint failures
   - Networking: mTLS handshake, clock drift, NATS publish latency
   - Quick reference diagnostic commands section

## Key decisions

| Decision | Rationale |
|----------|-----------|
| Two deployment paths (pilot vs production) | Pilot uses Docker Compose on one host; production uses Ansible + optional Terraform on multiple hosts. Different audiences and complexity levels. |
| Hardware formulas, not just tables | Included `FPS_demand = cameras x fps x duty x sample` formula so customers can calculate their own sizing for non-standard camera counts. |
| Generic switch config example | Camera VLAN config is vendor-specific. Provided Cisco-like pseudocode with clear comments so any network engineer can adapt. |
| Referenced existing runbooks | Did not duplicate content from the 6 existing runbooks. Linked to them for ongoing operations (incident response, scaling, backup, etc.). |
| Troubleshooting organized by symptom | Customer IT teams diagnose from symptoms, not from component names. Each entry starts with what they'll see, not what's broken. |
| Included pilot model limitations | Documented the COCO→taxonomy class mismatch in the installation guide so pilot users aren't surprised by unexpected detection behavior. |

## Patterns followed

- YAML frontmatter on every document (version, status, created_by, date) matching runbook pattern
- Specific commands with placeholders (not just descriptions)
- Tables for structured data, code blocks for commands
- Cross-references to existing docs rather than duplication
- Consistent section ordering: what → why → how → verify

## Gotchas

- The `bootstrap-site.sh` PKI script is referenced but its exact CLI flags may differ from what's documented here — verify against the actual script before customer handoff.
- The `health-check-all.sh` script requires Python + PyYAML on the operator machine when used with `--inventory`.
- Terraform modules were not `terraform validate`-checked (no Terraform CLI in the dev environment). AMI/image IDs are placeholders.
- The network guide's switch config is generic pseudocode, not vendor-tested. Customer network teams should adapt to their hardware.
- MinIO lifecycle policies for `raw-video`, `event-clips`, and `thumbnails` are defined in Ansible group_vars but the actual lifecycle artifacts are not yet committed to the repo (noted in privacy-framework.md).

## Open questions

1. **Customer branding**: Should the docs reference "Cilex Vision" in page headers, or should they be white-labeled?
2. **Air-gapped deployment**: The docs assume internet access for Docker pulls and NTP. Should an air-gapped installation path be documented?
3. **High availability**: The current docs describe single TimescaleDB and single MinIO. Should HA configurations (TimescaleDB patroni, distributed MinIO) be documented?
4. **Windows support**: All docs assume Ubuntu 24.04. Should Windows Server or other Linux distros be addressed?
5. **Cloud-specific guides**: The Terraform section is provider-agnostic. Should AWS/GCP-specific guides with actual resource types be provided?
