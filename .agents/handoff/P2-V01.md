# P2-V01 — Attribute Extraction Service

## What Was Built

A Python service at `services/attribute-service/` that consumes tracklets from Kafka, applies quality gates, detects IR/night mode, runs color classification via Triton, aggregates results over track lifetime, and stores final attributes in PostgreSQL.

### Files Created
- `main.py` — Kafka consumer (confluent-kafka), orchestrates the full pipeline: tracklet parse → class routing → frame fetch → crop → quality gate → white balance → classify → aggregate → flush to DB
- `quality_gate.py` — 6-check pipeline: bbox height, Laplacian sharpness, brightness range, IR/night-mode (HSV saturation), occlusion (frame-edge clipping), composite quality score
- `white_balance.py` — `cv2.xphoto.createSimpleWB()` correction, skipped for IR crops
- `classifier_client.py` — Triton gRPC client following detector_client.py pattern. Lazy import, sync _infer() via asyncio.to_thread(). Preprocessing: resize 224x224, BGR→RGB, normalize, HWC→CHW. Softmax post-processing with 0.30 confidence threshold.
- `aggregator.py` — Confidence-weighted voting per track per attribute type. Accumulates (color, confidence*quality) observations. Flush returns AggregatedAttribute per type.
- `publisher.py` — asyncpg writer: write_attribute, write_aggregated (executemany batch), buffered writes with periodic flush, get_detection_bbox for frame fetching.
- `config.py` — Pydantic Settings v2 with YAML loading and ATTR_ env prefix. All thresholds configurable.
- `metrics.py` — 6 Prometheus instruments: classified_total, quality_rejected_total, ir_skipped_total, classification_latency, tracks_flushed, db_write_latency.
- `mypy.ini` — strict mode with ignore_missing_imports for third-party libs.
- `gen_proto.sh`, `requirements.txt`, `Dockerfile` — standard patterns.
- `tests/test_quality_gate.py` — 8 tests: daytime pass, too_small, blurry, dark exposure, bright exposure, IR mode, occlusion, quality score range.
- `tests/test_aggregator.py` — 10 tests: single obs, accumulation, mixed voting, no obs, flush, cleanup, multi-type, count, zero weight, nonexistent.

## Key Decisions

1. **Person upper/lower split**: For person tracks, the full-body bbox crop is split vertically at the midpoint — upper half for `person_upper_color`, lower half for `person_lower_color`. Simple and effective for standing poses.

2. **Frame fetching via detection lookup**: The Tracklet proto doesn't carry frame_uri. We query the `detections` hypertable for the latest detection's bbox and frame_seq, then construct the MinIO path as `{camera_id}/{frame_seq}.jpg`. This avoids consuming a second Kafka topic.

3. **Observation capping**: Tracks are capped at `max_observations_per_track` (default 20) to bound memory. Early flush happens when the cap is reached, not only on TERMINATED state.

4. **Buffered DB writes**: Attributes are buffered and flushed either when buffer reaches `flush_batch_size` (default 50) or every `flush_interval_s` (default 1.0s). Uses `executemany` for batching.

5. **IR detection before brightness**: The quality gate checks IR (HSV saturation < 15) after brightness. IR crops are a distinct skip reason, not conflated with exposure rejection. Metric: `attr_ir_skipped_total`.

6. **Occlusion via visible area ratio**: Clipping ratio = `1 - visible_area / bbox_area` using min/max clamping to [0, 1]. More accurate than edge-delta heuristics.

## Patterns Established

- Quality gate as a standalone module with `QualityResult` dataclass (composable, testable)
- White balance as a thin cv2.xphoto wrapper (skip-on-IR pattern)
- Triton color classifier client with softmax + threshold → "unknown" fallback
- Confidence-weighted voting aggregator for per-track attribute fusion
- Person upper/lower body split by vertical midpoint of crop

## Gotchas

1. **Frame URI convention**: MinIO path is `{camera_id}/{frame_seq}.jpg`. If the decode service uses a different naming scheme, the frame fetch will fail silently (returns None, skips the tracklet).

2. **cv2.xphoto availability**: Requires `opencv-contrib-python-headless`, not the regular `opencv-python-headless`. The Dockerfile installs the contrib variant.

3. **Quality gate ordering**: Checks run in order — a blurry bright crop is rejected as "blurry", not "exposure". Only the first failing check is reported. This is by design for efficiency.

4. **Track ID as UUID string**: The publisher passes `local_track_id` as a string to asyncpg. The DB column is UUID type — asyncpg handles the conversion but the string must be a valid UUID.

5. **Flush on shutdown**: `shutdown()` calls `flush_buffer()` to persist any remaining buffered attributes before closing the pool.

## Open Questions

1. **Frame caching**: Currently each tracklet observation triggers a fresh MinIO download. A per-camera LRU frame cache (keyed by frame_seq) would reduce MinIO load significantly.

2. **Batch Triton inference**: The classifier processes one crop at a time. With Triton's dynamic batching (100ms queue delay, max batch 32), grouping crops from the same poll batch would improve throughput.

3. **Attribute Kafka topic**: The spec doesn't publish classified attributes to a Kafka topic (only writes to DB). A future task may add a `attributes.classified` topic for downstream consumers.
