# Handoff: P0-E01 — Model Bake-Off Protocol

## What was built

- `docs/bake-off-protocol.md` — full bake-off protocol replacing the stub
- `scripts/bakeoff/run_detector_bakeoff.py` — detector harness covering ONNX -> TensorRT -> Triton -> eval -> MLflow
- `scripts/bakeoff/compare_bakeoff.py` — MLflow comparison tool emitting Markdown and matplotlib charts

## Key decisions

- The detector protocol is anchored to the existing Triton deployment envelope from `docs/triton-placement.md`: FP16 TensorRT, 640x640 RGB, batch <= 8, preferred batches 1/4/8, 50 ms queue delay.
- Detector scoring uses the prompt’s formula, but the throughput term is normalized against the pilot requirement of 40 FPS aggregate (`4 cameras x 10 FPS`) instead of against the best run in a single comparison. That makes `detector_score` stable across MLflow runs.
- The safe-default fallback is explicit in both the protocol and comparison logic:
  - detector: `YOLOv8-L`
  - tracker: `ByteTrack`
  - attribute: `ResNet-18`
- The detector harness uses a COCO-style manifest contract with an `is_night` image flag and derives `small_object_AP` from GT boxes with area `< 32^2` pixels.
- The harness is intentionally strict: missing eval data, missing ONNX exports, missing Triton reachability, or missing optional Python dependencies all fail fast with concise CLI errors.

## Patterns established

- Optional runtime dependencies (`numpy`, `Pillow`, `mlflow`, `matplotlib`) are imported lazily so `--help` works even on a thin environment.
- MLflow runs are tagged with:
  - `bakeoff.phase=detector`
  - `bakeoff.candidate=<name>`
  - `bakeoff.safe_default=true|false`
- Detector per-class AP metrics are logged as individual MLflow metrics using the prefix `detector_ap50_95_<class>` so `compare_bakeoff.py` can build a heatmap without needing raw artifacts.
- CLI tools catch exceptions at the boundary and return a short error string instead of a traceback for expected operational failures.

## Gotchas for the next agent

- `data/eval/` is still effectively empty in this repo. Running `scripts/bakeoff/run_detector_bakeoff.py` right now fails by design with:
  - `evaluation manifest not found: data/eval/detector/manifest.json`
- The default ONNX paths are placeholders under `artifacts/models/detector/`; those model exports do not exist yet.
- `run_detector_bakeoff.py` assumes a Triton model repository path that is already mounted into a running Triton instance. It writes `config.pbtxt` plus `model.plan` into that repository and then calls Triton’s repository API.
- The built-in RT-DETR parsing logic is intentionally flexible, but real exported output names and shapes still need to be checked against the chosen ONNX export once actual candidate artifacts exist.
- I validated Python syntax and CLI help locally:
  - `python3 -m py_compile scripts/bakeoff/run_detector_bakeoff.py scripts/bakeoff/compare_bakeoff.py`
  - `python3 scripts/bakeoff/run_detector_bakeoff.py --help`
  - `python3 scripts/bakeoff/compare_bakeoff.py --help`
  - `python3 scripts/bakeoff/run_detector_bakeoff.py` returns the expected missing-data error
