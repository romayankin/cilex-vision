# P4-X02 — Architecture Reference

## What I built

- `docs/architecture/README.md`
  - overview of the platform scope, technology stack, and document map
  - high-level Mermaid diagram from edge ingest through search and investigation
- `docs/architecture/system-context.md`
  - **C4 Level 1** system-context view with operators, cameras, CVAT, and MLflow as external actors
- `docs/architecture/container-diagram.md`
  - **C4 Level 2** runtime container view covering the core services, storage, inference, message buses, and UI layer
  - explicitly documents the Kafka topic inventory and where current runtime wiring differs from the broader topic catalog
- `docs/architecture/data-flow.md`
  - end-to-end sequence diagram from RTSP ingest through MinIO, Kafka, inference, events, persistence, and Query API retrieval
  - includes the "no bytes on Kafka" rule and latency-budget framing
- `docs/architecture/security-architecture.md`
  - trust-zone diagram and narrative for PKI, transport security, RBAC, camera scoping, signed URLs, and audit logging
- `docs/architecture/deployment-architecture.md`
  - deployment reference for single-node pilot, multi-node production, and multi-site topologies
  - includes a network-zones diagram and links to hardware guidance
- `docs/architecture/technology-decisions.md`
  - summary of ADR-001/002/003 with rationale, design principles, and current implementation-drift notes

## Key decisions

- **Documented the implementation that exists, not the idealized target**
  - The prompt asked for architecture docs that are accurate against the current repository.
  - I treated code, config, ADRs, deployment docs, and recent handoffs as the source of truth and called out places where runtime wiring differs from the broader architectural contract.

- **Used C4 terminology, but kept the diagrams GitHub-renderable**
  - All diagrams are Mermaid-based so they render directly in Markdown viewers used by engineers and evaluators.
  - I avoided bespoke syntax or non-portable extensions.

- **Made implementation drift explicit**
  - `topology` exists as a service/domain, but its router is currently mounted inside `query-api`.
  - `attribute-service` currently consumes `tracklets.local` and writes directly to PostgreSQL instead of using an active `attributes.jobs` lane.
  - `event-engine` both publishes `events.raw` and persists events directly.
  - The archive/transcode topics are present, but a dedicated transcode worker is not part of the active 13-service inventory.

## Patterns followed

- one document per architecture concern: context, containers, flow, security, deployment, decisions
- "current implementation notes" sections where the codebase diverges from the canonical contract
- concise engineering prose with tables and diagrams instead of internal design speculation

## Gotchas

- **Do not present `topology` as a separate deployed API process today**
  - It is an architectural domain and code area, but current routing is through `query-api`.

- **Do not overstate the topic graph as fully live**
  - The repository keeps a broader Kafka topic catalog than the default runtime currently exercises.

- **Portal and journey docs should stay careful**
  - The frontend routes exist, but some site-management and journey-backend pieces remain partial or indirect.
