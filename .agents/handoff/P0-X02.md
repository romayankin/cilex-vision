## Key decisions and why

- **Mapped each data class to the actual architecture, not an idealized one**
  - The framework references the stores that really exist in the repo today:
    - TimescaleDB / PostgreSQL tables from `services/db/models.py`
    - MinIO buckets from `infra/docker-compose.yml` and Ansible vars
    - Kafka `mtmc.active_embeddings`
  - Where the architecture and local stack differ, the doc calls that out explicitly instead of pretending the gap does not exist.

- **Kept RBAC aligned with implemented code**
  - The role matrix uses the exact four roles already enforced by `services/query-api/auth/jwt.py`.
  - The document also notes that debug trace access is enforced separately at router level, not through the `ROLE_PERMISSIONS` dict.

- **Separated implemented controls from required hooks**
  - Implemented today:
    - Timescale retention on `detections` and `track_observations`
    - query API audit middleware
    - engineering/admin debug-trace restriction
  - Required before production:
    - full MinIO lifecycle reconciliation
    - DSAR export flow
    - deletion workflow
    - per-camera retention override
    - export-time redaction

- **Used regulator guidance only for practical triggers, not legal overreach**
  - DPIA trigger criteria are framed as an internal engineering standard that can be applied outside EU/US deployments.
  - Official sources cited:
    - EDPB DPIA trigger guidance
    - UK ICO surveillance guidance
    - Singapore PDPC CCTV access/retention guidance

## Patterns established

- **Privacy docs should distinguish three states**
  - implemented in code today
  - defined in architecture / inventory but not fully wired
  - missing and required before production

- **Privacy controls should be described by data class, not just by service**
  - This makes retention, access, and encryption review easier across DB, object store, and Kafka.

- **Future privacy operations should be job-based and admin-only**
  - The framework recommends asynchronous export/deletion jobs rather than synchronous endpoints because these operations span multiple stores.

## Gotchas for the next agent

- **`raw-video` is architectural, not fully present in local compose**
  - `docs/taxonomy.md`, `docs/kafka-contract.md`, and Ansible vars assume a `raw-video` bucket.
  - `infra/docker-compose.yml` currently creates `frame-blobs`, `event-clips`, `debug-traces`, and `thumbnails`, but not `raw-video`.

- **There is no committed `infra/minio/lifecycle/` directory in this repo**
  - Lifecycle intent exists in Ansible vars and in task handoffs, but not as checked-in lifecycle JSON artifacts.

- **Metadata retention is split**
  - Hypertable data is explicitly 30 days.
  - Relational metadata is currently indefinite in the schema, even though taxonomy.md states a 365-day metadata target.
  - Any future implementation task needs to decide whether to preserve indefinite retention or add explicit relational retention jobs.

- **Audit coverage is API-centric today**
  - Query API requests are logged.
  - Signed URL issuance and direct object-store retrieval are not yet fully modeled as privacy-audit events in one place.
