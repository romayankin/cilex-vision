# P4-A01: Multi-Site Dataset — Handoff

## What was built

3 new scripts + 1 docs template:

1. **`scripts/annotation/aggregate_datasets.py`** — Multi-site CVAT dataset aggregation. Connects to one or more CVAT instances (potentially different versions), fetches project tasks and annotations, normalizes labels to canonical ObjectClass enum (with alias mapping for common variants like "pedestrian"→"person"), tags each sample with site_id/conditions/camera_model metadata, deduplicates by item_id, and writes a unified manifest JSON. Supports both multi-site config file mode and single-site CLI shortcut.

2. **`scripts/annotation/dataset_analysis.py`** — Dataset statistics and gap analysis. Reads a unified manifest and computes: per-class counts (total, per-site, per-condition), condition distribution, camera model coverage, camera counts per site, and gap analysis (classes below minimum threshold at any site). Generates a Markdown coverage report with tables. Optional JSON output for machine consumption.

3. **`scripts/annotation/balanced_split.py`** — Balanced train/val/test splitting. Reads a unified manifest and creates splits with proportional representation from all sites and conditions. Groups items into temporal sequences per camera (reusing split_dataset.py's gap-based logic), then allocates sequences to splits per stratum (site × class × conditions). Writes split manifests, summary JSON, summary Markdown, and optional DVC .dvc tracking files.

4. **`docs/dataset-coverage-report.md`** — Template coverage report with placeholder tables for class distribution, per-site breakdown, condition coverage, camera model coverage, gap analysis, split statistics, and recommendations.

## Key decisions

| Decision | Rationale |
|----------|-----------|
| Label alias mapping (LABEL_ALIASES dict) | CVAT annotators may use "pedestrian", "vehicle", "bike" etc. — map to canonical ObjectClass before merging. Keeps normalization explicit and extensible. |
| Stratified splitting by (site, class, conditions) | Ensures every split has proportional representation from all strata. Avoids a split being dominated by one site or class. |
| Temporal ordering within strata | Within each stratum, sequences are sorted chronologically and allocated in order. This prevents temporal leakage (future data in training, past data in test). Reuses split_dataset.py's gap-based sequence grouping. |
| Pagination for CVAT fetch (fetch_all_pages) | CVAT list endpoints paginate. The aggregator follows `next` links to collect all results, handling both dict-with-results and bare-list response formats for version compatibility. |
| DVC files as optional output | Not all operators use DVC. The `--dvc` flag writes .dvc files with md5/size metadata alongside split manifests. |
| Sites config file with per-site password env vars | Each site can have a different CVAT password stored in a different env var (cvat_password_env). Avoids hardcoding credentials in the config file. |
| Gap threshold as CLI param (default 100) | Different teams have different minimum sample requirements. Configurable via `--min-samples`. |

## Patterns followed

- `urllib.request` with basic auth (no third-party HTTP libs) — matching all existing annotation scripts
- `argparse` CLI with env var fallbacks (`CVAT_URL`, `CVAT_USERNAME`, `CVAT_PASSWORD`)
- `build_headers`, `request_json`, `build_url`, `create_ssl_context` HTTP helpers identical to setup_cvat_projects.py
- `if __name__ == "__main__"` with `try/except SystemExit(str(exc))` entry pattern
- JSON manifest format compatible with split_dataset.py's `items` array contract
- `from __future__ import annotations` for forward reference support
- `pathlib.Path` for all file I/O
- Markdown report generation with table formatting

## Gotchas

1. **CVAT API version differences.** The aggregator checks version via `/api/server/about` but only warns (doesn't fail) for non-2.x versions. Some older CVAT versions may return annotations in a different shape — the script handles both `dict.results` and bare list responses.

2. **Annotation task filtering.** `fetch_all_pages("/api/tasks")` fetches ALL tasks, then filters by `project_id` client-side. For CVAT instances with thousands of tasks across many projects, this is inefficient. The CVAT API supports `?project_id=X` filtering, but some versions don't — the safe approach fetches all and filters.

3. **Camera ID extraction is heuristic.** `extract_camera_id_from_task()` tries to parse camera IDs from task names using common patterns. Tasks with non-standard names will fall back to `task-{id}`. Operators should use consistent task naming.

4. **Capture timestamps are approximate.** If CVAT tasks don't carry explicit capture timestamps, the aggregator derives them from task `created_date` with frame offsets. Real capture timestamps should be embedded in task metadata for accurate temporal splitting.

5. **Balanced split rebalancing.** If a stratum is too small to have sequences in all three splits, the rebalancer takes items from the largest split. This can slightly violate temporal ordering for very small strata.

6. **MD5 in DVC files.** Uses `hashlib.md5` for DVC compatibility (DVC's default hash). The `# noqa: S324` suppresses the Bandit security warning since this is for content addressing, not cryptographic security.

## Open questions

1. **CVAT task data upload automation.** The aggregator fetches existing annotations but doesn't upload frames to CVAT tasks. A future script could automate task data population from MinIO.
2. **Incremental aggregation.** Currently the aggregator does a full fetch each run. For large annotation pools, an incremental mode that fetches only new/updated tasks since the last run would be more efficient.
3. **Cross-site identity overlap.** The aggregator treats each site independently. If the same real-world entity appears at multiple sites, the manifest will have separate entries with no cross-site identity linkage.
