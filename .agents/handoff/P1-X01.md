  - Replaced the stub with a real compatibility document.
  - Added the required matrix columns:
    - Brand
    - Model
    - Firmware
    - ONVIF

  - Seeded the matrix with:

    - `Hikvision DS-2CD2387G2-LU (ColorVu)`
  - Both rows are explicitly `UNTESTED` and sourced from published vendor data because no physical cameras were available.

- `scripts/camera-compat/probe_camera.py`
  - Real ONVIF + RTSP probe CLI.
  - Live mode:
    - probes ONVIF device info
    - enumerates media profiles and stream URIs
    - derives stream count, codec list, H.265 support, and basic PTZ presence
    - attempts an RTSP frame read with OpenCV
  - Published-only mode:
    - emits a valid JSON report without network access
    - useful for bootstrapping matrix rows from vendor datasheets
  - JSON output includes `matrix_columns`, so the suite can render Markdown without re-deriving display fields.

- `scripts/camera-compat/run_compat_suite.sh`
  - Reads CSV or YAML inventory rows.
  - Calls `probe_camera.py` once per row.
  - Writes one JSON report per camera under `artifacts/camera-compat/.../reports/`.
  - Renders a Markdown summary matrix at `artifacts/camera-compat/.../matrix.md`.

## Key decisions and why

- **Published-only mode was added intentionally**
  - The prompt requires a real probe tool, but this environment has no live cameras.
  - Published-only mode keeps the output contract identical while still allowing the matrix workflow to run end-to-end for `UNTESTED` rows.

- **Dependency failures are fatal in live mode**
  - Missing `onvif-zeep` or OpenCV now exits immediately with a clear install hint.
  - This matches the bake-off harness pattern used elsewhere in the repo.

- **Matrix output is data-driven**
  - The probe emits a `matrix_columns` object containing exactly the Markdown table values.
  - The shell suite only aggregates; it does not need to understand ONVIF internals.

- **Used the existing edge-agent connection assumptions**
  - The document and probe design align to `services/edge-agent/rtsp_client.py`:
    - RTSP over TCP
    - codec-agnostic `decodebin`
    - H.264/H.265 family support

## Patterns established

- **For hardware-unavailable tasks, provide a real harness plus a published-data bootstrap path**
  - This is the same practical pattern used in the bake-off tasks.

- **Shell suites can parse CSV/YAML inventories with small embedded Python helpers**
  - No extra repo-local helper module was added just to support inventory parsing.

- **Compatibility JSON should include display-ready matrix fields**
  - This avoids duplicating formatting logic in every downstream report generator.

## Gotchas for the next agent

- **The Dahua row uses the closest verified public 5449T-family datasheet**
  - The exact `IPC-HDW5449T` suffix in procurement may vary by lens or regional SKU.
  - Treat the row as a family-level placeholder until the shipped unit is probed.

- **IK10 is still not resolved for the starter rows**
  - The reviewed Hikvision product page clearly lists `IP67` but not `IK10`.
  - The reviewed Dahua family datasheet did not let me assert the exact chassis rating for the requested row.
  - If procurement requires vandal resistance, verify the exact SKU datasheet before changing those fields.

- **YAML inventories should quote `Yes`/`No` if used as plain text**
  - The suite now normalizes booleans for safety, but quoted strings are still clearer for human-edited inventories.

- **Live probing still needs actual runtime packages**
  - `probe_camera.py --help` works without extras.
  - Real ONVIF probing needs `onvif-zeep`.
  - Real RTSP validation needs OpenCV.
