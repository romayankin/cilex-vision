# P3-X02: Operations Handbook — Handoff

## What was built

6 Markdown files under `docs/operations/`:

1. **README.md** — Handbook overview, section navigation, quick reference table (all 8 dashboard UIDs, service URLs), emergency contacts placeholder, links to all 6 existing runbooks.

2. **daily-checks.md** — 8-item morning checklist ordered by priority: camera health, pipeline throughput, storage health, alert review, MTMC health, model quality, calibration status, drift monitoring. Each item links to the relevant Grafana dashboard, lists specific panels and thresholds, and includes escalation criteria.

3. **monitoring-guide.md** — Dashboard-by-dashboard reference for all 8 Grafana dashboards (Stream Health, Inference Performance, Bus Health, Storage, Model Quality, MTMC Health, Shadow Comparison, Storage Tiering). Each section includes the full panel table, what to look for, and when to escalate. Complete alert rule reference covering all 46 alerts across 10 rule files, organized by service group with severity labels and descriptions. Screenshot placeholders included.

4. **maintenance-windows.md** — Pre-maintenance checklist, service restart order (16 services), TimescaleDB maintenance (compression/retention verification, chunk health, VACUUM, materialized view refresh), MinIO maintenance (lifecycle policy table with all 8 buckets, apply/report commands), Kafka maintenance (topic inspection, consumer group reset), calibration baseline refresh, drift baseline recapture, post-maintenance verification.

5. **capacity-planning.md** — Current resource inventory (cameras, GPU, storage, infrastructure), scaling indicators (when to add cameras/GPUs/storage/Kafka brokers with dashboard-based thresholds), per-camera resource estimates (compute, storage growth, network), growth projections (4→20 cameras, 20→100 cameras).

6. **glossary.md** — 80+ terms covering object classes (7 types), pipeline stages (edge → ingest → inference → tracking → MTMC → storage → query), infrastructure terms, monitoring terms, model terms, and event types.

## Key decisions

| Decision | Rationale |
|----------|-----------|
| Organized by operator workflow, not by service | Operators think in terms of "morning checks" and "maintenance windows," not individual microservices. |
| Dashboard panels listed as tables with thresholds | Operators need to know what number is bad, not just what the panel shows. |
| Cross-referenced existing runbooks instead of duplicating | Avoids content drift between the handbook and the runbooks (6 existing runbooks in docs/runbooks/). |
| Alert reference includes all 46 actual alerts | Counted from the actual rule files, not the task spec's estimate of 42. Ensures nothing is missing. |
| Capacity planning uses per-camera estimates | Operators scale by camera count, so resource estimates should be denominated per camera. |
| Glossary organized by domain area | Pipeline-stage grouping helps operators understand where each term fits in the data flow. |

## Patterns followed

- All dashboard references use exact UIDs from the Grafana JSON files (e.g., `stream-health`, `bus-health`).
- All alert names match the exact `alert:` field values from the Prometheus rule YAML files.
- Screenshot placeholders use `<!-- screenshot placeholder: [name] -->` format as required.
- Cross-references to runbooks use relative Markdown links.
- Service restart order matches the canonical order from `docs/runbooks/service-restart.md`.
- MinIO lifecycle policies match `infra/minio/lifecycle-policies.json` exactly.

## Gotchas

1. **46 alerts, not 42**: The task spec mentions 42 rules across 9 files, but the actual codebase has 46 alerts across 10 files (clock-alerts.yml and clock-drift.yml are separate files).

2. **Shadow dashboard is only relevant during model rollout**: The monitoring guide notes this clearly. Operators should not be alarmed by empty panels when no shadow deployment is active.

3. **Emergency contacts are placeholders**: The README includes a placeholder contact table that needs to be filled in with actual contact information.

4. **Calibration and drift baselines are external state**: The maintenance procedures reference artifacts stored in MinIO (`s3://debug-traces/baselines/confidence-baseline.json`) and local directories (`artifacts/calibration/`, `artifacts/monitoring/`). These must exist before the procedures can be followed.

5. **Per-camera resource estimates are approximate**: The capacity planning numbers are derived from the pipeline architecture specs and model inventory, not from measured production data. They should be validated against the pilot deployment.
