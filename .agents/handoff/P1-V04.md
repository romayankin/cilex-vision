# Handoff: P1-V04 — Detection & Tracking Inference Worker

## What was built

- `services/inference-worker/main.py` — Kafka consumer pipeline: consume FrameRef from `frames.sampled.refs`, download JPEG from MinIO, detect → track → embed → publish, graceful shutdown via SIGINT/SIGTERM
- `services/inference-worker/detector_client.py` — Triton gRPC client for YOLOv8-L: letterbox preprocessing (640×640, gray pad, /255), NMS post-processing (per-class greedy NMS), confidence filtering at 0.40 threshold, un-letterbox to normalised [0,1] coordinates
- `services/inference-worker/tracker.py` — ByteTrack single-camera tracker: two-pass IoU matching (high-conf then low-conf detections), linear-velocity Kalman prediction, Hungarian assignment via scipy, track lifecycle NEW→ACTIVE→LOST→TERMINATED, majority-class voting, best-frame selection for embedding
- `services/inference-worker/embedder_client.py` — Triton gRPC client for OSNet Re-ID: crop extraction, bilinear resize to 256×128, ImageNet normalisation, L2-normalised 512-d output
- `services/inference-worker/publisher.py` — Kafka producer for 3 topics: Detection protos → `bulk.detections` (with `x-frame-seq`, `x-local-track-id`, `x-proto-schema` headers), Tracklet protos → `tracklets.local`, Embedding protos → `mtmc.active_embeddings` (with tombstone on track termination)
- `services/inference-worker/debug_trace.py` — configurable sampling (1–5%), always traces low-confidence detections, stores JSON traces to MinIO
- `services/inference-worker/config.py` — Pydantic Settings v2 with `from_yaml()` and `INFERENCE_` env prefix; nested configs for Triton, Kafka, MinIO, Tracker, Detector, Debug
- `services/inference-worker/metrics.py` — 8 Prometheus metrics: detections_total{class}, tracks_active{camera_id}, tracks_closed{camera_id}, inference_latency_ms, embedding_latency_ms, frames_consumed, publish_errors{topic}, consumer_lag{topic,partition}
- `services/inference-worker/gen_proto.sh` — proto generation script (Docker + local paths)
- `services/inference-worker/Dockerfile` — python:3.11-slim, proto generation at build time
- `services/inference-worker/requirements.txt` — 12 pinned dependencies
- `services/inference-worker/README.md` — usage, config, metrics, test instructions
- `services/inference-worker/tests/conftest.py` — sys.path setup
- `services/inference-worker/tests/test_detector_client.py` — 13 tests: preprocessing shape/range/letterbox, NMS correctness/per-class/empty, postprocess with synthetic YOLOv8 output, end-to-end mock, class mapping
- `services/inference-worker/tests/test_tracker.py` — 13 tests: IoU matrix, track lifecycle (create/active/lost/terminated), multi-object tracking, continuity, moving objects, trajectory accumulation, class voting, low-conf second-pass matching
- `services/inference-worker/tests/test_debug_trace.py` — 11 tests: sampling on/off/low-conf/high-conf, trace construction, JSON serialisation, MinIO storage, TraceStage duration

## Key decisions and why

- **NMS is client-side, not Triton-side**: YOLOv8 TensorRT engine outputs raw [11, 8400] tensor. The P0-D10 handoff explicitly states "NMS is the client's responsibility." Post-processing transposes to [8400, 11], filters by confidence, then applies per-class greedy NMS.
- **YOLOv8 output is transposed**: shape is [batch, 4+num_classes, 8400], NOT [batch, 8400, 4+num_classes]. This matches the P0-D10 gotcha. The detector_client transposes before processing.
- **ByteTrack on CPU, not Triton**: per P0-D10 handoff: "ByteTrack is NOT a Triton model." Implemented with scipy `linear_sum_assignment` for Hungarian matching and numpy for IoU computation.
- **Two-pass matching**: ByteTrack's signature approach — first match high-confidence detections (≥0.5) to tracks, then try matching remaining tracks to low-confidence detections. This recovers occluded objects that might have low detection scores.
- **Lost track recovery**: unmatched high-confidence detections are also matched against LOST tracks before creating new tracks. This prevents duplicate track creation when a temporarily lost object reappears.
- **Pillow for image resize**: used for letterbox resize and crop resize, consistent with the edge-agent pattern. Avoids heavy cv2 dependency.
- **Bulk detections topic = `bulk.detections`**: matches the P1-V05 handoff which says "defaulted it to `bulk.detections`". The bulk-collector is already configured to consume from this topic.
- **Detection proto headers match bulk-collector expectations**: `x-proto-schema`, `x-frame-seq` (required), `x-local-track-id` (optional) — exactly what the bulk-collector's `parse_message` method expects.
- **Tombstone on track termination**: when a track reaches TERMINATED, a null-value message is published to `mtmc.active_embeddings` so the compacted topic eventually removes the stale embedding entry.
- **Best-frame embedding**: embeddings are only extracted when the current detection has higher confidence than all previous frames for that track. This ensures we send the highest-quality crop for Re-ID.
- **tracker_version = "bytetrack-1.0"**: hardcoded in publisher.py. Matches the bake-off result that selected ByteTrack as the tracker baseline.

## Patterns established that future agents should follow

- **Triton client pattern**: lazy import of `tritonclient.grpc`, sync `_infer()` wrapped in `asyncio.to_thread()`. Same client instance reused across calls.
- **Proto serialisation in publisher.py**: lazy import of generated proto types from `proto_gen/`, separate `build_*_proto()` functions for each message type. All proto construction is in publisher.py, not in the pipeline code.
- **Kafka headers**: every message includes `x-proto-schema` with the fully qualified protobuf name. Detection messages also include `x-frame-seq` and `x-local-track-id` for the bulk-collector.
- **Per-camera tracker state**: `_trackers` dict in main.py keyed by `camera_id`. Each camera gets its own ByteTracker instance with independent track IDs and state.
- **Debug trace integration**: trace stages are recorded inline in the pipeline (`download`, `detect`, `track`, `embed`, `publish`) with epoch timestamps. The tracer decides sampling before detection, then re-checks after detection for low-confidence forcing.

## Gotchas

- **Triton must be in EXPLICIT mode with models loaded**: the inference worker calls `infer()` directly — it does NOT call `/v2/repository/models/{name}/load`. The ops/startup layer must ensure models are loaded before the worker starts consuming.
- **Bbox coordinate spaces**: YOLOv8 outputs in 640×640 pixel space → un-letterboxed to original resolution → normalised to [0,1] for the proto BoundingBox. The bulk-collector then converts `x_min/y_min/x_max/y_max → x/y/w/h` for the DB schema.
- **`bulk.detections` topic does not exist in `infra/kafka/topics.yaml`**: the P1-V05 handoff notes this gap. The topic is used by convention between this service and the bulk-collector, but it's not yet formally defined.
- **scipy is a runtime dependency**: required for `linear_sum_assignment` (Hungarian algorithm). It's a heavy package (~35 MB wheel). If a lighter alternative is preferred, a pure-Python Hungarian implementation could replace it.
- **Track trajectory is truncated to last 10 points**: to keep Tracklet proto messages small on Kafka. If full trajectory is needed downstream, the consumer should accumulate points across messages.
- **No Schema Registry serialisation**: detection/tracklet/embedding messages are published as raw protobuf bytes, consistent with the edge-agent and ingress-bridge pattern. The `x-proto-schema` header identifies the message type.
- **frame_data reference in STrack**: the tracker stores a reference to the full frame (`best_frame_data`) for embedding extraction. This means frames are held in memory until the track is terminated or a better frame replaces it. For 4 cameras this is ~4 frames in memory; for 64 cameras this could grow.

## Open questions or known limitations

- **No batched Triton inference**: the detector processes one frame at a time. For higher throughput, frames from multiple cameras could be batched into a single Triton request (up to batch=8). This would require accumulating frames in the consumer loop.
- **No color classification**: the task prompt focuses on detection, tracking, and Re-ID. Color classification (color_classifier model) is not part of this service — it will be handled by a separate attribute service consuming from `tracklets.local`.
- **Track-to-detection assignment is approximate**: the `track_assignments` dict in main.py matches detections to tracks by centroid proximity. A more robust approach would pass detection indices through the tracker.
- **No dead-letter queue**: failed messages are logged but not published to a DLQ. The kafka-contract.md requires DLQ after 3 retries — this should be added when the DLQ infrastructure is in place.
- **No Kafka consumer lag alerting**: the metric is exported but no alert rule is defined. The kafka-contract.md specifies alerting at > 10,000 messages for > 60s.
