# P3-O01 — Deployment Automation

## What I built

- `infra/terraform/modules/compute/*`
  - generic compute module for `aws`, `gcp`, and `bare_metal`
  - supports `gpu_inference`, `cpu_service`, `edge_gateway`, and `monitoring` node roles
  - outputs hostnames, private/public IPs, and SSH key references
- `infra/terraform/modules/network/*`
  - provider-aware network module with core, edge, and camera subnets
  - AWS security groups and GCP firewall rules for core-internal, edge-to-core, camera-to-edge, monitoring, and public HTTPS access
- `infra/terraform/modules/storage/*`
  - block-volume provisioning for TimescaleDB, MinIO, FAISS spool, and Kafka log storage
- `infra/terraform/environments/production/main.tf`
  - production topology: 3 Kafka, 1 TimescaleDB, 1 MinIO, 2 Triton GPU nodes, 1 monitoring node, 2 service nodes, N edge gateways, N NATS edge nodes, 1 MLflow node
  - `backend "s3" {}` for configurable S3 + DynamoDB state locking
- `infra/terraform/environments/staging/main.tf`
  - smaller staging topology with single-node variants of the core services
- `infra/terraform/environments/{production,staging}/terraform.tfvars`
  - baseline environment defaults
- `infra/ansible/playbooks/deploy-multi-node.yml`
  - multi-node orchestration in dependency order
  - prepares GPU hosts first, deploys infra, then Triton, edge gateways, application services, MTMC, monitoring, MLflow, and smoke tests
- `infra/ansible/roles/gpu-node/tasks/main.yml`
  - NVIDIA driver/toolkit bootstrap, Docker runtime config, GPU verification, Triton model repo sync
- `infra/ansible/roles/edge-gateway/tasks/main.yml`
  - GStreamer install, PKI distribution, rendered edge-agent config, repo-root build-context staging, edge-agent deployment, local buffer budget marker, node-exporter
- `infra/ansible/inventory/production.yml`
  - multi-node production inventory template with core, edge, monitoring, service, MLflow, and optional CVAT groups
- `scripts/deploy/health-check-all.sh`
  - checks Kafka, NATS, TimescaleDB, MinIO, Triton, Prometheus, Grafana, edge-agent, MTMC, and inventory-defined app endpoints
  - supports either env-var targeting or `--inventory`
  - prints PASS/FAIL table and exits non-zero on any failure

## Key decisions

- **Generalized `deploy-services.yml` instead of adding more one-off service playbooks**
  - The existing generic service deployer only handled service-local Docker build contexts.
  - Most newer services now build from repo root because they `COPY proto/`.
  - I added `build_context_mode: repo_root` plus `dockerfile:` support so the shared playbook can deploy those services without breaking pilot-era `service_dir` builds.

- **Kept multi-node deployment additive to the pilot path**
  - Existing pilot playbooks and `inventory/pilot.yml` were left intact.
  - New roles, the new inventory template, and the new orchestration live beside the pilot assets instead of replacing them.

- **Made edge gateways first-class monitored nodes**
  - `deploy-monitoring.yml` and the Prometheus template now include `edge_gateways` for node-exporter coverage.
  - Prometheus also scrapes edge-agent `/metrics` directly when that group exists.

- **Used placeholder-friendly Terraform environments**
  - The Terraform environments are intentionally structured for `plan/apply`, but still rely on operator-supplied AMI/image IDs, backend config, and optionally bare-metal host maps.
  - This keeps the modules reusable across cloud and on-prem without hardcoding an opinionated provider-specific bootstrap.

## Patterns followed

- Reused the existing Ansible structure from `P1-O01`: `community.docker`, `group_vars/all.yml`, `repo_root`, `common` role, idempotent `docker_container` / `docker_image` tasks, and post-start smoke checks
- Kept shared infra defaults in `infra/ansible/group_vars/all.yml`
- Used host networking on edge-agent and node-exporter to stay consistent with the pilot infra playbooks and simplify scrape/RTSP wiring
- Staged repo-root-compatible Docker build contexts only when needed, instead of copying the full repository to every host

## Gotchas

- The GPU role writes `/etc/docker/daemon.json` with the NVIDIA runtime as default. That is fine for dedicated Triton hosts, but if those nodes already have daemon settings, merge them before running the role.
- The generalized `deploy-services.yml` now supports repo-root builds by copying `proto/` plus the requested `services/...` path. If a future service Dockerfile needs more root-level paths, extend the staging logic before relying on it.
- `health-check-all.sh --inventory` depends on Python + PyYAML on the operator machine because it parses Ansible inventory YAML directly.
- The Terraform modules were not `terraform validate`-checked locally because the Terraform CLI is not installed in this environment.
