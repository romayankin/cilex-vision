# P4-O02 — Disaster Recovery

## What I built

- `infra/backup/backup-db.sh`
  - cron-safe PostgreSQL / TimescaleDB logical backup script
  - captures a `pg_dump -Fc` database dump plus `pg_dumpall --globals-only`
  - timestamps outputs, validates the target container, and prunes backups older than the configured retention window
- `infra/backup/backup-minio.sh`
  - MinIO bucket backup script using dockerized `mc mirror`
  - mirrors the critical buckets to a backup endpoint and skips missing buckets with warnings instead of failing the whole run
- `infra/backup/backup-config.sh`
  - configuration backup archive for `infra/ansible`, `infra/pki`, `infra/terraform`, repo `.env` files, and a snapshot of the `mtmc-checkpoints` bucket
  - keeps only the newest N archives
- `infra/restore/restore-db.sh`
  - full logical restore helper for TimescaleDB / PostgreSQL
  - validates the container, recreates the target database, optionally reapplies globals, restores the dump, and verifies critical tables plus row counts
- `infra/restore/restore-full.sh`
  - end-to-end restore wrapper
  - restores config files, restores the database, restores MinIO buckets from a backup endpoint, restores `mtmc-checkpoints` from the config snapshot, runs `scripts/deploy/health-check-all.sh`, and prints RPO / RTO summary data
- `infra/failover/health-watchdog.py`
  - service watchdog for Kafka, NATS, TimescaleDB, MinIO, Prometheus, and Triton
  - supports direct TCP / HTTP checks, optional Prometheus `up` confirmation, disk-usage and Kafka-lag threshold checks, optional YAML config, and optional webhook alert delivery
- `docs/disaster-recovery.md`
  - DR reference covering targets, schedules, restore order, failover guidance, monthly drills, and escalation placeholders
- `scripts/test-restore.sh`
  - isolated restore drill harness that starts a temporary TimescaleDB container, restores a dump, verifies critical tables, and tears the container back down

## Key decisions

- **Kept the automation container-centric**
  - The repo’s pilot and local environments are Docker-driven.
  - The backup and restore scripts therefore target existing container names and use dockerized `mc` so operators do not need extra host tooling beyond Docker.

- **Separated database-only restore from full restore**
  - `restore-db.sh` handles the high-frequency, low-scope case cleanly.
  - `restore-full.sh` composes config restore, DB restore, object restore, checkpoint restore, and health verification for a broader recovery event.

- **Used health-check reuse instead of new bespoke post-restore probes**
  - The repo already has `scripts/deploy/health-check-all.sh`.
  - Reusing it keeps restore verification aligned with the rest of the ops workflow instead of creating a second incompatible definition of “healthy”.

- **Made MinIO automation warning-tolerant for missing buckets**
  - Some buckets differ between pilot and multi-node deployments.
  - The backup / restore scripts warn and continue when a bucket is absent so the same script remains usable across both layouts.

## Patterns followed

- shell automation uses `set -euo pipefail`, explicit argument parsing, timestamped artifacts, and clear exit conditions
- Python automation stays stdlib-first and mirrors the repo’s monitoring tooling style with optional dependency loading only where needed
- operational verification reuses existing repo conventions such as `health-check-all.sh` and the documented container names from the compose files

## Gotchas

- **Kafka offsets are still not automated**
  - The backup runbook lists offsets as an important recovery asset, but `P4-O02` does not add a dedicated snapshot / restore script for them.
  - Operators still need the documented manual procedure until that gap is closed.

- **Database backups are logical dumps, not PITR**
  - `backup-db.sh` does not configure WAL archiving or point-in-time recovery.
  - That is fine for a baseline DR flow, but it is weaker than a true 15-minute write-heavy RPO under all failure modes.

- **`backup-minio.sh` and `restore-full.sh` require a reachable backup MinIO endpoint**
  - They expect `BACKUP_ENDPOINT` and credentials for the remote mirror target.
  - There is no built-in provisioning of that target in this task.

- **`restore-full.sh` assumes repo restoration onto an existing checkout path**
  - It extracts the config archive into the chosen restore root and copies the archived repo subtree over it.
  - It is a recovery helper, not a bare-metal bootstrap workflow.

- **`test-restore.sh` validates structural recovery, not application correctness**
  - It checks table existence and counts for critical tables.
  - It does not replay application traffic or validate downstream query semantics.
