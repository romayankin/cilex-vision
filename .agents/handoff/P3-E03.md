# P3-E03 — Drift Monitoring

## What I built

- `scripts/monitoring/baseline_snapshot.py`
  - asyncpg-based baseline capture for per-camera, per-class confidence histograms
  - supports local JSON output or MinIO `s3://bucket/key` output via lazy `minio` import
  - stores metadata including snapshot time, window, bin count, model-version totals, and per-class / per-camera detection totals
- `scripts/monitoring/drift_detector.py`
  - asyncpg-based drift scan over the recent `detections` window
  - loads the stored baseline from local disk or MinIO
  - computes KS-test statistics and p-values via lazy `scipy.stats.ks_2samp`
  - computes KL divergence on the same normalized histograms
  - writes:
    - Markdown report
    - Prometheus textfile metrics for drift score / p-value / KL / drift flag
  - exits `1` when any camera/class group breaches the drift thresholds
- `infra/cron/drift-monitoring-cron.yml`
  - hourly cron/systemd artifact that runs the detector against the canonical baseline key
  - points the metrics output at `/var/lib/node_exporter/textfile_collector/confidence_drift.prom`

## Key decisions

- **Use TimescaleDB histograms instead of Prometheus.**
  The inference worker still does not export a production confidence histogram, so the only trustworthy source for production confidence distribution is the `detections` hypertable.

- **Use both KS and KL, but gate on the task’s default thresholds.**
  The detector records:
  - KS statistic
  - KS p-value
  - KL divergence

  Drift is flagged when either:
  - `p_value < 0.01`
  - `KL > 0.5`

  That keeps the raw score visible while still giving operators one binary outcome.

- **Reconstruct samples from histogram buckets for `ks_2samp`.**
  The database query returns grouped counts, not raw samples. To stay aligned with the prompt’s requirement to use `scipy.stats.ks_2samp`, the script expands each bucket count at the bucket midpoint before calling SciPy.

- **Fail fast on incompatible baselines.**
  The detector errors if:
  - the baseline file is missing
  - the baseline has a different bin count than the current run
  - no overlapping camera/class groups exist between baseline and current data

  That is safer than silently emitting misleading “no drift” output.

- **Write textfile metrics atomically.**
  Both the Markdown report and Prometheus textfile are written through temp files and renamed into place, matching the repo’s monitoring collector pattern and avoiding partial reads.

## Patterns followed

- CLI-first `argparse` entrypoints with `if __name__ == "__main__"`
- lazy optional imports for `asyncpg`, `minio`, and `scipy`
- clear `RuntimeError` / `SystemExit` failure messages for missing runtime dependencies
- local-or-MinIO path handling for operator artifacts
- repo-relative defaults under `artifacts/monitoring/`

## Gotchas

- The detector observes the `detections` table, not raw detector output. In practice that means the monitored confidence distribution is already truncated by the production ingest threshold, so sub-threshold drift is invisible today.
- Baseline creation is manual. The hourly detector assumes `s3://debug-traces/baselines/confidence-baseline.json` already exists.
- The cron definition is an operator artifact only; it is not installed by any deployment playbook yet.
- Version changes will often look like drift. The report includes baseline and current `model_version` sets per camera/class to make those transitions explicit.
