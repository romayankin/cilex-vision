# P4-V05: Multi-Site Portal — Handoff

## What was built

6 new files + 2 modified:

1. **`frontend/app/portal/page.tsx`** — Site overview dashboard. Lists all sites as health cards in a responsive grid (1-4 columns). Paginated (24 per page) for 50+ sites. Each card shows name, camera count, alerts, storage, health status. Click navigates to site settings. "New Site" and "Compare Sites" buttons in header. Admin only.

2. **`frontend/app/portal/sites/new/page.tsx`** — Site creation wizard. Form: name (required), address, timezone dropdown (12 common timezones). Submits via `createSite()` POST to `/api/sites`. Redirects to portal on success. Admin only.

3. **`frontend/app/portal/sites/[siteId]/settings/page.tsx`** — Per-site settings page. Loads site details + topology (camera list) in parallel. Edit name/address/timezone. Saves via `updateSite()` PUT. Shows camera table (ID, name, zone, status) and retention policy summary. Admin only.

4. **`frontend/app/portal/comparison/page.tsx`** — Cross-site comparison. Select 2-5 sites via toggle buttons. Side-by-side table with metrics (camera count, detection rate, alert frequency, storage). Four bar charts using pure CSS/HTML (no chart library). Metrics are derived from camera count for now (mock) until real metrics API exists. Admin only.

5. **`frontend/components/SiteHealthCard.tsx`** — Pure presentation component. Color-coded border and badge (green/yellow/red). Shows name, camera count, alerts, storage percentage with progress bar. Site ID in monospace. Click handler prop.

6. **`frontend/components/SiteSwitcher.tsx`** — Nav bar dropdown component. Shows current site with health indicator dot. Native `<select>` with custom styling. Props: sites list, currentSiteId, onChange callback. Pure presentation.

Modified:
- **`frontend/app/layout.tsx`** — Added "Portal" link to nav bar (admin only, before Admin link).
- **`frontend/lib/api-client.ts`** — Added `SiteResponse`, `SiteListResponse`, `CreateSiteRequest`, `UpdateSiteRequest` types and `getSites()`, `getSite()`, `createSite()`, `updateSite()` API methods.

## Key decisions

| Decision | Rationale |
|----------|-----------|
| Client-side pagination (24 per page) | Matches existing search pattern. 24 = clean grid at all breakpoints (1/2/3/4 columns). |
| Mock metrics in comparison page | Detection rate, alerts, storage are derived from camera count until a real metrics API exists. Easy to swap in real data later. |
| Pure CSS bar charts | No chart library needed — keeps bundle small. CSS `height` percentage bars with labels are sufficient for 2-5 site comparison. |
| SiteSwitcher as pure component (not wired into layout) | Parent provides sites list and onChange — decoupled from data fetching. Ready to integrate into layout when site context is a global concern. |
| Health status derived from camera_count | `deriveHealth()` returns "warning" for 0 cameras, "healthy" otherwise. Will need real health endpoint integration later. |
| Timezone dropdown with 12 common zones | Covers most deployments without requiring a full IANA timezone picker. Easily extendable. |
| `"use client"` on all pages | Matches existing codebase pattern. Pages own data fetching via useEffect. |

## Patterns followed

- Same `"use client"` + `useEffect` + `useCallback` data fetching pattern as search/admin pages
- Same loading/error/content rendering pattern
- Same `isAdmin(role)` guard at top of every admin page
- Same `apiFetch`/`apiMutate` for all API calls
- Same Tailwind class patterns (card borders, table styling, badges, buttons)
- Same pagination button pattern from search page
- Same form styling from CameraForm component
- Same nav bar link pattern with `pathname?.startsWith()` active state

## Gotchas

1. **Sites API endpoints don't exist yet.** The portal calls `GET /sites`, `GET /sites/:id`, `POST /sites`, `PUT /sites/:id`. These need to be added to the query-api (or a dedicated site management API). The pages will show API errors until the backend is wired up.

2. **SiteSwitcher is not wired into the layout.** It's a standalone component ready for integration. To make it global, the layout would need to fetch sites and pass them down. This was left out to avoid a data-fetching pattern change in the root layout (which uses `"use client"` but doesn't fetch data).

3. **Comparison metrics are derived/mock.** `deriveMockMetrics()` generates detection rate, alert frequency, and storage from camera count. Replace with real per-site metrics API when available.

4. **Health status is simplified.** Only checks camera count (0 = warning, else healthy). Real implementation needs health check endpoint per site (stream health, inference status, bus health).

5. **Retention policy is hardcoded.** The site settings page shows fixed retention values (7d/90d/30d/30d) matching the MinIO lifecycle policies. When per-site retention becomes configurable, fetch from API instead.
