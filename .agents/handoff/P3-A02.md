# P3-A02 — Re-ID Training Data Collection

## What I built

- `scripts/annotation/collect_reid_training_data.py`
  - queries high-confidence MTMC links from PostgreSQL with `asyncpg`
  - builds directed anchor-positive pairs across cameras for the same `global_track_id`
  - mines hard negatives on the anchor camera from temporally nearby same-class tracks outside the matched global identity
  - resolves representative detections, scans debug traces for `frame_uri`, downloads frames from MinIO, and writes a triplet manifest with local frame/crop assets when available
- `scripts/annotation/validate_reid_pairs.py`
  - derives both positive and hard-negative pair-review items from the triplet manifest
  - creates or reuses a `reid-training-validation` CVAT project with the canonical 7 labels plus a `match` checkbox attribute
  - uploads side-by-side pair images, seeds a full-frame rectangle annotation with the expected `match` value, and exports completed tasks to compute validation accuracy plus approved triplet IDs
- `scripts/annotation/reid_dataset_builder.py`
  - filters triplets by approved validation outcomes when a validation report is supplied
  - copies or regenerates anchor/positive/negative crops into a versioned dataset directory
  - writes `manifest.json`, `train.json`, `val.json`, and `test.json`
  - optionally runs `dvc add` on the versioned dataset root

## Key decisions

- **Debug traces are the crop bridge for MTMC data.**
  The database has the associations and best detection boxes, but it still does not store a direct frame URI on `local_tracks`. The collector therefore uses debug traces as the only trustworthy link from an MTMC identity back to a recoverable frame object.

- **Validation samples both positives and mined negatives.**
  The validator does not review only anchor-positive pairs. It also turns each triplet’s anchor-negative leg into a review item so the final dataset can reject false negatives and false positives symmetrically.

- **Dataset approval is strict when validation exists.**
  If a validation report is passed to the dataset builder, a triplet is kept only when both its positive and negative review items were approved. If no validation report is passed, the builder emits a warning and uses the raw triplets instead.

- **Small datasets still bias toward train.**
  The builder handles tiny approved sets explicitly: one triplet goes to `train`, two triplets split `train` / `test`, and larger sets use chronological 70/15/15 boundaries.

## Patterns followed

- CLI-first scripts with concise `SystemExit(str(exc))` boundaries
- `asyncpg` for database access
- `urllib.request` + reused CVAT helpers from the existing annotation scripts
- reuse of existing annotation patterns from `P3-A01` for multipart media upload and dataset export polling
- JSON manifests as the contract between collection, validation, and dataset assembly

## Gotchas

- The actual ORM schema does **not** contain `local_tracks.embedding_ref`, even though the task prompt mentioned it. The implementation therefore avoids relying on any embedding-ref column and instead uses `global_track_links`, `local_tracks`, `detections`, and debug traces.
- The collector can only emit training-ready crops for tracks that have both:
  - a representative detection in `detections`
  - a matching debug trace carrying `frame_uri`
- The validator’s CVAT upload/export flow was validated via lint, compile, CLI help, and local synthetic dataset-builder smoke tests, but not against a live CVAT deployment in this environment.
- The dataset builder currently produces a versioned triplet dataset manifest, but the repo still has no actual Re-ID training loop that consumes it.
