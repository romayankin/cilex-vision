# P2-E02 — MTMC Evaluation

## What I built

- `scripts/evaluation/reid_metrics.py`
  - pure dataclasses and metric helpers for cross-camera Re-ID evaluation
  - computes Rank-1, Rank-5, mAP, false positive / negative rate, precision,
    recall, F1, total pair counts, and per-camera-pair breakdowns
  - includes JSON-safe serialization helpers used by the CLI harness
- `scripts/evaluation/export_reid_gt.py`
  - converts CVAT or pre-exported Re-ID pair JSON into an evaluation-ready
    ground-truth file
  - validates that identities have at least 2 sightings across at least
    2 cameras
  - fails fast if `local_track_id` values are not DB UUIDs
- `scripts/evaluation/run_mtmc_eval.py`
  - loads ground truth, queries MTMC output from PostgreSQL via `asyncpg`,
    computes metrics, logs everything to MLflow experiment
    `mtmc-evaluation`, and writes local artifacts
  - emits:
    - `metrics.json`
    - `per_camera_pair_report.json`
    - `evaluation_report.md`
    - `evaluation_summary.json`
- `docs/evaluation-results/mtmc-evaluation.md`
  - proxy report documenting the method, the Rank-1 `> 0.70` gate, and why a
    real go / no-go decision is still blocked

## Key decisions

- **Scoring is restricted to annotated track IDs only.**
  The database contains unlabeled site traffic. If that traffic were scored as
  part of the denominator, false positive rates would be polluted by missing
  labels instead of real MTMC mistakes.

- **Rank metrics are computed from final assignments, not raw candidate lists.**
  The current MTMC service writes only final `global_track_id` assignments to
  the database. It does not persist the ranked candidate list that FAISS and
  the matcher considered. The harness therefore derives Rank-1 / Rank-5 / mAP
  from final assignment groups and documents them as proxy retrieval metrics.

- **Ground-truth export is strict about `local_track_id` UUIDs.**
  A ground-truth file without DB-backed `local_track_id` values cannot be used
  for evaluation. `export_reid_gt.py` stops immediately rather than producing a
  file that will fail later with confusing DB join errors.

- **Evaluation window defaults to the annotated tracks’ DB lifetime.**
  If the operator does not pass `--start-time` / `--end-time`, the harness
  derives the window from the referenced `local_tracks` rows. This keeps the
  query bounded without requiring timestamps in the GT export.

## Patterns followed

- `argparse` CLIs with env-var fallbacks and fail-fast errors
- lazy imports for optional runtime dependencies (`asyncpg`, `mlflow`)
- `urllib`-based CVAT access reused from the existing annotation export path
- manual JSON-safe serialization rather than raw dataclass dumping where paths
  and nested objects are involved
- `if __name__ == "__main__"` guard with short `SystemExit` error reporting

## Gotchas

- `scripts/annotation/export_reid_pairs.py` currently creates synthetic
  `local_track_id` values from CVAT shape IDs. That does not match the
  evaluation contract described in the docs and task prompt. A real evaluation
  run will stay blocked until the annotation export preserves the DB UUIDs.

- Per-camera-pair metrics normalize camera pairs lexicographically
  (`cam-a|cam-b`). This is intentional so `cam-a → cam-b` and `cam-b → cam-a`
  collapse into one cross-camera bucket.

- The DB query for `global_tracks` uses overlap semantics
  (`first_seen <= end AND last_seen >= start`) instead of the narrower sample
  query from the prompt. This avoids dropping predicted groups that span the
  boundary of the evaluation window.

## Open questions

2. Should the MTMC service persist candidate-level scoring details so future
   evaluation can compute true retrieval metrics instead of assignment-derived
   proxies?
