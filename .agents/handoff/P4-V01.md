## What was built

Zone-based sharding for the MTMC service so sites with 50+ cameras can partition matching by zone, with a lighter cross-zone associator for boundary tracks.

### New files

1. **`services/mtmc-service/zone_sharding.py`** â€” Zone management module:
   - `ZoneConfig` dataclass: zone_id, boundary_cameras, zone_cameras

   - `ZoneShardingManager` class: computes zone membership and boundary cameras from topology graph, provides `is_in_zone()`, `is_boundary_camera()`, `get_adjacent_zones()`, `refresh()`
   - `publish_boundary_event()`: publishes boundary events to Kafka via confluent-kafka Producer

   - `CrossZoneMatch` dataclass: result of a cross-zone association

   - CLI entry point: `--config`, `--zone-id` arguments

## Key decisions

| Decision | Rationale |
|----------|-----------|
| JSON serialization for ZoneBoundaryEvent | No proto defined yet for cross-zone events. JSON is interim; proto can be added without changing the interface (to_bytes/from_bytes). |
| Separate BoundaryIndex class | Lighter than reusing full FAISSIndex (no track_map cleanup, no horizon expiry). Each zone gets its own small index. |
| Batch matching every 5s | Cross-zone matching is lower priority than intra-zone. Batching reduces overhead and Kafka consumer lag. |
| Relaxed threshold (0.55 vs 0.65) | Cross-zone tracks have more appearance variance (lighting, angle changes between zones). ADR-008 version boundary still enforced. |
| Best-match-per-adjacent-zone | Takes first match above threshold per adjacent zone per event, not global optimum. Avoids quadratic cost with many zones. |
| Optional db_writer | `site_global_links` table does not exist yet. The persistence interface is ready for when the schema migration is added. |

## Patterns followed

- Thread-safe FAISS operations with `threading.Lock` (same as faiss_index.py)
- Pydantic Settings with `MTMC_` env prefix (same as config.py)
- confluent-kafka Consumer with cooperative-sticky assignment (same as main.py)
- Mock DB/Kafka with `unittest.mock.AsyncMock`/`MagicMock` (same as test_matcher.py)
- Test helpers from `helpers.py`: `make_l2_normalised()`, `make_similar_vector()`
- Deterministic RNG via `conftest.py` fixture for reproducible tests

## Gotchas

1. **Integration not wired yet**: `zone_sharding.py` and `cross_zone_associator.py` are standalone modules. To activate zone sharding, `main.py` needs to:
   - Create `ZoneShardingManager` after topology load
   - Check `zone_manager.is_in_zone(camera_id)` before indexing embeddings
   - Publish boundary events on tombstone processing at boundary cameras
   - The cross_zone_associator runs as a separate process/container

2. **No proto for ZoneBoundaryEvent**: Uses JSON serialization. Should be replaced with a proper proto in `proto/vidanalytics/v1/mtmc/` and registered in Schema Registry.

3. **No `site_global_links` DB table**: The `persist_matches()` method calls `db_writer.create_site_global_link()` which requires a new Alembic migration to create the table.

4. **No Kafka topic definition**: `mtmc.cross_zone` needs to be added to `infra/kafka/topics.yaml` and `docs/kafka-contract.md`.

5. **Cross-zone associator metrics port**: Uses `metrics_port + 1` to avoid conflict with the main MTMC service. In production, each container gets its own port.

## Open questions

1. **Cross-zone event TTL**: How long should boundary events stay in the FAISS index? Currently no expiry â€” may need a cleanup similar to `active_horizon_minutes`.
2. **Multi-zone overlap**: Can a camera belong to multiple zones? Current implementation assumes one zone per camera.
3. **Cross-zone event volume**: For medium deployments (50 cameras, ~10 boundary cameras), cross-zone events should be low volume. For large deployments, may need partitioning of the cross_zone topic.
