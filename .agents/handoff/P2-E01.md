# P2-E01 — Attribute Classifier Bake-Off

## What I built

- `scripts/bakeoff/prepare_color_eval_data.py`
  - urllib-based CVAT export for the `attribute-eval` project
  - supports `CVAT for images 1.1` and `Datumaro 1.0`
  - extracts task archives, parses annotations, converts object boxes into
    attribute-specific crops, and writes `data/eval/attribute/<color>/...`
  - writes a rich `manifest.json` with the prompt’s required summary fields plus
    per-crop entries used by the evaluator
- `scripts/bakeoff/run_attribute_bakeoff.py`
  - strict manifest validation with clear fail-fast errors
  - candidate profiles for `resnet18` and `efficientnet_b0`
  - preprocessing aligned to `services/attribute-service/classifier_client.py`:
    resize `224x224`, RGB tensor, `[0, 1]`, `CHW`
  - Triton gRPC inference when available, ONNX Runtime CPU fallback otherwise
  - computes overall accuracy, per-color accuracy, macro per-color accuracy,
    confusion matrix, `unknown_rate`, latency p50/p95/p99, throughput, and the
    three operational confusion-pair metrics
  - logs params, metrics, and artifacts to MLflow experiment
    `attribute-bakeoff`
- `docs/bakeoff-results/attribute-comparison.md`
  - proxy recommendation report that explicitly falls back to the protocol safe
    default because the repository still lacks the real evaluation set and
    challenger artifacts

## Key decisions

- **The prep manifest is richer than the prompt minimum.**
  The prompt’s summary-only manifest is not enough to replay evaluation crops.
  I kept the required top-level keys and added an `items` array with per-crop
  file paths and metadata so the evaluator does not need a second sidecar index.

- **Person crops are split the same way as the attribute service.**
  `person_upper_color` uses the upper half of the annotated person box,
  `person_lower_color` uses the lower half, and `vehicle_color` uses the full
  box. That keeps bake-off data aligned with the deployed inference path.

- **The evaluator prefers Triton but falls back to ONNX Runtime.**
  The task asked for Triton gRPC when available and CPU-pilot compatibility
  otherwise. The script tries Triton first and includes the Triton error in the
  final failure message if it has to fall back or fails completely.

- **The report is explicitly a proxy, not fabricated benchmark output.**
  `data/eval/attribute/manifest.json` is still absent in this repo state, so I
  did not invent accuracy numbers. The report applies the protocol fallback
  honestly: no clear winner means choose `ResNet-18`.

## Patterns followed

- lazy imports for optional runtime dependencies (`numpy`, `Pillow`, `mlflow`,
  `tritonclient.grpc`, `onnxruntime`)
- urllib-based HTTP helpers matching the existing CVAT scripts
- CLI boundary catches exceptions and exits with a short operational error
- MLflow tags follow the existing bake-off pattern:
  - `bakeoff.phase=attribute`
  - `bakeoff.candidate=<candidate>`
  - `bakeoff.safe_default=true|false`

## Gotchas

- `run_attribute_bakeoff.py` intentionally fails if any of the 10 colors are
  missing from the manifest counts. That is stricter than “some data exists”,
  but it matches the protocol’s requirement for per-color accuracy across the
  full vocabulary.

- The CVAT prep script assumes the `attribute-eval` project uses the schema from
  `scripts/annotation/setup_cvat_projects.py`:
  - `vehicle_color`
  - `person_upper_color`
  - `person_lower_color`
  If the live project drifts from that schema, the export will fail loudly.

- The evaluator applies the same softmax-plus-threshold logic as the current
  attribute service, even though the Triton output tensor is named
  `probabilities`. That is deliberate: it preserves consistency with the live
  service behavior.
