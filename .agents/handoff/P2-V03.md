# P2-V03 — Event Engine

## What was built

- `services/event-engine/main.py` — Kafka consumer for `tracklets.local`, per-track FSM routing, periodic timer checks, best-effort camera-motion tracking, graceful shutdown, Prometheus startup
- `services/event-engine/state_machine.py` — per-track FSM with entered/exited/stopped/loitering logic, duration-event open/close tracking, ROI + loitering polygon parsing, point-in-polygon helpers, generic event trigger dataclasses
- `services/event-engine/event_emitter.py` — Event protobuf construction, `x-proto-schema` header publishing to `events.raw`, asyncpg `INSERT` / `UPDATE` writes to the `events` table
- `services/event-engine/config.py` — Pydantic Settings v2 YAML loader with `EVENT_` env prefix
- `services/event-engine/metrics.py` — emitted events counter, active FSM gauge, state-transition counter, DB write latency histogram, consumed tracklets counter
- `services/event-engine/gen_proto.sh`, `requirements.txt`, `Dockerfile`, `mypy.ini`, `README.md`
- `services/event-engine/tests/test_state_machine.py` — 8 FSM tests covering entered_scene, stopped, loitering, resume close, termination close, class applicability, bicycle support
- `services/event-engine/tests/test_event_emitter.py` — 6 emitter tests covering point-in-time/duration protobuf mapping, closed-update mapping, DB write parameters, Kafka header wiring

## Key decisions and why

- **Duration events keep one event_id across open and close**
  - `stopped` and `loitering` open with a client-side UUID, then the close update reuses the same `event_id` for both Kafka and PostgreSQL.
  - This matches the topic contract for `events.raw` keyed by `event_id`.

- **`stopped` / `loitering` start times are backdated to condition start, not threshold-crossing time**
  - A stopped event emitted after 3 seconds starts at the first stationary observation, not 3 seconds later.
  - Same for loitering: start time is when the person entered the zone continuously, not when the 30-second threshold elapsed.

- **Camera config parser is intentionally tolerant**
  - The repo says ROI and loitering zones live in `cameras.config_json`, but there is no canonical JSON schema for those polygons yet.
  - The parser accepts common list-of-pairs and list-of-`{x,y}` forms so the service can work now without freezing the repo into an unannounced format.

- **Camera-level motion events are derived from tracklet activity as a proxy**
  - The taxonomy defines `motion_started` / `motion_ended` from the edge motion detector, but no such signal is published into the core pipeline yet.
  - To keep P2-V03 functional on current inputs, `main.py` infers camera activity from `NEW` / `ACTIVE` tracklet flow per camera.

## Patterns established

- **Event trigger dataclasses separate detection from emission**
  - `state_machine.py` only produces `EventTrigger` objects.
  - `event_emitter.py` owns protobuf construction and DB writes.

- **Per-camera config is loaded lazily from PostgreSQL and cached in-process**
  - No extra topology or config service dependency was added.

- **Point-in-time events use a single insert path**
  - `entered_scene`, `exited_scene`, `motion_started`, and `motion_ended` are emitted as immediately closed records with `end_time == start_time` and `duration_ms == 0`.

## Gotchas

- **Exact frame-motion semantics are still blocked on an upstream motion signal**
  - The current implementation emits camera-level motion events from tracklet activity timing, not from the edge pixel-change metric itself.
  - If a camera has motion that never results in a tracklet, the event engine will not see it.

- **`cameras.config_json` polygon shape is still a convention, not a formal contract**
  - The service now supports several shapes, but future agents should standardize one repo-wide schema before other components depend on it.

- **Open duration events are in-memory state**
  - If the service restarts, open `stopped` / `loitering` events are not restored from PostgreSQL.
  - The current task did not require checkpoint/restore for FSM state.

- **No event dedup / suppression layer beyond the FSM itself**
  - The core pipeline component summary mentions dedup+suppression, but this task prompt did not define a separate suppression policy.
  - The implementation therefore stays with deterministic FSM rules only.
