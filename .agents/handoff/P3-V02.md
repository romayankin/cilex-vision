# Handoff: P3-V02 — Shadow Deploy Tooling

## What was built

- `infra/triton/shadow/README.md` documents the shadow model repository purpose, layout, lifecycle, and rollback flow for Triton EXPLICIT mode deployments
- `infra/kafka/shadow-topics.yaml` defines the three isolated shadow topics:
  - `detections.shadow`
  - `tracklets.shadow`
  - `embeddings.shadow`
- `scripts/shadow/deploy_shadow_model.py` implements a standard-library-only deployment helper that:
  - copies `model.plan` into `{model_repo}/{model_name}/{version}/`
  - loads the model through Triton's EXPLICIT HTTP API
  - verifies the candidate version metadata endpoint
  - supports rollback with `--unload` by removing the version directory, unloading the model, and reloading the remaining versions
- `scripts/shadow/shadow_inference_worker.py` implements a detector-shadow worker that:
  - consumes `frames.decoded.refs` with consumer group `shadow-detector-worker`
  - downloads decoded frames from MinIO
  - runs Triton gRPC inference against an explicit model version
  - tracks per camera with ByteTrack-style logic
  - publishes only to `detections.shadow` and `tracklets.shadow`
  - exposes Prometheus metrics:
    - `shadow_detections_total`
    - `shadow_inference_latency_ms`
    - `shadow_frames_consumed_total`
    - `shadow_publish_errors_total`
    - `shadow_detection_confidence`
- `scripts/shadow/compare_shadow.py` compares production vs shadow Prometheus metrics over a requested window and writes a Markdown pass/fail report, with optional Pushgateway publishing for summary metrics

## Key decisions

- **Detector-focused worker first**: the task prompt's worker contract is detector/tracker-oriented, so the shadow worker mirrors the production detector path and publishes isolated detection + tracklet outputs. The shadow embeddings topic is defined now for future Re-ID shadowing but is not emitted yet
- **Explicit Triton version pinning**: the shadow worker always passes `model_version` to Triton gRPC instead of relying on `latest`. This avoids accidental promotion semantics and matches the Stage 2 SOP
- **`urllib.request` for management and comparison HTTP**: both the Triton deployment helper and Prometheus comparison script stay on the stdlib HTTP stack to match existing repo script patterns
- **Graceful degradation on missing metrics**: `compare_shadow.py` marks unavailable comparisons as `WARN` instead of pretending the data exists. This is especially important for confidence histograms because the current production inference worker still does not export a canonical confidence histogram

## Patterns followed

- CLI-first Python scripts with `argparse` and `if __name__ == "__main__"`
- No production-topic writes from shadow tooling
- Protobuf serialization matches existing `Detection` and `Tracklet` contracts, including `x-proto-schema` Kafka headers
- Triton lifecycle follows ADR-005 EXPLICIT mode:
  - copy version
  - load via HTTP API
  - verify
  - unload by removing the candidate version and reloading remaining versions

## Gotchas

- **Prometheus scrape wiring is still external**: the worker exposes metrics on its own port, but no monitoring config was changed here. `compare_shadow.py` assumes those metrics are already being scraped
- **Confidence comparison is partial today**: the shadow worker exports `shadow_detection_confidence`, but the production inference worker still lacks `inference_detection_confidence`, so confidence drift will show as `WARN`
- **Rollback is version-directory-based**: Triton's unload API is model-scoped, not version-scoped. The unload helper therefore removes the candidate version from the repository, unloads the model, and reloads the remaining versions
- **`embeddings.shadow` is reserved only**: this task defines the topic contract but does not yet publish shadow embeddings. Re-ID shadow rollout still needs the OSNet crop + publish path

## Open questions

- Whether the production inference worker should eventually gain an in-process duplicate-request shadow mode keyed off `INFERENCE_SHADOW_MODEL_VERSION`, as described in the rollout SOP, instead of relying on a separate worker process
- Whether a dedicated shadow Grafana dashboard should be added so operators do not need to rely on the comparison script alone for 24-48 hour watch windows
