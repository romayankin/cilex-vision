# Handoff: P4-E01 — 50-100 Camera Load Test

## What I built

- `scripts/load-test/replay_streams.py`
  - FFmpeg-backed replay tool for 50-100 virtual cameras
  - scans a source recording directory, extracts replayable JPEG frame caches, assigns them round-robin to virtual camera IDs, uploads frames to MinIO, and publishes `FrameRef` protobufs to Kafka at real-time rate
- `scripts/load-test/measure_e2e.py`
  - Prometheus + direct-probe measurement harness
  - collects per-stage histogram quantiles, direct Query API timings, throughput, lag, GPU/resource signals, MinIO bucket growth, optional DB probe latency
  - writes raw JSON snapshots plus an aggregated CSV summary
- `scripts/load-test/chaos_scenarios.py`
  - extended chaos runner for scale validation
  - includes Kafka outage, consumer pause, WAN disconnect, TimescaleDB restart, MinIO node failure skip/execute logic, and optional GPU overload via operator-supplied command
  - writes structured JSON output for report generation
- `scripts/load-test/generate_report.py`
  - generates `docs/evaluation-results/load-test-100cam-report.md`
  - applies NFR gates, builds latency/throughput/resource/chaos tables, identifies bottlenecks, and compares the run to the measured P3 cost model inputs
- `docs/evaluation-results/load-test-100cam-report.md`
  - committed template matching the generated report structure

## Key decisions

- **Kept the P2-E03 harness intact**
  - I added scale-specific tooling beside the existing 4-camera stress harness instead of rewriting it.
  - This avoids breaking the pilot-oriented workflow while still reusing shared patterns like local `sys.path` injection and JWT helpers.

- **Replay is FFmpeg extraction + FrameRef publication, not RTSP serving**
  - The deployed central ingest contract already expects `FrameRef` messages plus MinIO object storage.
  - To stay aligned with the actual repo interfaces, the new replay script uses FFmpeg to sample frames from recordings and pushes them through that contract at scale.
  - This is operationally useful, but it is not a full edge/RTSP emulator.

- **Missing observability still fails the report**
  - The report does not silently substitute optional metrics for canonical ones.
  - If `e2e_latency_ms`, GPU metrics, cAdvisor metrics, or canonical lag metrics are missing, the report surfaces that explicitly.

- **Cost comparison validates drivers, not only monthly dollars**
  - The generator compares the run against P3-X03 measured assumptions such as FPS per camera, active tracks per camera, cameras per GPU, and hot-storage growth.
  - It also includes the predicted P50 monthly cost row, but leaves true “actual monthly cost” as `n/a` unless billing data exists.

- **GPU overload is deployment-aware and intentionally external**
  - There is no standard additive overload command in the repo.
  - The chaos script therefore supports `--overload-command` / `--overload-clear-command` instead of guessing how the operator wants to inject extra traffic.

## Patterns followed

- Standalone scripts under `scripts/load-test/` with `SCRIPT_DIR` import bootstrapping and `if __name__ == "__main__"` entry points
- Standard-library-first HTTP access through existing `common.py` helpers
- Docker-native reversible chaos matching the Phase 2 harness approach
- Markdown report structure aligned to the existing stress-test reporting style, but expanded for 100-camera scale and cost-model validation

## Gotchas

- **`replay_streams.py` depends on FFmpeg and real source media**
  - It fails fast if `ffmpeg` is not installed or the source directory has no supported video files.

- **Some report sections may be intentionally sparse**
  - Disk/network/GPU sections depend on scrape targets and exporters that are not guaranteed in the current pilot Prometheus config.
  - The script handles that by reporting `n/a` instead of inventing values.

- **The MinIO storage-growth estimate is extrapolated**
  - The report estimates steady-state `frame-blobs` storage from observed bucket growth over the measurement window.
  - This is useful for comparison, but still a projection rather than billing truth.

- **Chaos JSON is optional to the report generator**
  - If operators skip `chaos_scenarios.py`, the report will still generate, but the chaos section will show that no structured results were provided.

- **The 4-hour sustained requirement is checked in the report**
  - The tooling does not hard-stop an operator from running a shorter measurement window.
  - Instead, the report marks that requirement as failed if `duration_s < 14400`.
