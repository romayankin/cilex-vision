# Handoff: P0-D06 — Ingress Bridge Design Specification

## What was built
- docs/adr/ADR-001-ingress-bridge.md — full ADR replacing the stub (Context → Decision → Consequences)
- docs/diagrams/ingress-bridge-flow.mermaid — flowchart showing normal, Kafka outage (spool), and drain flows

## Key decisions
- **At-least-once delivery**, not exactly-once. Downstream dedup by idempotent key. Exactly-once would require Kafka transactions coordinated with NATS ack — too much latency/complexity for the benefit.
- **Idempotent key format**: `{site_id}:{camera_id}:{capture_ts_epoch_us}:{frame_seq}`. Partition key is `camera_id` alone (matches kafka-contract.md).
- **NVMe spool: 50 GB max**, resume at 40 GB (80%). Spool drain at 80% of per-site rate limit so live traffic isn't starved.
- **Per-site rate limit: 500 msg/s default**. Replay lane gets 50% of the site limit.
- **Live vs. replay lane separation**: same Kafka topic, different NATS subjects, different priority. Replay pauses when live traffic approaches rate limit.
- **`core_ingest_ts` stamped here** — this is the first core boundary. Never overwrite `edge_receive_ts` or `source_capture_ts` (per time-sync-policy.md).
- **Blob offload threshold: 100 KB**. Only auxiliary payloads — `frame_uri` already points to MinIO.

## Patterns established
- Failure modes documented as a table with Detection/Behavior/Recovery columns
- Prometheus metrics organized by type (counters, histograms, gauges) with labels and descriptions
- Health endpoints: `/health` (liveness), `/ready` (readiness), `/metrics` (Prometheus)
- All metrics prefixed with service name: `bridge_`
- DLQ handling: 3 retries on NATS, then publish to `frames.dlq.>{site_id}`

## Gotchas for next agent
- The spool is per-instance and NOT replicated. If an instance crashes, its spool is orphaned until Kubernetes restarts the pod and the persistent volume remounts.
- The Kafka partition key is `camera_id` (matching kafka-contract.md §3.1), but the full idempotent key is the compound `site:camera:ts:seq` string. Don't confuse the two.
- Schema Registry is a hard dependency for validation. If it's down, the bridge should cache last-known schema — this is mentioned in Risks but not deeply specified. The P1-V02 dev agent should implement a local schema cache with TTL.
- The config example uses Pydantic BaseSettings loaded from YAML — follow the pattern in CONVENTIONS.md §Configuration.
- Rate limiting applies AFTER validation but BEFORE produce. This means invalid messages don't count against the rate limit.
- `core_ingest_ts >= edge_receive_ts` is validated but violations are logged as warnings, not rejections — clock drift is a monitoring concern, not a data integrity gate.

## Open questions
- Exact NATS subject hierarchy for multi-site is `frames.{lane}.{site_id}` — but this may evolve when more message types flow through the bridge (currently only FrameRef). P1-V02 should design for extensibility.
- The security-design.md is still a stub (P0-D08). The bridge references mTLS for NATS and SASL_SSL for Kafka — the exact cert paths and credential rotation will be defined there.
