# Handoff: P2-E03 — End-to-End Stress Test

## What I built

- `scripts/load-test/run_stress_test.py`
  - phase-aware orchestrator with 3 phases: ramp-up, sustained load, ramp-down
  - continuous Prometheus collection
  - optional chaos schedule during the sustained window
  - final Markdown report generation
- `scripts/load-test/load_generator.py`
  - publishes synthetic `FrameRef` protobufs to `frames.sampled.refs`
  - uploads matching JPEG source frames to MinIO `frame-blobs`
  - generates authenticated query load for `/detections`, `/tracks`, `/events`
  - supports `--replay-frame-dir` for operator-supplied real frames
- `scripts/load-test/metric_collector.py`
  - Prometheus instant-query collector for stage latencies, throughput, resource usage, lag, and errors
  - retains in-memory snapshot history for sustained-phase reporting
  - includes explicit fallbacks where the repo does not yet expose the exact NFR metric
- `scripts/load-test/chaos_runner.py`
  - reversible Docker-native chaos actions:
    - stop/start Kafka broker container
    - stop/start consumer-group service container
    - disconnect/reconnect the edge-agent container from the pilot Docker network
  - records recovery time and a simple row-count-based data-loss signal through the Query API
- `scripts/load-test/generate_stress_report.py`
  - emits `docs/evaluation-results/stress-test-report.md`
  - sections: configuration, NFR pass/fail, per-stage latency, throughput, resource usage, lag, chaos, bottlenecks, cost model comparison
- `scripts/load-test/common.py` and `scripts/load-test/models.py`
  - shared utilities for JWT cookie generation, Prometheus / Query API HTTP calls, lazy `FrameRef` proto loading, and shared dataclasses
- `docs/evaluation-results/stress-test-report.md`
  - committed placeholder/template with the final report structure

## Key decisions

- **Synthetic frames by default, replay frames optional**
  - The repo does not include a bundled media corpus.
  - Default mode therefore generates JPEGs on the fly and pushes them through MinIO + Kafka so decode and inference see the real transport path.
  - Because those synthetic frames may not trigger realistic detections, the harness also supports `--replay-frame-dir` for operator-supplied image sequences when event / clip / MTMC-heavy runs are needed.

- **Missing observability is surfaced as a failed validation**
  - The taxonomy names metrics like `e2e_latency_ms`, `inference_fps{camera_id}`, and canonical `kafka_consumer_lag{group,topic}`.
  - The current services do not expose all of those exact metrics.
  - The report therefore treats absent canonical metrics as `FAIL` and only uses fallback signals for operator context, never as silent substitutes.

- **Chaos scenarios are Docker-native and cleanup-first**
  - This task is eval-only, so I did not change infra playbooks or compose files.
  - `chaos_runner.py` uses `docker stop/start` and `docker network disconnect/connect` only, with restore logic in every failure path.

- **The report uses sustained-phase snapshots**
  - Ramp-up and ramp-down are intentionally excluded from steady-state summaries when enough snapshots exist.
  - Per-stage latency rows report the worst observed rolling quantiles during the sustained phase.

## Patterns established

- Standalone scripts under `scripts/load-test/` can share local helpers by injecting the script directory onto `sys.path`, while still keeping `if __name__ == "__main__"` entry points on every deliverable script.
- Query API auth for eval tooling can be generated locally with an HS256 cookie using the configured JWT secret; no production auth code changes are required.
- Prometheus collection is strict about missing data and preserves raw snapshots for later Markdown summarisation.

## Gotchas

- **Synthetic frames are transport-realistic but vision-unrealistic**
  - They are enough to exercise Kafka, MinIO, decode, query traffic, and recovery behavior.
  - They are not enough to guarantee realistic detections, events, clips, or MTMC matches.
  - Use `--replay-frame-dir` with representative images for event-heavy validation.

- **Current pilot deployment wiring still lags the Phase 2 service list**
  - `infra/docker-compose.pilot.yml` currently scrapes only the original pilot services plus Triton.
  - If operators expect attribute, event-engine, clip-service, or MTMC metrics in Prometheus, those services must actually be deployed and scraped first.

- **The canonical NFR metrics are not all implemented yet**
  - `inference_fps{camera_id}` and `e2e_latency_ms` are specified in `docs/taxonomy.md` but are not currently exported by the relevant services.
  - The harness reports that gap explicitly instead of hiding it.

- **Query API health does not expose active stream count**
  - The taxonomy expects a health-based validation for 4 pilot cameras, but the current `/health` endpoint only returns `{"status": "ok"}`.
  - The harness therefore validates configured camera fan-out, not a true active-stream count.

## Open questions

- Should a future task add a small committed replay corpus or a documented external dataset contract for realistic end-to-end stress runs?
- Should the missing canonical NFR metrics (`e2e_latency_ms`, `inference_fps{camera_id}`, canonical `kafka_consumer_lag`) be implemented in production services before pilot sign-off?
