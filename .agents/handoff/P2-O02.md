# P2-O02 — Storage Tiering

## What I built

- `infra/minio/lifecycle-policies.json`
  - single source of truth for MinIO retention and tier placement
  - covers `frame-blobs`, `decoded-frames`, `event-clips`, `thumbnails`,
    `raw-video`, `archive-warm`, `debug-traces`, and `mtmc-checkpoints`
- `infra/minio/apply-lifecycle.py`
  - idempotent operator script that loads the policy file, creates buckets if
    needed, and applies lifecycle rules with `mc`
  - supports local `mc` or Docker fallback via `minio/mc`
  - supports `--dry-run`
- `infra/minio/storage-report.py`
  - operator report script that queries MinIO bucket usage with `mc`
  - groups usage by hot / warm / cold tier
  - projects monthly cost using the repo cost-model rates
  - flags buckets above 80% of a configurable capacity threshold
- `infra/grafana/dashboards/storage-tiering.json`
  - storage-tiering dashboard for MinIO bytes, object counts, tier mix, ILM
    activity, and projected monthly cost
- `infra/prometheus/rules/storage-tier-alerts.yml`
  - MinIO bucket-capacity, drive-usage, and lifecycle-stall alerts

## Key decisions

- **Lifecycle policies now live in one repo-level JSON file**
  - The older Ansible vars only expressed a subset of the intended retention
    rules and there was no committed artifact operators could inspect directly.
  - The new JSON file is the canonical tiering definition for both the apply
    script and the reporting script.

- **`mc` import payloads use the current MinIO JSON shape, not raw XML**
  - The task prompt referenced S3 lifecycle XML, but the current `mc`
    lifecycle-import flow accepts JSON documents with S3 field names
    (`Rules`, `Filter`, `Expiration`, etc.).
  - The apply script translates the repo’s lower-case source file into the
    `mc` import format and falls back from `mc ilm rule import` to
    `mc ilm import` for older clients.

- **Cold-tier cost is a warm-tier proxy**
  - `scripts/cost-model/params.yaml` defines `hot_object` and `warm_object`,
    but no separate `cold_object` rate.
  - The storage report and dashboard therefore price cold buckets with the warm
    rate until the cost model is extended.

- **MinIO metrics were wired as part of the task**
  - The repo had no MinIO Prometheus scrape jobs and MinIO deployments were not
    configured for public metrics scraping.
  - I added `MINIO_PROMETHEUS_AUTH_TYPE: public` to the repo’s MinIO
    deployments and added `cluster`, `bucket`, `resource`, and `node` scrape
    jobs to the shared Prometheus configs so the new dashboard and alerts have
    data.

## Patterns followed

- `argparse` CLIs with explicit defaults and fail-fast error messages
- stdlib-only Python for operator scripts
- `subprocess.run()` for external `mc` calls
- JSON / YAML artifacts placed in the existing `infra/` layout used by
  Grafana, Prometheus, and Ansible
- additive infra changes only; pilot and multi-node paths remain intact

## Gotchas

- `infra/ansible/playbooks/deploy-minio.yml` now exposes MinIO metrics and the
  shared vars now include the full bucket/rule inventory, but the playbook
  still does not execute `infra/minio/apply-lifecycle.py`. Operators must run
  the apply script after MinIO bootstrap until the playbook is extended.
- The Docker fallback for the MinIO scripts uses `--network host` so `localhost`
  MinIO endpoints work from inside the `minio/mc` container. That is correct
  for this Linux-oriented repo setup, but worth keeping in mind on non-Linux
  operator workstations.
- Local compose now creates the additional storage-tiering buckets
  (`raw-video`, `archive-warm`, `mtmc-checkpoints`, plus `decoded-frames`
  consistently across both compose files). Existing local volumes will only
  pick that up after `minio-init` reruns.
