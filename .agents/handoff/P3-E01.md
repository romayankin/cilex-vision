# P3-E01 — Retraining Validation

## What I built

- `scripts/evaluation/regression_checker.py`
  - pure dataclasses and helpers for slice-by-slice regression detection
  - compares two metric dictionaries, applies a configurable threshold, and
    renders a reusable Markdown gate table
- `scripts/evaluation/validate_retrained_model.py`
  - full retrained-vs-production validation harness
  - loads metrics from either MLflow runs or exported JSON from
    `scripts/training/evaluate.py`
  - normalizes all required offline-qualification slices into one canonical
    schema
  - writes local artifacts:
    - `candidate_metrics.json`
    - `baseline_metrics.json`
    - `regression_report.json`
    - `validation-report.md`
    - `validation_summary.json`
  - logs the validation run to MLflow experiment `retraining-validation`
  - exits `0` for GO and `1` for NO-GO
- `scripts/training/Makefile`
  - added `validate` target that calls the new harness and accepts additional
    CLI arguments via `VALIDATE_ARGS`

## Key decisions

- **Gate only on the offline-qualification slices from the SOP.**
  The hard gate uses:
  - `map50`
  - `map50_95`
  - `small_object_ap`
  - `night_ap`
  - `operational_precision`
  - `operational_recall`
  - `operational_f1`
  - `ap_<class>` for all 7 detector classes

  Latency and throughput are reported when both sources contain them, but they
  are not part of the absolute regression gate because the prompt explicitly
  defines the required gate slices as accuracy / operational metrics.

- **Normalize both MLflow and JSON inputs to the same metric names.**
  Existing scripts log a mix of names:
  - `mAP@0.5`
  - `mAP@0.5:0.95`
  - `small_object_AP`
  - `per_class_ap_person`
  - `AP_person`
  - `best_mAP@0.5:0.95`

  The harness maps these into canonical internal names such as `map50`,
  `map50_95`, and `ap_person` so the checker is not tied to one storage format.

- **Keep the regression checker pure.**
  `regression_checker.py` has no MLflow, file, subprocess, or path logic. That
  keeps it reusable for future validation/reporting tasks and easy to test.

- **Use `models/latest_run_id.txt` as a candidate convenience only.**
  The training pipeline already writes that file. The validator will use it if
  `--candidate-run-id` is omitted, but it does not guess the production
  baseline because there is no canonical baseline pointer in the repo today.

## Patterns followed

- CLI-first `argparse` scripts with `if __name__ == "__main__"` wrappers
- lazy import for optional `mlflow`
- manual JSON-safe serialization for local artifacts
- short operational `SystemExit` messages on failure
- Makefile target kept thin; the Python harness owns the logic

## Gotchas

- A real validation run still requires MLflow availability because the task
  requires every validation result to be logged to the `retraining-validation`
  experiment.
- The new `validate` Make target is intentionally generic. Operators still need
  to pass either:
  - `BASELINE_RUN_ID=<run>`
  - `VALIDATE_ARGS="--baseline-json ..."`
  - or equivalent explicit CLI args
- The harness fails fast if any required slice is missing from either source.
  That is deliberate: silently skipping a class or condition metric would
  weaken the rollout gate.
