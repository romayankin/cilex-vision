# Handoff: P1-O02 — Observability Dashboards & Alerts

## What was built

### Grafana Dashboards (5 files in `infra/grafana/dashboards/`)

- **stream-health.json** (uid: `stream-health`, 7 panels) — Camera uptime % (stat), active/offline camera count, edge buffer fill (gauge), decode errors/min, NATS publish latency p95, motion duty cycle, motion vs static frames/s. Template variables: `$datasource`, `$camera_id`.
- **inference-performance.json** (uid: `inference-performance`, 9 panels) — Frames consumed FPS, detections/s by class (stacked), detection latency p50/p95/p99, embedding latency p50/p95/p99, GPU VRAM % (gauge with 85%/95% thresholds), Triton queue delay, active tracks per camera, tracks closed/min, consumer lag. Template variables: `$datasource`, `$camera_id`.
- **bus-health.json** (uid: `bus-health`, 9 panels) — Kafka consumer lag (bulk collector), messages consumed/s by topic, NATS consumer lag (bridge), bridge throughput (received vs produced), spool fill % (gauge with 80%/95% thresholds), spool depth messages, bridge NATS-to-Kafka latency p95, schema rejections & DLQ, Kafka producer in-flight. Template variables: `$datasource`, `$site_id`.
- **storage.json** (uid: `storage`, 9 panels) — Rows written/s by table, write latency p99, write errors/min, rows staged, batch size average, duplicates skipped/min, messages rejected/min, decode service throughput pipeline, decode consumer lag. Template variables: `$datasource`, `$table`.
- **model-quality.json** (uid: `model-quality`, 8 panels) — Detection count per class/hour, total detections by class bar gauge (1h), detection rate by camera, active tracks, track close rate/min (ID switch proxy), track turnover ratio (closed/active), inference pipeline throughput (frames vs detections), publish errors by topic. Template variables: `$datasource`, `$camera_id`, `$object_class`.

- **stream-alerts.yml** (5 rules) — CameraOffline (uptime=0, 5m), DecodeErrorRateHigh (>100/min, 2m), MotionDutyCycleAnomaly (<1%, 15m), NatsPublishLatencyHigh (p95>500ms, 5m), EdgeBufferFillHigh (>500MB, 5m).
- **inference-alerts.yml** (5 rules) — InferenceLatencyP99High (>200ms, 5m), InferenceVramHeadroomLow (<15%, 5m), EmbeddingLatencyP99High (>500ms, 5m), InferenceConsumerLagHigh (>5000, 5m), InferencePublishErrors (rate>0, 5m).
- **bus-alerts.yml** (6 rules) — KafkaConsumerLagHigh (>10000, 5m), KafkaConsumerLagCritical (>50000, 2m), BridgeSpoolFillWarn (>80%, 2m), BridgeSpoolFillCritical (>95%, 1m), BridgeSchemaRejectionRate (>0.1/s, 5m), BridgeDLQPublishing (rate>0, 5m).
- **storage-alerts.yml** (5 rules) — TimescaleDBWriteErrors (rate>0, 2m critical), BulkWriteLatencyHigh (p99>1s, 5m), BulkRowsStagedHigh (>10000, 5m), DecodeServiceErrorRateHigh (>0.5/s, 5m), DecodePublishErrors (rate>0, 5m).

**Total: 24 new alert rules + 8 existing (triton-alerts.yml: 6, clock-drift.yml: 2) = 32 across 7 files.**

## Key decisions and why

- **All metric names reference actual `metrics.py` files**: Every PromQL expression uses metric names defined in the services' `metrics.py` files — `edge_*`, `inference_*`, `bridge_*`, `bulk_*`, `decode_*`, `nv_*`, `clock_skew_ms`. No invented metrics.
- **No duplication with existing alert files**: `triton-alerts.yml` (6 rules for VRAM, queue delay, error rate, model readiness) and `clock-drift.yml` (2 rules for skew >500ms, >2000ms) are untouched. `InferenceVramHeadroomLow` in inference-alerts.yml uses headroom framing (100% - used < 15%) which is mathematically equivalent to TritonVramWarn but framed for ML-team audiences. `clock-alerts.yml` adds complementary collector-health and multi-pair alerts, not duplicates of ClockSkewWarn/Critical.
- **Dashboards auto-provisioned via existing config**: `infra/grafana/provisioning/dashboards/dashboards.yml` already provisions from `/var/lib/grafana/dashboards`, and `infra/docker-compose.yml` mounts `./grafana/dashboards` to that path. No provisioning changes needed.
- **Alert rules auto-loaded via existing config**: `infra/prometheus/prometheus.yml` loads `rule_files: ["/etc/prometheus/rules/*.yml"]`, and `infra/docker-compose.yml` mounts `./prometheus/rules` to `/etc/prometheus/rules:ro`. New `.yml` files are picked up automatically.
- **Template variables for drill-down**: Each dashboard includes a `$datasource` variable plus relevant label variables (`$camera_id`, `$site_id`, `$table`, `$object_class`) populated via `label_values()`. This enables per-entity filtering.
- **Track turnover ratio as ID-switch proxy**: There's no direct `id_switches_total` metric (that would require ground-truth evaluation). Instead, `inference_tracks_closed_total / inference_tracks_active` gives a turnover ratio — a high ratio relative to active tracks suggests tracking instability.
- **Confidence distribution not available as histogram**: The inference worker emits `inference_detections_total{object_class}` (a counter), not a confidence histogram. The model-quality dashboard uses detection rate by class as the closest available proxy. A confidence histogram would require adding a new metric to the inference worker.

## Patterns established

- **Alert severity convention**: `warning` for soft thresholds with 5m `for` duration, `critical` for hard thresholds with 1-2m duration. All alerts include `service` label for routing.
- **Dashboard uid convention**: lowercase hyphenated matching the filename without extension (e.g., `stream-health.json` → uid `stream-health`).
- **Panel gridPos layout**: 24-column grid, panels in rows of 2 (12+12) or 3 (8+8+8). Gauges and stats at top, time series below.
- **PromQL conventions**: `rate()[5m]` for counters, `histogram_quantile()` for latency percentiles, `* 60` for per-minute display, `* 3600` for per-hour display.

## Gotchas

- **No Prometheus scrape targets for services yet**: `prometheus.yml` only has a self-scrape target. When services are deployed, scrape targets for the edge agent, inference worker, ingress bridge, bulk collector, and decode service need to be added. The dashboards will show "No data" until those targets exist.
- **Grafana datasource uid must match**: The dashboards use `${datasource}` template variable with `prometheus` as default. This matches the provisioned datasource uid in `infra/grafana/provisioning/datasources/prometheus.yml`. If the uid changes, dashboards need updating.
- **`InferenceVramHeadroomLow` overlaps with `TritonVramWarn`**: Both fire at the same effective threshold (85% used / 15% headroom). This is intentional — different teams may route alerts by `service` label (`inference-worker` vs `triton`). If this causes alert fatigue, inhibition rules can suppress one when the other fires.
- **Clock alerts depend on `clock_drift_check.py` running**: The existing `clock-drift.yml` rules and the new `ClockDriftCollectorDown` alert both depend on the textfile collector cron job that runs `services/monitoring/clock_drift_check.py`. If it's not deployed, both `clock_skew_ms` metrics and the absence alert will be absent.
- **`absent()` in ClockDriftCollectorDown**: Prometheus `absent()` returns 1 when a metric is completely missing. If the collector runs but reports zero pairs, the metric still exists with value 0 — the alert won't fire. This is correct behavior: zero skew is healthy.
- **Dashboard JSON schema version 39**: This matches Grafana 11.x. Older Grafana versions may not support all panel options. The docker-compose uses `grafana/grafana:11.1.3`.

## Open questions or known limitations

- **No MinIO bucket size metrics**: The task prompt mentions "MinIO bucket sizes by tier" but MinIO's Prometheus endpoint (`/minio/v2/metrics/cluster`) exports total storage, not per-bucket sizes. Per-bucket metrics require MinIO's bucket-level metrics feature or a custom exporter. The storage dashboard covers TimescaleDB and pipeline throughput but not MinIO bucket sizes.
- **No confidence distribution histogram**: Adding a `inference_detection_confidence` histogram metric to the inference worker would enable per-class confidence heatmaps in the model-quality dashboard. Currently only detection counts are available.
- **No retention compliance panel**: The storage dashboard shows write rates and errors but doesn't verify that TimescaleDB retention policies (30d) and compression policies (2d) are actually executing. That would require querying `timescaledb_information.jobs` or a custom exporter.
- **No Grafana alerting integration**: These are Prometheus alerting rules, not Grafana-managed alerts. Alert notifications depend on Alertmanager configuration, which is not yet part of the docker-compose stack.
