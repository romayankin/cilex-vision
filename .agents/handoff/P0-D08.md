# Handoff: P0-D08 — Security Design - mTLS & ACLs

## What was built

- `docs/security-design.md` — full security design replacing the stub
- `infra/pki/step-ca-config.json` — step-ca template with JWK + ACME provisioners, authority policy, and 90-day claims
- `infra/pki/bootstrap-site.sh` — site bootstrap script that issues NATS server, edge client, and bridge client certificates
- `infra/nats/nats-server.conf` — NATS TLS + `verify_and_map` authorization template with per-site subject ACL examples
- `infra/kafka/server.properties.template` — Kafka broker TLS/SASL template using `SASL_SSL` + `SCRAM-SHA-256`

## Key decisions you made and why

- Kafka auth is `SASL_SSL` with `SCRAM-SHA-256`, not certificate-CN authorization, because that now matches the resolved repo-wide decision and ADR-001.
- NATS stays on mTLS with certificate-subject identity because site isolation is cleaner with per-site certificates and subject ACLs.
- Bridge NATS credentials are one cert per site instead of a wildcard bridge cert so a site compromise can be revoked independently.
- JetStream streams and durable consumers are assumed to be pre-created by Ops so application principals do not need broad NATS admin rights.
- Signed MinIO URLs stay user-facing GET-only with a 1-hour default TTL so browsers never see raw storage credentials.

## Patterns established that future agents should follow

- Use `User:svc-<service>` as the Kafka SCRAM principal naming convention.
- Tie Kafka ACLs to exact topic names from `infra/kafka/topics.yaml`; do not introduce wildcard ACLs.
- Use NATS subject hierarchy `frames.live.<site_id>.>`, `frames.replay.<site_id>.>`, `frames.dlq.<site_id>.>`.
- Keep NATS client certs SAN-less when CN/subject-based `verify_and_map` identity is required.
- Put security bootstrap automation under `infra/pki/` and broker templates under `infra/nats/` or `infra/kafka/`.

## Gotchas

- `verify_and_map` maps to the certificate subject if SANs are absent, so NATS client cert templates must stay SAN-free for CN-style identity.
- Kafka ACLs depend on SCRAM principals, so rotating TLS broker certs does not rotate Kafka client identities.
- `bootstrap-site.sh` is a bootstrap helper, not a full secret-management system. It assumes `step` is installed and optional deployment uses `ssh` / `scp`.
- The NATS config is a template with a `site-a` example block. Additional sites need copied user blocks and corresponding JetStream resources.

## Open questions or known limitations

- Kafka ACL creation commands are documented by convention in the design, but not yet materialized as an executable automation script.
- The repository still lacks production Ansible or Kubernetes wiring for step-ca enrollment, SCRAM credential provisioning, and secret distribution.
- Internal service-to-MinIO auth remains secret-based rather than mTLS-based; that is acceptable for now but may be tightened later.
