# P3-X03: Updated Cost Model — Handoff

## What was built

3 new files:

1. **`scripts/cost-model/params-measured.yaml`** — Measured parameters from pilot deployment. Same structure as `params.yaml` but with `source` provenance and `original_estimate` fields instead of `REPLACE WITH MEASURED` markers. Includes new sections: `mtmc`, `annotation`, `shadow`. Camera counts updated to [10, 50, 100].

2. **`scripts/cost-model/cost_model_v2.py`** — Extended cost model script. Loads `params-measured.yaml`, computes costs with 4 new cost categories (MTMC infrastructure, tiered storage, annotation pipeline, shadow deployment), generates estimated-vs-measured comparison table, and writes Excel to `artifacts/cost-model/cost-model-v2.xlsx`. Handles all 9 Kafka topics (vs 7 in v1).

3. **`docs/cost-model-v2.md`** — Comparison document with executive summary, parameter delta table, cost projections for 10/50/100 cameras, new cost category breakdowns, storage tiering analysis, and recommendations.

## Key decisions

| Decision | Rationale |
|----------|-----------|
| Separate `params-measured.yaml` instead of updating `params.yaml` | Original params preserved for comparison; v1 model continues to work unchanged. |
| `original_estimate` on every measured field | Enables the comparison table without needing to load both param files. |
| Frame blob retention updated to 7 days (was 30) | Matches the actual MinIO lifecycle policy in `infra/minio/lifecycle-policies.json`. This is the single largest cost reduction. |
| Shadow overhead amortized at 33% (1 month per quarter) | Shadow deploys don't run continuously. Quarterly rollout cadence is a reasonable default. |
| Annotation cost is flat, not per-camera | CVAT hosting and annotator time don't scale with camera count. Mining compute queries debug traces which are already sampled at 2%. |
| Cold storage modeled as debug traces + MTMC checkpoints | Raw-video bucket exists but the platform doesn't centrally store full raw streams. Cold tier volume is small. |
| Handles 9 Kafka topics (added frames.decoded.refs, bulk.detections) | Topics were added in later phases. The v2 model accounts for all current topics. |

## Patterns followed

- Same dataclass patterns as v1: frozen dataclasses for all value types, `CostModelInputs` aggregates everything.
- Same CLI/argparse structure as v1.
- Same Excel generation pattern with openpyxl (lazy import via `require_module`).
- Same `format_table` and `format_float` text rendering.
- `MeasuredNumber` replaces `AssumedNumber` with provenance tracking (`source`, `original_estimate`, `delta_pct`).

## Gotchas

1. **The v1 test `test_load_topic_catalog_matches_repo_contract` asserts 7 topics, but `topics.yaml` now has 9.** Later phases added `frames.decoded.refs` and `bulk.detections`. The v2 model handles 9 topics correctly. The v1 test may need updating.

2. **`cold_object` storage cost uses the warm rate ($0.0125/GB-month) as a proxy.** There is no actual cold-tier pricing in the infrastructure since MinIO doesn't have storage classes. The rate is a placeholder for future tiered storage.

3. **Annotation cost dominates at small scale.** At 10 cameras, annotation ($450/month) is more than all storage costs combined. This is by design — annotation quality doesn't scale down — but operators at pilot scale may want to reduce `daily_mining_hours`.

4. **MTMC infra cost is modeled as Re-ID GPU fraction only.** FAISS memory (~50MB/10k embeddings) and checkpoint storage (~100MB) are informational but negligible in dollar terms. They're included in the model for capacity planning, not cost.

5. **The model does not run the v1 script for direct comparison.** The comparison comes from `original_estimate` fields in the measured params. To get a v1-vs-v2 dollar comparison, run both scripts independently.
