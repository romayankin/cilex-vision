# P3-E02 — Shadow Comparison Dashboard

## What I built

- `infra/grafana/dashboards/shadow-comparison.json`
  - new Grafana dashboard for Stage 2 shadow rollout monitoring
  - includes:
    - production vs shadow detection-count trend
    - shadow-only confidence percentiles
    - class-distribution comparison
    - disagreement-rate stat
    - production vs shadow latency percentiles
    - shadow worker health stats
    - debug-trace drill-down panel
- `infra/prometheus/rules/shadow-alerts.yml`
  - four Prometheus rules:
    - `ShadowDetectionCountDivergence`
    - `ShadowLatencyHigh`
    - `ShadowPublishErrors`
    - `ShadowWorkerDown`

## Key decisions

- **Count-level divergence proxy instead of exact disagreement rate**
  The current metrics surface does not provide per-frame or per-camera matched
  detection diffs, so the dashboard and alerts use the same count-level proxy
  already implied by `scripts/shadow/compare_shadow.py`:
  `abs(prod_rate - shadow_rate) / prod_rate`.

- **Shadow-only confidence panel**
  Production still does not export a detection-confidence histogram. Rather than
  inventing a fake comparison, the dashboard shows shadow p50/p90/p99 only and
  points operators to debug traces for deeper drift investigation.

- **Prometheus job name fixed to `shadow-inference-worker`**
  The worker-availability panel and `ShadowWorkerDown` alert need one concrete
  job label. I chose `shadow-inference-worker` because it is the clearest match
  to the deployed process name. This assumption is documented as a deployment
  prerequisite in `todo_before_deployment.md`.

## Patterns followed

- Grafana dashboard metadata matches existing repo conventions:
  - uid matches filename
  - tags are phase-specific
  - schemaVersion is `39`
  - refresh is `30s`
  - 24-column layout with 12/12 and 6/6/6/6 rows
- PromQL follows existing dashboard/alert conventions:
  - `rate()[5m]` for counters
  - `histogram_quantile()` for latency/confidence percentiles
  - warning alerts use longer windows (`5m` to `10m`)
- No service code changes; this task is provisioning artifacts only

## Gotchas

- The dashboard defines a `$camera_id` variable because that is the repo
  dashboard convention, but the current production and shadow detection counters
  are not labeled by `camera_id`. The variable is therefore only useful for
  drill-down workflows today, not for query filtering inside the main panels.
- `ShadowWorkerDown` only works once Prometheus scrapes the shadow worker using
  job label `shadow-inference-worker`.
- Alert auto-loading already exists through
  `rule_files: ["/etc/prometheus/rules/*.yml"]`, so no Prometheus config change
  was needed in this task.
