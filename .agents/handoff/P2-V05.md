# P2-V05: Search UI & Timeline — Handoff

## What was built

Next.js 14 App Router frontend (`frontend/`) with Tailwind CSS providing four views:

- **Search** (`/search`) — Faceted search across detections, tracks, and events with URL query-param sync, pagination, and result cards showing thumbnails, confidence, attributes.
- **Camera Timeline** (`/timeline/[cameraId]`) — Live-polling (5s) timeline of detections and events for a single camera, with HLS/MP4 video player and optional track detail highlight via `?track=` param.
- **Cross-Camera Journey** (`/journey/[globalTrackId]`) — Sequential camera-visit diagram with numbered nodes, entry/exit times, duration, attribute badges, and thumbnails.
- **Admin** (`/admin`) — Links to 5 Grafana dashboards (stream health, inference, bus, storage, model quality) plus placeholder sections for camera management and system health.

Supporting infrastructure:
- Typed API client (`lib/api-client.ts`) with proxy rewrite to avoid CORS.
- Cookie-based RBAC (`lib/auth.ts`) with 4 roles; null role = dev mode (show everything).
- HLS.js video player with native Safari fallback and direct MP4 support.
- Multi-stage Dockerfile (node:20-alpine, standalone output).

## Key decisions and why

| Decision | Rationale |
|----------|-----------|
| `next.config.js` rewrites for API proxy | Avoids CORS entirely; frontend calls `/api/*`, Next.js proxies to Query API |
| `user_role` plain cookie + `getUserRole()` | httpOnly JWT isn't readable from JS; a hint cookie lets the UI hide/show elements while the API enforces real auth |
| Dev mode (null role = show all) | Allows full UI testing without auth infrastructure running |
| `<img>` instead of `next/image` | Thumbnails come from MinIO with dynamic URLs; `next/image` requires allowlisting remote domains and adds optimization overhead for already-small crops |
| HLS.js dynamic import | Avoids SSR crash (no `window`); falls back to native HLS on Safari |
| URL query-param sync on search | Enables shareable/bookmarkable search URLs; re-runs search on mount if params present |
| Standalone Docker output | Smaller image (~100MB vs ~500MB), only ships server.js + static assets |

## Patterns established

- **Page structure**: `"use client"` at top, hooks for state, `useEffect` for data fetching, loading/error/content pattern.
- **Component composition**: Pages own data fetching; components are pure presentation (props in, JSX out).
- **API client**: All fetch calls go through `apiFetch()` which prepends `/api` and handles JSON parsing. Query param builders use `URLSearchParams`.
- **Polling**: `setInterval` + `useEffect` cleanup for live views (timeline).
- **Filter→search routing**: Search filters determine which API endpoint to call (events if `event_type` set, tracks if `state` set, detections otherwise).

## Gotchas

- `next/image` lint warnings are intentional — suppress with `/* eslint-disable @next/next/no-img-element */` if they become noisy.
- The journey page currently uses generic track data (`getTracks`) as a placeholder. The real implementation needs a dedicated `global_tracks` API endpoint (P3 scope).
- `SearchFilters` uses controlled inputs with `FilterState` — adding a new filter requires updating the type, the form JSX, and the search routing logic in `search/page.tsx`.
- The `clipUrl` state in timeline page only captures the first available clip URL (not updated on re-polls). This is intentional for now to avoid jarring video player resets.

## Open questions

1. **Image optimization**: Should we configure `next/image` with MinIO remote patterns, or keep raw `<img>` for simplicity?
2. **WebSocket for timeline**: The 5s polling works but a WebSocket/SSE push from the Query API would reduce latency and load. Deferred to P3.
3. **Global track API**: Journey view needs a real `/api/global_tracks/{id}` endpoint returning ordered camera visits. Currently mocked with generic track data.
4. **Map integration**: The journey view uses a vertical step diagram. A real floor-plan or map overlay is a P3+ feature.
5. **i18n / a11y**: No internationalization or accessibility audit done. Should be addressed before production.
