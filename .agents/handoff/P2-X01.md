# Handoff: P2-X01 — API Documentation

## What I built

- `scripts/api/generate_openapi.py`
  - Generates `docs/api/openapi.yaml` from the real `services/query-api` FastAPI app without starting lifespan hooks.
  - Patches the generated spec with:
    - cookie-based auth documentation
    - explicit `/metrics` path coverage
    - tag metadata
    - role/error response metadata
    - example query/path/body values
- `scripts/api/generate_postman_collection.py`
  - Converts the generated OpenAPI YAML into a Postman Collection v2.1 JSON file.
  - Groups requests by tag and emits example request bodies / example responses derived from the schema.
- `docs/api/openapi.yaml`
  - committed generated OpenAPI contract for the current Query API route surface
- `docs/api/postman-collection.json`
  - committed generated Postman collection
- `docs/api/README.md`
  - operator-facing guide covering auth, roles, endpoint summary, pagination, errors, signed URLs, and regeneration commands
- `docs/api/examples/search-detections.sh`
- `docs/api/examples/search-events.sh`
- `docs/api/examples/get-journey.py`
  - runnable examples for curl and Python

## Key decisions

- **Spec generation stays tied to the real FastAPI app**
  - `create_app()` can be imported safely without opening DB/MinIO connections because the startup work lives in the lifespan context.
  - That lets the docs stay synchronized with real route metadata instead of maintaining a hand-written contract.

- **Patch the generated spec instead of hand-writing around it**
  - FastAPI already provides the path/query/body/schema structure.
  - The script only adds what the current app wiring does not express automatically:
    - cookie auth
    - public `/metrics`
    - consistent 401/403/404/409/503 response docs
    - example values

- **Document the current API, not the idealized one from the prompt**
  - Response envelopes use `detections`, `tracks`, and `events` arrays, not a generic `items`.
  - `EventResponse` currently exposes `track_id` + `metadata`, not `local_track_id` + `metadata_json`.
  - `thumbnail_url` exists in the track-detail schema but is still `null` in practice.

- **Journey example is a best-effort composition**
  - There is no Query API endpoint for `global_track_links` or a cross-camera journey.
  - The Python example therefore combines:
    - track detail
    - same-window events
    - optional topology context
  - That keeps the example runnable against the current deployment instead of pretending the MTMC journey endpoint already exists.

## Patterns followed

- CLI-first generators with `argparse` and `if __name__ == "__main__"`
- Standard-library-first implementation, except for `yaml` and `httpx` where those were already part of the repo/tooling patterns
- Generated artifacts committed alongside their generator scripts
- No service-code changes required; docs are driven from route inspection and current schema definitions

## Gotchas

- `/metrics` is a mounted Prometheus ASGI app, so it does not appear in FastAPI's OpenAPI output automatically. The generator inserts it explicitly.
- The auth dependency reads cookies manually, so FastAPI does not emit a cookie security scheme on its own. The generator adds `cookieAuth`.
- `camera_scope` is only enforced on `/detections`, `/tracks`, and `/events` today. `/topology/*` and `/debug/*` are role-gated but not scope-filtered; the docs call that out and `todo_before_deployment.md` now records it.
- There is still no token minting endpoint in Query API itself. The README therefore documents external/admin-issued JWT usage and a local PyJWT dev flow instead of a nonexistent login route.
