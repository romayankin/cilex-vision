# P4-E02 — Zone Benchmark

## What I built

- `scripts/evaluation/generate_zone_dataset.py`
  - deterministic synthetic dataset generator for zone-sharding evaluation
  - builds linear multi-zone camera layouts, marks boundary cameras, creates intra-zone and adjacent-zone identities, emits L2-normalized 512-d embeddings, and writes a JSON payload with identities plus evaluation-ready ground truth pairs
- `scripts/evaluation/zone_benchmark.py`
  - offline benchmark harness that compares:
    - full-site matching with one global search pool
    - zone-sharded matching with per-zone search, boundary-camera-only cross-zone search, and the relaxed `0.55` cross-zone threshold from P4-V01
  - computes Re-ID quality via `scripts/evaluation/reid_metrics.py`
  - measures per-query search latency and proxy index memory growth
  - logs metrics and artifacts to MLflow experiment `zone-benchmark`
  - writes local artifacts:
    - `benchmark_results.json`
    - `zone_benchmark_report.md`
- `docs/evaluation-results/zone-benchmark-report.md`
  - committed report template matching the generated artifact structure

## Key decisions

- **Reused the existing Re-ID evaluation core**
  - The benchmark converts synthetic identities into `IdentityGroup` / `PredictedAssociation` inputs and delegates Rank-1, Rank-5, mAP, FP/FN, precision, and recall to `reid_metrics.py`.
  - This keeps the Phase 4 zone benchmark aligned with the Phase 2 MTMC evaluation math.

- **Kept the benchmark offline and deterministic**
  - The harness benchmarks the matching policy, not the live service stack.
  - It uses NumPy cosine search over synthetic embeddings so operators can compare full-site vs sharded behavior without a running Kafka / PostgreSQL / MTMC deployment.

- **Matched the actual sharding rules instead of inventing new ones**
  - Intra-zone matching uses the normal `0.65` threshold.
  - Cross-zone recovery only runs for boundary cameras.
  - Cross-zone search is limited to adjacent zones and uses the relaxed `0.55` threshold.
  - Candidate selection is “best match across the per-adjacent-zone best candidates,” which stays close to the P4-V01 intent without pulling the live service into the harness.

- **Benchmarked several zone sizes against the same camera ordering**
  - The dataset keeps a global camera order.
  - `zone_benchmark.py` re-shards that same camera list into chunked zone layouts such as `10`, `25`, or `50` cameras per zone.
  - That makes the comparison about partition size, not about generating a different site each time.

## Patterns followed

- `argparse` CLI with fail-fast validation
- lazy optional imports for `numpy` and `mlflow`
- manual JSON-safe artifact writing
- `if __name__ == "__main__"` entry points with short `SystemExit` errors
- local `SCRIPT_DIR` path injection so the benchmark can import `reid_metrics.py` reliably from anywhere

## Gotchas

- **MLflow is required**
  - The benchmark intentionally fails fast if `mlflow` is missing because the task requires logging every run to experiment `zone-benchmark`.
  - In this environment, `mlflow` was not installed, so only lint / compile / help / dataset-generation checks could be executed locally.

- **This is not a live MTMC service benchmark**
  - The harness does not instantiate `services/mtmc-service/main.py`, FAISS, Kafka, or the boundary-event consumer.
  - It is a deterministic policy benchmark for accuracy/latency/memory deltas under sharding.

- **Zone sizes larger than the camera count collapse to full-site**
  - `build_sharding_layout()` returns a single-zone layout when `zone_size >= camera_count`.
  - The benchmark skips those as redundant rather than double-counting full-site.

- **Boundary accuracy depends on the tested partition**
  - The report recomputes which identities are “cross-zone” for each zone size.
  - An identity can be intra-zone for one partition and cross-zone for another, which is expected.
