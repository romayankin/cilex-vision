# P2-V04 — Clip Pipeline

## What was built

- `services/clip-service/main.py` — Kafka consumer for `events.raw`, closed-event filtering, clip-window calculation, MinIO source-frame lookup, FFmpeg clip extraction, thumbnail generation, PostgreSQL update, Kafka completion publish, and graceful shutdown
- `services/clip-service/clip_extractor.py` — FFmpeg concat-manifest builder and subprocess runner for H.264 baseline MP4 extraction with `+faststart`
- `services/clip-service/thumbnail_gen.py` — thumbnail generation from the middle source frame using Pillow resize
- `services/clip-service/minio_client.py` — MinIO bucket ensure/list/download/upload wrapper with async `to_thread()` calls
- `services/clip-service/db_updater.py` — asyncpg helpers for existing-clip checks, camera→site lookup, and event asset updates
- `services/clip-service/config.py`, `metrics.py`, `mypy.ini`, `gen_proto.sh`, `requirements.txt`, `Dockerfile`, `README.md`
- `services/clip-service/tests/test_clip_extractor.py` — 3 tests covering real FFmpeg extraction, subprocess argument construction, and non-zero exit handling
- `services/clip-service/tests/test_thumbnail_gen.py` — 3 tests covering resize output, middle-frame selection, and empty-input rejection

## Key decisions and why

- **`thumbnail_uri` is stored in `events.metadata_jsonb`**
  - The `events` table has `clip_uri` but no `thumbnail_uri` column.
  - The service updates `metadata_jsonb` with `{"thumbnail_uri": "s3://thumbnails/..."}` and leaves `state` untouched.

- **Asset keys include `site_id` when it can be resolved from PostgreSQL**
  - The prompt and task context disagreed on whether the MinIO key should include `site_id`.
  - The service queries `cameras.site_id` and uses `site_id/camera_id/YYYY-MM-DD/...` when available, falling back to `camera_id/YYYY-MM-DD/...` only if the camera lookup is missing.

- **`archive.transcode.completed` is published as `FrameRef` pointing at the generated clip**
  - The canonical topic contract already binds that topic to `vidanalytics.v1.frame.FrameRef`.
  - The completion message therefore uses `frame_id = event_id`, `frame_uri = clip_uri`, `camera_id = event.camera_id`, `codec = "h264-baseline-mp4"`, and `frame_sequence = 0`.

- **Source frame selection uses MinIO `last_modified`**
  - The current decoded-frame object key is `camera_id/YYYY-MM-DD/frame_id.jpg`, not a timestamped key.
  - Because no separate decoded-frame index exists in PostgreSQL, clip window filtering currently uses date-prefix listing plus MinIO object `last_modified`.

- **Short or point-in-time events are expanded to the minimum configured duration**
  - If `end_time - start_time < min_clip_duration_ms`, the service extends the effective end time before adding pre-roll and post-roll.
  - This keeps point events clip-capable without adding a separate “skip point events” switch.

## Patterns established

- **FFmpeg stays isolated in a single module**
  - `clip_extractor.py` is the only module that shells out to FFmpeg.
  - The rest of the pipeline stays pure Python and async-friendly.

- **MinIO sync SDK is wrapped with `asyncio.to_thread()`**
  - This matches the repo’s existing MinIO usage pattern from attribute-service, decode-service, and MTMC checkpointing.

- **Event dedupe starts with the DB, not the Kafka payload**
  - The service checks the persisted `events.clip_uri` before doing work so replayed or duplicated closed events do not re-extract clips after the DB has already been updated.

## Gotchas

- **Clip window accuracy is only approximate today**
  - The clip service cannot recover exact capture-time windows from decoded-frame keys because those keys contain UUID `frame_id`, not timestamps.
  - MinIO `last_modified` is the best available proxy until a proper decoded-frame index exists.

- **`archive.transcode.completed` is semantically awkward for event clips**
  - The topic schema is `FrameRef`, but the payload is really a clip reference.
  - This works for now because it preserves the repo’s existing topic contract, but it is not a clean long-term abstraction.

- **Query API does not yet sign thumbnail URIs**
  - `thumbnail_uri` is now stored in `metadata_jsonb`, but the current query-api only signs `clip_uri`.
  - Consumers will need either direct MinIO access or a later API enhancement for first-class thumbnail URLs.

## Open questions

- Should the platform add a dedicated decoded-frame metadata table or timestamped object key scheme so clip extraction can use exact frame windows?
- Should event clip completion move to a dedicated clip-oriented topic/schema instead of reusing `archive.transcode.completed` as `FrameRef`?
