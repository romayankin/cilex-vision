# P3-V04: Adaptive Transit Time — Handoff

## What was built

Adaptive transit-time learning from observed MTMC transitions: a SQL migration for the materialized view, a blending/update module, a new API endpoint, and tests.

### New files

1. **`services/db/migrations/add_transit_time_aggregate.sql`** — SQL migration:
   - Adds `transit_distributions JSONB DEFAULT '[]'` column to `topology_edges`
   - Creates `transit_time_stats` materialized view joining `global_track_links` → `local_tracks` → `global_tracks` to compute per-edge, per-class percentile transit times
   - Creates unique index `idx_transit_stats_edge_class` (required for `REFRESH MATERIALIZED VIEW CONCURRENTLY`)

2. **`services/topology/adaptive_transit.py`** — Blending module:
   - `LearnedDistribution` dataclass: one row from the materialized view
   - `UpdateResult` dataclass: summary of a run (edges_updated, distributions_blended, total_learned_rows)
   - `refresh_transit_stats(conn)`: non-blocking `REFRESH MATERIALIZED VIEW CONCURRENTLY`
   - `fetch_learned_distributions(conn)`: reads all rows from the view
   - `blend_distribution(prior, learned, min_samples)`: linear ramp blend — weight = min(sample_count / min_samples, 1.0)
   - `update_edge_distributions(conn, edge_id, blended)`: writes JSONB to `topology_edges`
   - `run_adaptive_update(dsn, min_samples)`: full pipeline (refresh → fetch → blend → update)
   - CLI: `--db-dsn`, `--min-samples`

3. **`services/topology/tests/test_adaptive_transit.py`** — 10 tests:
   - 5 blend tests: 0 samples, 50/100 blend, 100+ samples, object_class preservation, exactly min_samples
   - 1 pipeline integration test with `_FakeConn` mock
   - 4 API endpoint tests: returns stats, custom min_samples, viewer forbidden, empty stats

### Modified files

4. **`services/topology/api.py`** — Added:
   - `TransitStatEntry` Pydantic response model (from_camera, to_camera, object_class, p50/p90/p99_ms, sample_count, blend_weight)
   - `GET /topology/{site_id}/transit-stats` endpoint with `min_samples` query param, auth: admin + operator

## Key decisions

| Decision | Rationale |
|----------|-----------|
| Standard materialized view, not TimescaleDB continuous aggregate | `global_track_links` is a regular table, not a hypertable. Standard `REFRESH MATERIALIZED VIEW CONCURRENTLY` is correct and non-blocking. |
| Linear blend ramp (0 → 1.0 over min_samples) | Simple, predictable. With 0 samples the static prior dominates. With enough Re-ID matches the learned data fully replaces it. No abrupt transition. |
| Default min_samples = 100 | Enough to be statistically meaningful for percentile estimates. Tunable via CLI arg and API query param. |
| Directed transit times (b.start_time > a.start_time) | Preserves directionality — camera A→B may have different transit than B→A (e.g., uphill vs downhill, one-way corridors). |
| Check both directions when updating edges | Topology edges are undirected (camera_a, camera_b), but learned data is directed. The blender checks both (A→B) and (B→A) learned rows, preferring the forward direction. |
| JSON column for distributions, not separate table | Matches the Pydantic model structure — each edge carries its own distributions as a self-contained JSONB array. No join needed at read time. |

## Patterns followed

- asyncpg raw SQL via connection (same as topology api.py and seed.py)
- FakePool / FakeConn test pattern from topology conftest.py
- Pydantic BaseModel for API response models
- FastAPI router with `_require_read` auth dependency (same as GET /topology/{site_id})
- Lazy import for asyncpg (same as seed.py)
- `try: from models` / `except: inline fallback` for standalone CLI use (same as mtmc-service topology_client.py)

## Gotchas

1. **Migration is raw SQL, not Alembic**: The repo has no Alembic setup yet (per task spec). The `.sql` file must be run manually: `psql -f services/db/migrations/add_transit_time_aggregate.sql`

2. **CONCURRENTLY requires unique index**: The unique index on `(from_camera, to_camera, object_class)` must exist before the first `REFRESH MATERIALIZED VIEW CONCURRENTLY`. The initial `CREATE MATERIALIZED VIEW` populates the data, so the index creation after it is fine.

3. **View requires existing data**: The `transit_time_stats` view will be empty until the MTMC service has produced `global_track_links` records. The blender gracefully handles empty views (no edges updated).

4. **API endpoint queries the view directly**: The `GET /topology/{site_id}/transit-stats` endpoint reads from `transit_time_stats`. If the view hasn't been refreshed recently, data may be stale. The `run_adaptive_update` pipeline should be run periodically (e.g., hourly cron).

5. **MTMC service reads transit_distributions at topology load**: The existing `topology_client.py` in the MTMC service derives distributions from `default_distributions()` — it does not yet read the `transit_distributions` JSONB column. A follow-up change to the topology API's `get_topology` endpoint (or the MTMC topology_client) is needed to prefer the JSONB column when populated.

## Open questions

1. **Refresh frequency**: How often should `run_adaptive_update` be called? Hourly seems reasonable for most sites, but high-traffic sites may benefit from more frequent refreshes.
2. **Per-direction vs bidirectional**: Current implementation supports directed transit times. Should the API expose both directions separately, or average them?
3. **Minimum transit time floor**: Should there be a floor on blended p50_ms to prevent unrealistically low values from noisy data?
