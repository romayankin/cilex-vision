# Handoff: P1-V02 — Ingress Bridge Service

## What was built

- `services/ingress-bridge/main.py` now implements the ingress bridge runtime:
  - binds to per-site JetStream durables for `frames.live.{site_id}.>` and `frames.replay.{site_id}.>`
  - validates `FrameRef` payloads against Schema Registry state before protobuf parse
  - stamps `core_ingest_ts` once at the bridge
  - offloads auxiliary blobs over 100 KB to MinIO
  - produces to Kafka with idempotent keys in the exact format `{site_id}:{camera_id}:{capture_ts_us}:{frame_seq}`
  - spools prepared or pending records to disk on Kafka / MinIO / prepare failures
  - drains the spool with per-site throttling and live-traffic headroom checks
  - publishes schema rejects to `frames.dlq.{site_id}` after the redelivery threshold
- `services/ingress-bridge/metrics.py` defines the Prometheus counters, gauges, and histograms required by ADR-001 and the task prompt.
- `services/ingress-bridge/config.py`, `requirements.txt`, `Dockerfile`, and `gen_proto.sh` follow the same flat-service pattern used by `services/edge-agent/`.
- `services/ingress-bridge/tests/test_ingress_bridge.py` covers:
  - exact key construction
  - blob offload before Kafka produce
  - Kafka failure -> spool -> recovery drain
  - schema failure nack behavior
  - DLQ publish after max redelivery
  - NATS consume path via an in-memory adapter

## Key decisions

- **Resolved the timestamp-spec conflict in favor of `docs/time-sync-policy.md` and the existing edge agent.**
  - ADR-001 says all three timestamps must already be present during inbound validation.
  - `P1-V01` and `docs/time-sync-policy.md` make `core_ingest_ts` the bridge's responsibility.
  - The service therefore validates `source_capture_ts` and `edge_receive_ts` on ingress, then stamps `core_ingest_ts` immediately before Kafka produce.
- **JetStream consumers are treated as pre-created Ops resources.**
  - This matches `docs/security-design.md`, which explicitly says application principals should not need broad JetStream admin rights.
  - The bridge binds to predictable durable names derived from `durable_prefix`, `site_id`, and lane.
- **Schema Registry integration is runtime-lazy.**
  - If `confluent-kafka` is installed, the validator will use the Schema Registry client / protobuf deserializer path first.
  - Because the current edge agent publishes raw protobuf bytes rather than Confluent wire-framed payloads, the validator falls back to generated protobuf parsing after confirming the registered subject is available.
- **NATS, Kafka, MinIO, and FastAPI imports stay lazy.**
  - This keeps unit tests runnable in thin environments where those runtime dependencies are not installed.

## Patterns established

- Flat service layout: `main.py`, `config.py`, `metrics.py`, `Dockerfile`, `requirements.txt`, `tests/`.
- `Settings.from_yaml()` with nested config models and env overrides.
- In-memory adapter fakes for external systems so service logic can be tested without live brokers.
- File-backed spool format uses a fixed binary header plus JSON metadata and payload bytes, with quarantine on decode failure.

## Gotchas

- `SchemaRegistryValidator` still has to tolerate the current edge-agent payload format. If the edge agent later moves to Confluent wire framing, the validator path can become strict and drop the fallback parse.
- JetStream lag is updated from `consumer_info()` when available. If the broker or client version omits those fields, the gauge will stay best-effort rather than exact.
- The service assumes per-site `sites:` entries exist in YAML. With an empty site list, the default adapter has nothing to bind to and readiness will stay false.
- Kafka delivery callbacks in `confluent-kafka` require polling; `KafkaProducerAdapter` now runs a background poll task per produce. Do not remove that loop unless the producer model changes.
