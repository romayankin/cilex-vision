# Handoff: P0-D05 — Camera Topology Graph Data Model

## What was built

- `services/topology/models.py` — Pydantic models for the camera topology graph:
  - `TransitTimeDistribution`: per-class distribution with p50_ms, p90_ms, p99_ms, sample_count, last_updated
  - `CameraNode`: wraps the `cameras` DB table, adds `zone_id` (extracted from `config_json` JSONB)
  - `TransitionEdge`: wraps `topology_edges` DB table, includes `transit_distributions` list (7 object classes), plus `default_distributions()` static method that derives seed values from `transition_time_s` using per-class speed factors
  - `TopologyGraph`: full site graph with `adjacent_cameras(camera_id)` and `downstream_cameras(camera_id, time_window_ms, object_class)` BFS helper
  - `EdgeCreateRequest`, `CameraCreateRequest`: request body models for write endpoints
- `services/topology/api.py` — FastAPI router with 4 endpoints:
  - `GET /topology/{site_id}` — full topology graph (admin + operator)
  - `PUT /topology/{site_id}/edges` — create/update edge with ON CONFLICT upsert (admin only)
  - `POST /topology/{site_id}/cameras` — add camera to site, zone_id stored in config_json (admin only)
  - `DELETE /topology/{site_id}/cameras/{camera_id}` — remove camera and its edges (admin only)
  - Inline JWT auth (mirrors query-api/auth/jwt.py pattern)
- `services/topology/seed.py` — Seed script for a 4-camera demo site:
  - cam-entrance (zone: entrance), cam-lobby (zone: lobby), cam-corridor (zone: corridor), cam-parking (zone: parking)
  - 6 directed edges with transition times from 5s to 90s (person baseline)
  - Per-class distributions derived via speed factors (car 0.3x, truck 0.5x, bus 0.4x, bicycle 0.6x, motorcycle 0.35x, animal 0.8x of person time)
  - `--apply --dsn` mode for database insertion, default prints JSON to stdout
- `docs/topology-schema.json` — JSON Schema (draft 2020-12) for TopologyGraph import/export, with definitions for CameraNode, TransitionEdge, TransitTimeDistribution
- `services/topology/tests/conftest.py` — FakePool (asyncpg mock), FakeSettings, JWT helper, httpx AsyncClient fixture
- `services/topology/tests/test_topology.py` — 37 tests:
  - 6 model construction tests (distributions, camera, edge)
  - 5 adjacency tests (direct neighbors, bidirectional, disabled edges, unknown camera)
  - 5 downstream BFS tests (within window, window too small, car faster, disabled excluded, source excluded)
  - 9 seed data tests (4 cameras, zone_ids, distributions present, transit times in 1s-120s range, JSON serializable, all edge cameras exist, adjacency, downstream)
  - 12 API CRUD tests (GET with graph, zone_id, distributions; PUT create edge, cameras not in site 404, operator 403; POST add camera, site not found; DELETE success, not found, operator 403; auth 401)

## Key decisions and why

- **zone_id in config_json, not a new column**: The `cameras` table already has a `config_json` JSONB column. Adding zone_id there avoids a database migration while giving CameraNode a first-class `zone_id` field. The `_extract_zone_id()` helper reads it from the JSON at query time.
- **Per-class speed factors for seed distributions**: Different object types traverse camera transitions at different speeds (cars are ~0.3x person walking time, bicycles ~0.6x). The `default_distributions()` static method derives all 7 class distributions from a single person-baseline `transition_time_s`, which is the value stored in the DB. This gives plausible initial values for MTMC matching before the adaptive learner has enough Re-ID samples.
- **BFS for downstream_cameras**: Uses breadth-first search with cumulative p99 cost so transitive paths are included (A→B→C is reachable if p99(A→B) + p99(B→C) ≤ window). This is what the MTMC matcher needs: "which cameras could this person appear on within N ms?"
- **Inline auth, not imported from query-api**: The topology service is a separate directory. Cross-service imports would create coupling. The inline `_require_admin` / `_require_read` functions follow the same JWT cookie pattern but are self-contained. When mounted in the query-api, the app's middleware handles the rest.
- **Upsert via ON CONFLICT**: The PUT /edges endpoint uses `INSERT ... ON CONFLICT (camera_a_id, camera_b_id) DO UPDATE`. This requires a unique constraint on (camera_a_id, camera_b_id) — see Gotchas below.
- **Transit distributions not persisted to DB**: The current `topology_edges` table only has `transition_time_s` (a single float). Per-class distributions are derived at query time via `default_distributions()`. As the adaptive learner accumulates Re-ID matches, a JSONB column or separate table can store learned distributions.

## Patterns established

- **Topology graph as a Pydantic model with helper methods**: `TopologyGraph.adjacent_cameras()` and `downstream_cameras()` encapsulate graph traversal logic. Any service that needs to query the topology (MTMC matcher, event engine) can deserialize the graph and call these methods.
- **Speed factor table**: `_CLASS_SPEED_FACTOR` maps object classes to relative transit speeds. This should be consistent across all services that reason about cross-camera timing.
- **Seed script dual mode**: `seed.py` either prints JSON (for review/testing) or applies to DB (for development setup). Other services can follow this pattern for their seed data.
- **FakePool with fetchrow**: Extended the query-api's FakePool pattern to support `fetchrow()` (single row return) and `set_execute_result()` (for DELETE result checking). Future topology tests can use these methods.

## Gotchas

- **ON CONFLICT requires unique constraint on (camera_a_id, camera_b_id)**: The `PUT /edges` endpoint uses `ON CONFLICT (camera_a_id, camera_b_id) DO UPDATE`. The current `topology_edges` table has a PK on `edge_id` but no unique constraint on the camera pair. A migration is needed: `CREATE UNIQUE INDEX uq_topology_edges_pair ON topology_edges (camera_a_id, camera_b_id)`. Without this, the upsert will always insert new rows.
- **Edges are directed**: `camera_a_id → camera_b_id` is a one-way edge. For bidirectional traversal (as in `adjacent_cameras()`), both directions need separate edge rows. The seed script creates explicit reverse edges where needed (e.g., corridor → lobby).
- **Transit distributions are derived, not stored**: The GET endpoint returns distributions computed from `transition_time_s` via `default_distributions()`. These are seed values with `sample_count=0`. The adaptive learning service should update the distributions in the topology data once it has enough Re-ID match data. A future migration adding a JSONB column (or a separate `edge_transit_distributions` table) will be needed.
- **site_id is UUID in the DB but passed as string in the API**: The `site_id` path parameter is a string. asyncpg handles UUID type coercion for the `sites.site_id` column. If the site_id isn't a valid UUID format, the DB query will fail with a type error.
- **No config.py for the topology service**: Unlike the query-api, the topology service doesn't have its own Pydantic Settings config file. The router reads `db_pool` and `settings` from `app.state`, which are provided by the host application (query-api or a standalone app). If the topology service needs its own configuration, a config.py should be added.
- **JSON Schema uses JSON `true`/`false` for defaults**: The `docs/topology-schema.json` uses JSON booleans which are flagged by ruff when scanning `.json` files. This is valid JSON Schema but ruff treats it as Python — can be excluded in ruff config.

## Open questions or known limitations

- **No learned distribution persistence**: The adaptive MTMC service will need a way to write updated distributions back. Options: (a) JSONB column on `topology_edges`, (b) separate `edge_transit_distributions` table with (edge_id, object_class) PK. Option (b) is cleaner and avoids wide rows.
- **No topology change notifications**: When an edge is created/updated/deleted, downstream consumers (MTMC matcher, event engine) are not notified. A Kafka event or LISTEN/NOTIFY mechanism would allow live topology updates.
- **No graph cycle detection**: The `downstream_cameras` BFS doesn't detect cycles (it uses visited tracking so it's safe, but it doesn't warn about cyclic topology configurations).
- **No import endpoint**: The JSON Schema defines the export format, but there's no `POST /topology/{site_id}/import` endpoint for bulk import from a JSON file. The seed script serves this role for development.
- **Camera deletion cascading**: `DELETE /cameras/{camera_id}` deletes edges first (FK constraint). If local_tracks or events reference the camera, the delete will fail. A production implementation should check for dependent records or use soft-delete.
