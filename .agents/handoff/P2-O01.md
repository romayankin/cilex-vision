# P2-O01 — MTMC Infrastructure

## What I built

- `infra/triton/models/osnet_reid/config.pbtxt` — dedicated MTMC Re-ID Triton model config using `tensorrt_plan`, input `[3,256,128]`, output `[512]`, preferred batches `[4,8,16]`, 50 ms queue delay, single GPU instance, and `latest 2` version policy
- `infra/ansible/playbooks/deploy-mtmc-infra.yml` — standalone MTMC infra playbook that:
  - stages a repo-root-compatible Docker build context for `services/mtmc-service`
  - builds and runs the MTMC service container with checkpoint storage and Prometheus-exposed metrics
  - waits for Kafka, TimescaleDB, and MinIO before startup
  - copies the MTMC dashboard and alert rule to the monitoring host
  - re-renders Prometheus with MTMC scrape targets and reloads it when already running
- `infra/grafana/dashboards/mtmc-health.json` — MTMC health dashboard with match rate, reject rate, match score heatmap, FAISS index size, checkpoint lag, checkpoint size, rebalance rate, and embedding throughput
- `infra/prometheus/rules/mtmc-alerts.yml` — MTMC alerts for zero match rate, checkpoint lag, FAISS index anomaly, high reject ratio, and stopped embedding consumption
- `infra/ansible/templates/monitoring/prometheus.yml.j2` — optional `mtmc-service` scrape job support driven by a playbook-provided target list

## Key decisions

- **Dedicated MTMC playbook instead of forcing `deploy-services.yml`**
  - The MTMC Dockerfile, like other newer service Dockerfiles, expects a repo-root build context with `proto/` plus `services/...`.
  - The generic `deploy-services.yml` still builds from a service directory only, so this playbook stages the minimum repo-root-compatible context explicitly.

- **Monitoring reload is part of the MTMC playbook**
  - Adding the dashboard and alert file to the repo is not enough to activate them on an already-running monitoring host.
  - The playbook copies both assets, re-renders Prometheus, and calls `/-/reload` when Prometheus is already up.

- **Prometheus scrape targets are computed from `services` inventory hosts**
  - The playbook assumes MTMC runs on service hosts and builds `host:port` scrape targets from inventory, with per-host overrides via `mtmc_enabled` and `mtmc_host_metrics_port`.
  - The monitoring template only emits the `mtmc-service` job when those targets are provided.

- **Kafka security gap is documented, not hidden**
  - The MTMC service config only supports `kafka_security_protocol`; it does not have SASL/SCRAM credentials or TLS file settings.
  - The infra playbook therefore defaults to `PLAINTEXT` for now and documents the mismatch against the secured Kafka deployment.

## Patterns followed

- Reused the existing Ansible structure from `deploy-triton.yml` / `deploy-monitoring.yml`: `community.docker`, `group_vars/all.yml`, `repo_root`, `common` role, and post-start smoke tests
- Kept monitoring assets in the existing repo locations: `infra/grafana/dashboards/` and `infra/prometheus/rules/`
- Kept Prometheus configuration source-of-truth in the shared template instead of hardcoding remote-only edits

## Gotchas

- `mtmc_host_metrics_port` defaults to `8080`; if a service host already publishes another container on host port `8080`, override this per host in inventory
- The MTMC dashboard’s match-score histogram is global because `mtmc_match_score` has no `site_id` label
- The playbook deploys MTMC to all hosts in the `services` inventory group unless `mtmc_enabled: false` is set on a host
- The playbook does not modify `deploy-all.yml`; operators need to run `deploy-mtmc-infra.yml` explicitly unless that orchestration is expanded later
