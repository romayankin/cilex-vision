# P2-A01: Cross-Camera Re-ID Annotation — Handoff

## What was built

Four deliverables extending the annotation workflow for cross-camera Re-ID ground truth:

1. **`docs/annotation-guidelines-reid.md`** — Annotation guide covering identity numbering (`{prefix}-{seq}`), quality criteria for valid pairs (visibility, temporal plausibility, class consistency, confidence >0.5), difficult cases (occlusion, look-alikes, groups), dual-annotator + adjudicator review workflow, and export format spec.

2. **`scripts/annotation/setup_reid_projects.py`** — Creates `reid-eval` CVAT project with 7 object-class labels, each carrying a `global_id` text attribute. Optionally creates annotation tasks for each camera pair when `--dsn` and `--site-id` are provided (queries topology_edges for adjacent cameras).

3. **`scripts/annotation/sample_reid_candidates.py`** — Queries local_tracks and topology_edges to find plausible exit→entry track pairs across adjacent cameras within a time window. Uses per-class speed factors and transit distribution p99 for window sizing. Ranks by temporal plausibility, outputs top N as JSON.

4. **`scripts/annotation/export_reid_pairs.py`** — Fetches annotations from the `reid-eval` CVAT project, groups sightings by `global_id` across tasks/cameras, computes Cohen's kappa on identity assignment across annotators. Outputs structured JSON with pairs and agreement metrics.

## Key decisions

| Decision | Rationale |
|----------|-----------|
| `global_id` as free-text attribute, not select | Annotators need arbitrary identity strings (`A-001`, `B-042`); a select dropdown would require pre-populating all possible IDs |
| Transit window = p99 from distributions or 3x baseline | Matches the MTMC matcher's window logic; p99 catches nearly all real transits without generating too many false candidates |
| Per-class speed factors in sampler | Same factors as topology docs (car 0.3x, truck 0.5x, etc.) so candidate windows are class-appropriate |
| Cohen's kappa (not Fleiss') for Re-ID agreement | Dual-annotator model means pairwise comparison; Fleiss' would only matter with 3+ annotators |
| Separate connections per query in sampler | Matches the asyncpg pattern used elsewhere; avoids connection pool complexity in a CLI script |

## Patterns followed

- All scripts use `urllib.request` with basic auth (no third-party HTTP libs), matching `setup_cvat_projects.py`
- `argparse` CLI with env var fallbacks (`CVAT_URL`, `CVAT_USERNAME`, `CVAT_PASSWORD`, `DATABASE_URL`)
- JSON contracts documented in module docstrings
- `if __name__ == "__main__"` with `try/except SystemExit` pattern
- `asyncpg` for DB queries in the sampler (matching repo convention)
- Shared HTTP helper functions (`build_headers`, `request_json`, `build_url`, `create_ssl_context`) identical to the existing setup script

## Gotchas

- The candidate sampler opens a new asyncpg connection per query. For very large sites with many edges, this could be slow — a connection pool would help but adds complexity for a CLI tool.
- `export_reid_pairs.py` tries per-job annotations when multiple jobs exist on a task (multi-annotator). If CVAT merges annotations at the task level, the agreement computation falls back gracefully.
- Camera ID extraction from task names relies on the `"reid: cam-a → cam-b"` naming convention. Tasks created manually with different names will need camera_id set another way.
- The `transit_distributions` JSONB column may be null for some edges; the sampler falls back to `transition_time_s * 3` in that case.

## Open questions

1. **Crop URI generation**: The export script leaves `crop_uri` as null. A follow-up could generate actual crop images from MinIO frames and attach them.
2. **Multi-annotator scaling**: Current agreement computation handles exactly 2 annotators. If 3+ annotators are used, only the first two are compared.
3. **Task data upload**: `setup_reid_projects.py` creates empty tasks. Actual frame/clip upload to CVAT tasks is left to the operator workflow or a future automation script.
