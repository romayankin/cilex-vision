# Handoff: P1-O01 — Pilot Infrastructure Deployment

## What was built

- `infra/ansible/ansible.cfg` — local Ansible config pointing at the example inventory and repo roles
- `infra/ansible/requirements.yml` — collection pin for `community.docker`
- `infra/ansible/group_vars/all.yml` — shared image tags, paths, ports, and pilot defaults
- `infra/ansible/inventory/pilot.yml` — pilot inventory showing the required per-host Kafka, NATS, CVAT, and service-deployment inputs
- `infra/ansible/playbooks/deploy-all.yml` — master orchestrator
- `infra/ansible/playbooks/deploy-kafka.yml` — 3-broker KRaft Kafka deployment with TLS/SCRAM config rendering and canonical topic bootstrap from `infra/kafka/topics.yaml`
- `infra/ansible/playbooks/create-kafka-topics.yml` — standalone canonical topic bootstrap against the secured Kafka cluster
- `infra/ansible/playbooks/deploy-nats.yml` — NATS JetStream deployment with rendered mTLS `verify_and_map` config
- `infra/ansible/playbooks/deploy-timescaledb.yml` — TimescaleDB deployment with the existing bootstrap SQL
- `infra/ansible/playbooks/deploy-minio.yml` — MinIO deployment plus bucket bootstrap matching the pilot object-store needs
- `infra/ansible/playbooks/deploy-triton.yml` — Triton deployment with explicit model-control flags and NVIDIA runtime validation
- `infra/ansible/playbooks/deploy-monitoring.yml` — Prometheus, Grafana, Loki, node-exporter, Promtail, and clock-drift collection
- `infra/ansible/playbooks/deploy-mlflow.yml` — MLflow deployment using TimescaleDB/PostgreSQL for the backend store and MinIO for artifact storage
- `infra/ansible/playbooks/deploy-cvat.yml` — internal-only CVAT deployment based on the current official Docker topology, bound to localhost by default
- `infra/ansible/playbooks/deploy-services.yml` — inventory-driven app-service deployment framework that only deploys services whose source directories actually contain Dockerfiles
- `infra/ansible/playbooks/smoke-test.yml` — standalone smoke test covering Kafka, NATS, TimescaleDB, MinIO, Triton, monitoring, MLflow, CVAT, and configured application services
- `infra/ansible/templates/**` and `infra/ansible/roles/common/**` — shared rendered configs and Docker host bootstrap tasks

## Key decisions

- Used Docker-based deployments everywhere so production playbooks stay close to `infra/docker-compose.yml` image tags and service behavior.
- Rendered Kafka and NATS configs from new Ansible templates derived from the repo’s existing security templates instead of trying to express the security model as ad hoc container env vars.
- Kept Kafka at `SASL_SSL` + `SCRAM-SHA-256` and NATS at mTLS with CN-based `verify_and_map`, matching `docs/security-design.md`.
- Used host networking for the infra containers to keep listener and scrape target wiring simple for a pilot rollout.
- Made monitoring deploy node-exporter and Promtail on all infra hosts so Prometheus and Loki have immediate signal once the stack is up.
- Reused `services/monitoring/clock_drift_check.py` directly and wired it into a cron-driven textfile collector on the monitoring host.
- Split Kafka topic creation into its own playbook while keeping the bootstrap embedded in `deploy-kafka.yml`, because the task now explicitly requires both entry points.
- Kept `deploy-services.yml` generic and inventory-driven because the repo still has zero `services/*/Dockerfile` files today; hardcoding future services here would have been fake infrastructure.
- Pinned CVAT to the officially released `v2.62.0` container images and supporting service versions current as of April 2, 2026.

## Patterns established

- Shared infra defaults live in `infra/ansible/group_vars/all.yml`; service playbooks only add the few per-service vars they need.
- Every service playbook includes smoke tests after container startup (`wait_for`, `uri`, or targeted `docker exec` checks).
- Repo source-of-truth files are copied from `infra/` or `services/monitoring/` at deploy time rather than duplicated into the playbooks.
- Pilot inventory lives at `infra/ansible/inventory/pilot.yml`; `ansible.cfg` points there by default.
- Secrets that must come from outside the repo are explicit inventory inputs:
  - Kafka keystore/truststore sources and passwords
  - NATS PKI bundle paths from `infra/pki/bootstrap-site.sh`
  - MinIO, Grafana, and Kafka admin credentials
  - CVAT database and admin credentials
  - per-host `service_deployments` entries for application services

## Gotchas for the next agent

- This environment did not have `ansible-playbook`, `ansible-lint`, or `yamllint`, so I could only validate YAML parsing and Jinja template rendering locally. Run full syntax/lint checks once the Ansible toolchain is available.
- `deploy-triton.yml` expects the NVIDIA runtime to already be configured in Docker. It validates that `docker info` exposes `nvidia`, but it does not install `nvidia-container-toolkit`.
- The Triton model repository in the repo currently contains `config.pbtxt` files only, not TensorRT `model.plan` artifacts. The playbook deploys the repo as-is; actual model loading still depends on those engine files being supplied later.
- Kafka ACL automation is still not materialized here. The playbook bootstraps the secured brokers and canonical topics, but service principals and topic ACL creation remain future work.
- MinIO lifecycle policy JSON is rendered under the deploy directory, but the playbook currently only guarantees bucket creation. If retention enforcement becomes required in this task scope, add an idempotent `mc ilm` reconciliation step rather than one-shot imperative commands.
- `deploy-services.yml` is intentionally a no-op with the current repo contents unless the operator populates `service_deployments` in `inventory/pilot.yml` and the referenced `services/<name>/` directories already contain Dockerfiles.
- CVAT is exposed through Traefik on `127.0.0.1:8080` by default. For LAN access, set `cvat_bind_address` and `cvat_host` explicitly in inventory so the Traefik host rules match the actual internal hostname.
