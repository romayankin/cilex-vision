  - `ensure_bucket()` — creates the `debug-traces` MinIO bucket with 30-day lifecycle policy (auto-expire)

- Added `date_str` property: derives date from `source_capture_ts` → `edge_receive_ts` → `datetime.now()` fallback chain

## Key decisions and why

- **TraceCollector alongside DebugTracer, not replacing it**: The existing `DebugTracer` is already used in `main.py`'s pipeline with its own key format (`traces/{camera_id}/{trace_id}.json`). Adding `TraceCollector` as a new class avoids breaking the existing integration while providing the enriched data collection. When `main.py` is ready to adopt `TraceCollector`, it can be swapped in.
- **Date-partitioned storage keys for TraceCollector only**: The new `{camera_id}/{date}/{trace_id}.json` format enables efficient prefix-based listing in MinIO (list all traces for a camera on a given date). The existing `DebugTracer` keeps its `traces/` prefix key format to avoid breaking existing stored traces.
- **MinIO-only query (no DB index)**: Debug trace listing queries MinIO directly via `list_objects()` rather than maintaining a separate DB index table. This is appropriate for a low-volume debug tool used only by engineering, avoids a DB migration, and keeps the trace lifecycle entirely in MinIO (30-day auto-expire cleans up everything).
- **track_id filtering via JSON download**: When `track_id` is passed to the listing endpoint, each candidate trace's JSON is downloaded and checked for the track_id in `track_ids`. This is acceptable because: debug trace volume is low (2% sample rate), the filter is used infrequently, and maintaining a secondary index isn't worth the complexity.
- **camera_id required on both endpoints**: MinIO has no global object search — you need a prefix. Requiring `camera_id` keeps queries efficient and scoped.
- **30-day lifecycle via MinIO bucket policy**: `ensure_bucket()` configures `Expiration(days=30)` on the bucket, matching the task spec and the detection hypertable retention policy. Gracefully degrades if the `minio` package lifecycle modules aren't available.

## Patterns established

- **TraceCollector collection pattern**: call `begin()` first with frame identity and Kafka metadata, then call individual `collect_*()` methods as each pipeline stage completes. This mirrors the existing `TraceStage` timing pattern but captures data rather than just timing.
- **MinIO listing for debug queries**: `list_objects(bucket, prefix, recursive=True)` + parse object key to extract metadata (camera_id, date, trace_id from the key segments). This pattern is reusable for any MinIO-backed listing endpoint.
- **Role-restricted router**: `dependencies=[require_role("engineering", "admin")]` on the router endpoints. Engineering-only resources follow this pattern — admin always gets access.

## Gotchas

- **minio lifecycle imports may not be available**: The `minio.commonconfig.ENABLED` and `minio.lifecycleconfig` imports require `minio>=7.0`. If an older version is installed, `ensure_bucket()` will create the bucket but skip the lifecycle policy (logged as warning). The test mocks `sys.modules` to handle this.
- **DebugTracer.store() key format is unchanged**: The existing `DebugTracer` still writes to `traces/{camera_id}/{trace_id}.json`. The new `TraceCollector` uses `{camera_id}/{date}/{trace_id}.json`. If `main.py` still uses `DebugTracer`, the query-api debug router won't find those traces (different key format). When migrating `main.py` to `TraceCollector`, the old traces will age out via retention.
- **MinIO list_objects returns an iterator**: The `list_objects()` call is lazy (returns an iterator). The router wraps it in `asyncio.to_thread()` but iterates synchronously within the thread. For very large result sets, the `limit` parameter caps the output.
- **to_json() now includes empty enrichment fields**: Existing traces created by `DebugTracer` will serialize with empty `raw_detections_pre_nms: []`, `tracker_state_delta: {}`, etc. This is harmless — the fields default to empty.
- **asyncio.to_thread for all MinIO calls in debug router**: Since `minio-py` is synchronous, every call (`list_objects`, `get_object`) is wrapped in `asyncio.to_thread()` to avoid blocking the event loop.

## Open questions or known limitations

- **main.py not yet updated to use TraceCollector**: The inference worker's `main.py` still uses the original `DebugTracer`. Switching to `TraceCollector` requires passing `msg.offset` (Kafka offset), pre-NMS raw boxes (from `DetectorClient`), and tracker state delta (track count before/after `tracker.update()`). This is a straightforward integration but was out of scope for the expected deliverables.
- **No pagination cursor for MinIO listing**: The listing endpoint uses `limit` but no offset/cursor. For cameras with thousands of traces, a cursor-based approach using the last-seen object name would be more efficient.
- **track_id post-filter downloads JSON**: This is O(n) in the number of traces returned. For high-volume debugging, a secondary key or MinIO object tagging with track_ids would be more efficient.
- **No RBAC permission entry for "debug"**: The `ROLE_PERMISSIONS` dict in `auth/jwt.py` doesn't include a `"debug"` entry — the debug router uses `require_role()` directly. This is fine since the permission matrix is checked per-endpoint, but if a unified permission check is added later, a `"debug"` permission should be added to the engineering and admin roles.
