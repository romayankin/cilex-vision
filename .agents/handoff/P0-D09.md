# Handoff: P0-D09 — Model Rollout & Cutover SOP

## What was built

- `docs/runbooks/model-rollout-sop.md` — Four-stage Standard Operating Procedure for updating Triton-served models (yolov8l, osnet, color_classifier) and the CPU-side tracker (ByteTrack/BoT-SORT):
  - **Stage 1: Offline Qualification** — trtexec TensorRT engine build commands for all three model types, bake-off evaluation via MLflow, 2% regression gate on primary metrics (mAP@0.5:0.95, IDF1), 5% per-class regression cap, throughput check
  - **Stage 2: Shadow Deployment** — Triton HTTP API `load` command, `version_policy { latest { num_versions: 2 } }` for dual loading, VRAM verification, 24-48h minimum shadow period with Prometheus monitoring queries, zero-impact rollback (unload shadow version)
  - **Stage 3: Site-Level Cutover** — Standard procedure (kubectl rollout restart) for detector/color classifier, plus Re-ID special case (FAISS flush + tracker reset per ADR-008, ~30s matching blackout), embedding version boundary enforcement
  - **Stage 4: Post-Cutover Watch** — 4h elevated monitoring, 7 Prometheus queries referencing `infra/prometheus/rules/triton-alerts.yml`, 15% degradation rollback trigger table, old version unload after watch passes
  - Each stage includes: gate criteria table, concrete commands, rollback procedure, copy-pasteable checklist template
  - Special case section for tracker software updates (CPU-only, application-level shadow via feature flag)
  - Security requirements section referencing `docs/security-design.md` (SASL_SSL, mTLS, admin JWT for FAISS flush)
  - Mermaid flowchart of the four-stage process
  - Runbook log template for recording each rollout execution
  - Acceptance criteria section (automated + human review)

## Key decisions and why

- **Four stages, not three**: Separated shadow deployment (Stage 2) from cutover (Stage 3) because shadow has zero production impact and a different rollback profile (just unload). This gives operators a clear "go/no-go" decision point before touching production traffic.
- **2% regression for offline, 15% for rollback**: Offline qualification uses a tight 2% threshold because there's no time pressure — reject early and cheaply. The 15% post-cutover rollback threshold is higher because production metrics are noisier than controlled evaluations and the cost of unnecessary rollbacks (another blackout) is significant.
- **Re-ID cutover as a distinct procedure**: Per ADR-008, embeddings from different OSNet versions MUST NOT be compared. The osnet cutover procedure (3.R1-3.R6) includes FAISS flush, tracker reset, and embedding version verification as mandatory steps. This is called out separately from the standard cutover to prevent operators from skipping it.
- **Concrete commands, not abstractions**: Every step has copy-pasteable `curl`, `kubectl`, `trtexec`, and `promql` commands. Operators should not have to figure out Triton API endpoints or Prometheus label selectors during a cutover window.
- **Checklist templates per stage**: Designed to be copied into a runbook log entry for each rollout. This creates an audit trail and prevents steps from being skipped.

## Patterns established

- **Runbook format**: YAML front-matter, Mermaid overview diagram, per-stage sections with Prerequisites → Procedure → Rollback → Gate Criteria → Checklist structure. Future runbooks (e.g., database migration, Kafka topic changes) can follow this template.
- **Shadow deployment via version_policy**: Using `latest { num_versions: 2 }` from the existing Triton config means shadow deployment requires only a `load` call — no config changes. The worker-side `INFERENCE_SHADOW_MODEL_VERSION` env var controls which requests go to the shadow.
- **Rollout log as structured markdown table**: The log template at the end of the SOP creates a consistent record format for post-incident review and compliance.

## Gotchas

- **FAISS flush endpoint assumed**: The SOP references `http://mtmc-service:8080/admin/faiss/flush` and `status` endpoints. These need to be implemented in the MTMC service. The endpoints are documented here so the MTMC implementer knows the expected contract.
- **Tracker reset via SIGUSR1 assumed**: Stage 3.R3 sends `SIGUSR1` to trigger tracker reinitialization. The inference worker needs a signal handler that reinitializes all per-camera ByteTrack instances. This is referenced in the P1-V04 handoff but not yet implemented.
- **No automation**: This SOP is manual. A CI/CD pipeline or Argo Rollouts integration could automate stages 2-4 with the gate criteria as automated checks. The SOP provides the basis for such automation but doesn't implement it.
- **Model repository path assumed as `/models/`**: The SOP uses `/models/{MODEL_NAME}/{VERSION}/` paths. In a Kubernetes deployment with PVC-backed model storage, the actual mount path may differ. Operators should substitute the correct path.
- **Tracker software update via feature flag**: The special case section notes that tracker shadow deployment uses an application-level feature flag (not Triton's version_policy). This feature flag mechanism is not yet implemented in the inference worker.

## Open questions or known limitations

- **No automated gate checks**: Gate criteria are manual Prometheus queries. An operator scripts or CI integration (e.g., a `scripts/rollout/check-gates.sh`) would reduce human error during cutover.
- **No multi-site coordination**: The SOP covers single-site cutover. For multi-site deployments, the sequencing (canary site → remaining sites) is not specified.
- **No model provenance verification**: The SOP doesn't verify that the TensorRT engine was built from the expected ONNX source (e.g., checksum matching). A model registry with signed artifacts would close this gap.
- **Shadow comparison dashboard not defined**: Stage 2 references a Grafana dashboard for comparing shadow vs production output distributions. This dashboard needs to be created.
- **Consumer lag rollback threshold (10,000 messages)**: This number is a starting point. The appropriate threshold depends on the number of cameras and expected event rate per site. Should be calibrated after initial deployments.
