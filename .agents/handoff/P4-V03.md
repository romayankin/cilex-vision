# P4-V03: LPR Module — Handoff

## What was built

- `services/lpr-service/main.py`
  - Kafka consumer for `tracklets.local`
  - vehicle-class filtering for `car`, `truck`, and `bus`
  - representative-frame lookup from MinIO
  - per-track best-result buffering
  - asyncpg COPY writes into `lpr_results`
  - graceful shutdown with pending-result flush
- `services/lpr-service/plate_detector.py`
  - Triton gRPC client for stage-1 plate detection
  - RGB resize/normalize preprocessing
  - confidence filtering and greedy NMS
- `services/lpr-service/ocr_client.py`
  - Triton gRPC client for stage-2 OCR
  - greedy CTC-style decoding
  - simple country-format heuristics
- `services/lpr-service/quality_gate.py`
  - plate-specific quality checks for minimum size, Laplacian sharpness, and aspect ratio
- `services/lpr-service/config.py`
  - Pydantic Settings v2 with `LPR_` prefix and `from_yaml()`
  - feature flag via `enabled`
- `services/lpr-service/metrics.py`
  - `plates_detected_total`
  - `plates_recognized_total`
  - `quality_rejected_total`
  - `detection_latency_ms`
  - `ocr_latency_ms`
  - `pipeline_errors_total`
- `services/lpr-service/{requirements.txt,gen_proto.sh,mypy.ini,Dockerfile,README.md}`
- `services/lpr-service/tests/test_plate_detector.py`
- `services/db/migrations/add_lpr_results.sql`
- `services/db/migrations/versions/xxx_add_lpr_results.py`

## Key decisions

- Per-track aggregation is kept inside `main.py` instead of adding another module.
  - Reason: this service only needs “best observed plate per track”, not a reusable voting subsystem like attribute-service.
- LPR writes use asyncpg `copy_records_to_table`.
  - Reason: this stays aligned with the repo’s COPY-first write pattern instead of doing row-by-row inserts.
- Frame lookup uses decoded-frame MinIO listing by `camera_id/date/` and nearest `last_modified`, then falls back to configurable key templates.
  - Reason: `Tracklet` and the current DB schema do not carry a canonical `frame_uri`, but the task still requires frame retrieval from MinIO.
- Query API LPR search is restricted to `admin` and `operator`.
  - Reason: plate search is more sensitive than the existing general read endpoints, and the prompt did not require viewer access.
- `enabled` defaults to `false`.
  - Reason: LPR is explicitly optional and should not break deployments where the Triton models or storage assumptions are not ready yet.

## Patterns followed

- Flat service layout, mirroring `attribute-service`
- Pydantic Settings v2 with YAML + env overrides
- Triton gRPC clients with lazy `tritonclient.grpc` imports and `asyncio.to_thread()`
- Prometheus metrics in a dedicated `metrics.py`
- Query API dynamic raw-SQL filtering pattern reused for `/lpr/results`
- Raw SQL migration file plus thin Python upgrade wrapper, matching `services/db/migrations/versions/xxx_add_transit_time_aggregate.py`

## Gotchas

1. The upstream pipeline still does not persist a canonical frame URI per detection or track.
   - The MinIO lookup is heuristic, not exact.
   - This is the biggest practical limitation in the current implementation.

2. `plate_detector` and `plate_ocr` Triton models are assumed by config only.
   - No model repo/config artifacts were part of this task.
   - Enabling the service without those models loaded will fail at runtime.

3. Query API docs were not regenerated.
   - The route exists and tests pass, but `docs/api/openapi.yaml` and the Postman collection do not yet include `/lpr/results`.

4. OCR decoding assumes a CTC-like output of `blank + alphabet`.
   - If the actual Triton OCR model exports a different layout, only `ocr_client.py` should need adjustment.

5. The migration creates `lpr_results`, but `services/db/models.py` was intentionally left untouched.
   - The task asked for a raw SQL migration, and there is no consistent ORM model pattern yet for this new table.
