# P3-V01: Model Retraining Pipeline — Handoff

## What was built

Eight deliverables forming a reproducible model retraining pipeline at `scripts/training/`:

1. **`conf/config.yaml`** — Hydra config with sections: data (manifests, batch/workers), model (yolov8l, 7 classes, 640px), training (100 epochs, cosine LR, early stopping), mlflow (tracking URI, experiment name), evaluation (regression threshold 0.02, operational threshold 0.40, 7 classes), export (ONNX opset 17, TensorRT FP16).

2. **`pull_data.py`** — Exports annotations from CVAT via REST API. Supports COCO 1.0, Datumaro 1.0, and CVAT for images 1.1 formats. Handles async dataset export (polling for 202 status), extracts zip archives per task, writes export manifest JSON with metadata.

3. **`version_data.py`** — Versions a dataset directory with DVC. Runs `dvc add` via subprocess, creates versioned symlinks, writes version metadata JSON, optionally pushes to a DVC remote.

4. **`train.py`** — Hydra-configured training script with `@hydra.main` decorator. MLflow integration logs all config params, per-epoch metrics (loss, mAP, LR), and best checkpoint as artifact. Early stopping on val mAP plateau. Writes `latest_run_id.txt` for downstream scripts. Skeleton training loop documents the exact interface for real PyTorch code.

5. **`evaluate.py`** — Computes all bakeoff metrics: mAP@0.5, mAP@0.5:0.95, per-class AP (7 classes), small_object_AP, night_AP, operational precision/recall/F1 at threshold 0.40. Fetches baseline from MLflow by run_id, compares all slices, exits non-zero if any regression > 2%. Generates Markdown comparison report. Logs results and eval_gate tag to MLflow.

6. **`export.py`** — PyTorch → ONNX (configurable opset) → TensorRT (via trtexec subprocess). Validates ONNX with onnx.checker and onnxruntime. Falls back gracefully when trtexec/onnxruntime not available, outputting manual commands.

7. **`promote.py`** — Registers model in MLflow Model Registry. Transitions stages (None → Staging → Production). Verifies eval_gate tag before Production promotion. Builds description from run params/metrics. `--force` flag bypasses gate check.

8. **`Makefile`** — Targets: pull-data, version-data, train, evaluate, export, promote. `make retrain` chains pull-data → version-data → train → evaluate → export. Promote is intentionally separate (requires manual review).

## Key decisions

| Decision | Rationale |
|----------|-----------|
| Skeleton training loop (not real PyTorch) | Task says "scripts must work in a thin environment (no GPU required for structure/logic)"; real forward pass slots into clearly documented interfaces |
| eval_gate tag for promotion gating | evaluate.py sets `eval_gate=passed/failed` and `regression_detected` tags on MLflow run; promote.py checks these before allowing Production stage |
| Non-zero exit code on regression | evaluate.py exits 1 if any slice regresses > threshold; this halts the `make retrain` chain at the evaluate step |
| Promote separate from retrain chain | `make retrain` stops after export; `make promote` is a deliberate human action after reviewing the eval report |
| DVC via subprocess (not Python API) | DVC's Python API is unstable across versions; subprocess `dvc add`/`dvc push` is the documented stable interface |
| CVAT dataset export with polling | CVAT dataset export returns 202 while preparing; pull_data.py polls until 200 with configurable timeout |
| Graceful degradation for missing tools | export.py handles missing trtexec/onnxruntime/onnx without crashing, prints manual commands instead |

## Patterns followed

- CLI-first with `argparse`, `if __name__ == "__main__"` with `try/except SystemExit`
- CVAT API via `urllib.request` with basic auth (same helpers as setup_cvat_projects.py)
- Env var fallbacks for credentials (CVAT_URL, CVAT_USERNAME, CVAT_PASSWORD, MLFLOW_TRACKING_URI)
- JSON contracts documented in module docstrings
- All 7 object classes from taxonomy: person, car, truck, bus, bicycle, motorcycle, animal
- Regression threshold 0.02 and operational threshold 0.40 from bake-off protocol

## Gotchas

- `train.py` requires `hydra-core`, `omegaconf`, and `mlflow` packages. These are not in a requirements.txt — they should be added to the training environment setup.
- Hydra changes the working directory by default. `train.py` resolves the original CWD via `hydra.utils.get_original_cwd()` for checkpoint paths.
- `evaluate.py` imports `mlflow` lazily (only when baseline comparison or logging is needed). It will fail if mlflow is not installed and `--baseline-run-id` is provided.
- `promote.py` uses the MLflow Model Registry's `transition_model_version_stage` API which may be deprecated in newer MLflow versions in favor of the Aliases API.
- The `pull_data.py` CVAT dataset export polling has a 300s max wait. Very large projects may need a longer timeout.
- DVC must be initialized (`dvc init`) in the repo before `version_data.py` can run `dvc add`.

2. **Training environment**: A `requirements.txt` or conda environment file for the training scripts is not yet created.
3. **Latency/throughput metrics**: evaluate.py has fields for latency_p50/p95/p99 and throughput_fps, but these require a running Triton instance. How should these be populated during offline evaluation?
