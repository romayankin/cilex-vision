# Handoff: P1-E03 — Edge Filter Pass-Through Rate Measurement

## What was built

- `scripts/calibration/edge_filter_calibration.py`
  - Real calibration harness for a single camera.
  - Live mode publishes the control command on `site.{site_id}.control.{camera_id}.calibrate`, subscribes to `frames.live.{site_id}.{camera_id}`, captures `FrameRef` traffic for a fixed window, downloads the referenced frames, and writes a capture manifest.
  - Analysis mode (`--analysis-only`) reuses an existing capture manifest and reruns the central detector + motion calibration without touching NATS.
  - Runs YOLOv8-L offline through the existing `services/inference-worker/detector_client.py` path, classifies each frame as object-positive / object-negative, and replays a grid of `MotionConfig` candidates against the same timestamped frame sequence.
  - Emits:
    - JSON scorecard
    - Markdown scorecard
    - updated `scripts/cost-model/params.yaml`
    - Prometheus textfile with `per_camera_pass_through_rate` and `calibration_freshness_hours`

- `scripts/calibration/schedule_calibration.py`
  - Cron-friendly scheduler wrapper.
  - Reads the edge-agent YAML config, picks the stalest enabled camera, and enforces the required one-camera-per-hour rate limit.
  - Persists scheduler state in `artifacts/calibration/schedule-state.json`.
  - Shells out to `edge_filter_calibration.py` with a deterministic UTC run id and refreshes the Prometheus textfile after every invocation.

- `scripts/cost-model/params.yaml`
  - New canonical calibration output file.
  - Stores defaults plus per-camera measured pass-through / miss / false-trigger rates and the recommended motion parameters.

- Focused tests:
  - `scripts/calibration/tests/test_edge_filter_calibration.py`
  - `scripts/calibration/tests/test_schedule_calibration.py`

## Key decisions

- **Single-pass calibration evaluation**
  - The harness does not load the entire 10-minute window into memory.
  - It streams frames one by one, runs the detector once per frame, and replays all motion candidates in parallel on the same frame. This keeps memory bounded and avoids rereading the same JPEGs for every candidate.

- **Timestamp-faithful motion replay**
  - The edge `MotionDetector` uses `time.monotonic()` internally for periodic reference refresh.
  - During offline replay, the harness monkeypatches the motion module’s monotonic clock per frame using `edge_receive_ts` deltas so `reference_update_interval_s` behaves as it would during live capture.

- **Calibration ranking**
  - Candidates are first filtered to those within `±0.05` of the target `0.15` pass-through rate when possible.
  - Within that preferred window, ranking uses a weighted score:
    - `0.60 * (1 - miss_rate)`
    - `0.25 * (1 - false_trigger_rate)`
    - `0.15 * closeness_to_target_pass_through`
  - If no candidate lands in the preferred window, the full grid is ranked.
  - Exact ties keep the baseline for the safer default.

- **Config pattern**
  - The scripts use local Pydantic `BaseSettings` models with `from_yaml()` methods, matching the service pattern established elsewhere in the repo.
  - `schedule_calibration.py` intentionally imports helpers from `edge_filter_calibration.py` instead of duplicating the params / metrics logic.

- **Prometheus textfile format**
  - The new metrics are not added to a running service process; they are emitted as a node-exporter textfile under `artifacts/calibration/prometheus/`.
  - Uncalibrated cameras are exported as `NaN` so dashboards can distinguish “no calibration yet” from `0.0`.

## Gotchas for the next agent

- **Edge control-plane gap still exists**
  - The scheduler and calibration harness publish to `site.{site_id}.control.{camera_id}.calibrate` exactly as requested.
  - The current `services/edge-agent/` implementation does **not** yet subscribe to or act on that control subject. In a real deployment, the edge agent must gain a compatible control consumer before live calibration will truly disable the motion filter.

- **NATS ACL template still lacks control subjects**
  - `infra/nats/nats-server.conf` currently documents data subjects but not the new `site.<site_id>.control.<camera_id>.calibrate` publish path.
  - Ops will need to extend the NATS authorization template before this can work in a locked-down environment.

- **Detector / Triton / MinIO dependencies are runtime-only**
  - The scripts import those dependencies lazily and help output works without them.
  - A real calibration run still requires:
    - `nats-py`
    - `minio`
    - `tritonclient[grpc]`
    - a reachable Triton server with the YOLOv8-L detector loaded
    - object-store access to the captured `frame_uri` objects

- **Proto generation**
  - The harness will use existing generated `proto_gen` directories if present.
  - If they are absent, it attempts an on-the-fly `grpc_tools.protoc` generation.
  - If `grpcio-tools` is also missing, it fails fast with a clear hint.

- **`params.yaml` is now the source of truth for calibration freshness**
  - The scheduler metrics exporter computes freshness from the `measured_at` timestamp stored in `scripts/cost-model/params.yaml`.
  - If another task changes that schema, update both `build_metrics_text()` and the scheduler’s inventory refresh path.
