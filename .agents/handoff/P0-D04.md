# Handoff: P0-D04 — Database Schema

## What was built
- services/db/models.py — 12 SQLAlchemy 2.0 async models
- Alembic migration 0001 — creates all tables, hypertables, retention policies
- docs/diagrams/schema.mermaid — ER diagram
- docs/adr/ADR-003-database-schema.md — design rationale

## Key decisions
- Enums as CHECK-constrained TEXT, NOT native PostgreSQL ENUM types.
  Reason: native ENUMs require migrations to add values, CHECK constraints
  are simpler to evolve and match the Python str enum pattern.
- UUIDs via gen_random_uuid() server default, not Python-side generation.
- TimescaleDB hypertables for detections and track_observations only.
  All other tables are standard PostgreSQL.
- Composite primary key (time, camera_id) on hypertables — required by
  TimescaleDB for partitioning.

## Patterns established
- TimestampMixin for created_at/updated_at on relational tables
- TSTZ = TIMESTAMP(timezone=True) alias used everywhere
- __all__ export list at bottom of models.py
- Check constraint names follow: ck_{table}_{column}

## Gotchas for next agent
- Don't add native PostgreSQL ENUM types — use CHECK constraints instead
- The migration uses raw SQL for TimescaleDB functions (create_hypertable,
  add_retention_policy, add_compression_policy) because Alembic doesn't
  have native TimescaleDB support
- camera_id is TEXT not UUID — deliberate, allows human-readable IDs like "cam-lobby-01"
