# P2-O03 — Calibration Scheduler

## What I built

- `scripts/calibration/calibration_scheduler.py`
  - DB-driven scheduler that queries online cameras from PostgreSQL instead of
    deriving rotation order from the edge-agent camera list.
  - Reuses the existing `edge_filter_calibration.py` worker via
    `subprocess.run()`.
  - Creates and writes to a new `calibration_results` table on successful runs.
  - Preserves the existing scheduler state file at
    `artifacts/calibration/schedule-state.json` and refreshes the shared
    Prometheus textfile after each invocation.
  - Enforces a single active scheduler via a file lock and still enforces the
    one-calibration-at-a-time / minimum-gap guard.

- `scripts/calibration/calibration_report.py`
  - Reads cameras plus calibration history from PostgreSQL.
  - Writes a Markdown report with:
    - per-camera latest status
    - last `N` calibration history rows per camera
    - stale online camera list
    - summary counts and average latest pass-through rate

- `infra/cron/calibration-cron.yml`
  - Systemd timer / service definition plus cron fallback for daily automated
    scheduler execution.

## Key decisions

- **DB selection, existing worker execution**
  - The new scheduler only changed *how the next camera is chosen* and where
    successful results are persisted.
  - It still shells out to `edge_filter_calibration.py` so the calibration
    analysis path, params YAML updates, and scorecard format stay consistent
    with `P1-E03`.

- **Existing artifact formats stay canonical**
  - `scripts/cost-model/params.yaml`, the Prometheus textfile under
    `artifacts/calibration/prometheus/`, and the JSON scheduler state file
    remain the operator-facing artifacts.
  - The DB table is additive, not a replacement for those files.

- **Calibration freshness uses multiple sources during transition**
  - Camera ordering prefers the latest success in `calibration_results`, but
    falls back to `params.yaml` and the old scheduler state so cameras that
    were calibrated before this task do not immediately look “uncalibrated”.

- **Operational site ID stays tied to edge config**
  - The DB schema models `cameras.site_id` as a UUID, while the calibration
    worker and existing params/state keys use the edge-config string site ID
    (for example `pilot-site`).
  - The scheduler therefore keeps using `edge_config.site_id` for artifact keys
    and worker invocation, while still using the DB as the source of online
    camera inventory.

## Patterns followed

- `argparse` CLIs with env-var fallback for `--db-dsn`
- lazy `asyncpg` import with clear install hint
- `subprocess.run()` for invoking the existing worker
- asyncpg for all DB access
- additive filesystem state compatible with the previous scheduler

## Gotchas

- **DB-driven does not mean DB-only runtime config**
  - The scheduler picks cameras from PostgreSQL, but the worker still requires
    `--edge-config` because the DB does not currently contain the NATS / MinIO /
    motion defaults needed for a calibration run.

- **The scheduler skips DB cameras missing from the enabled edge config**
  - Online cameras that are present in PostgreSQL but not enabled in the edge
    YAML are included in the Prometheus inventory, but they are not selected for
    execution until the edge config is updated.

- **`calibration_results` is created at runtime**
  - There is no Alembic migration for this table yet. The runtime DB user must
    have permission to execute `CREATE TABLE IF NOT EXISTS` and
    `CREATE INDEX IF NOT EXISTS`.

- **The older `schedule_calibration.py` still exists**
  - Keep using it for edge-config-only environments.
  - Use `calibration_scheduler.py` when PostgreSQL camera inventory is
    available and should drive the rotation.
