# Handoff: P1-A01 — CVAT Setup & Annotation Baseline

## What was built

- `scripts/annotation/setup_cvat_projects.py` — CVAT REST API bootstrap for the three pilot projects:
  - `detection-eval`
  - `tracking-eval`
  - `attribute-eval`
- `docs/annotation-guidelines.md` — full annotation guide replacing the stub
- `docs/diagrams/annotation-tight-box.svg` — tight-vs-loose box visual example
- `docs/diagrams/annotation-track-continuity.svg` — track continuity visual example
- `docs/diagrams/annotation-color-attributes.svg` — color labeling visual example
- `scripts/annotation/compute_iaa.py` — JSON-to-JSON/Markdown IAA tool
- `scripts/annotation/split_dataset.py` — chronological train/val/test splitting tool

## Key decisions

- The CVAT project labels match the canonical enums exactly from `docs/taxonomy.md` and `services/db/models.py`; I did not introduce alternate spellings or extra classes.
- `setup_cvat_projects.py` uses the REST API directly instead of depending on `cvat-sdk`, because this repo does not currently pin a Python environment for annotation scripts and the direct API keeps the script runnable in a thin environment.
- The setup script is fail-safe on existing projects: if a project already exists with drifted labels, it exits unless `--recreate-existing` is explicitly provided. Silent in-place mutation would be too risky for annotation state.
- `tracking-eval` includes MOT-oriented `visibility` and `ignored` attributes so the project schema is aligned with the intended `MOT 1.1` export workflow.
- `compute_iaa.py` uses a normalized JSON bundle format instead of trying to parse every CVAT export flavor. That keeps the agreement logic deterministic and testable.
- `split_dataset.py` splits whole temporal sequences, never individual frames, and preserves chronology globally so evaluation leakage across adjacent time windows is avoided.

## Patterns established

- Annotation scripts use documented JSON contracts in their module docstrings and in `docs/annotation-guidelines.md` rather than relying on implicit ad hoc inputs.
- All three scripts are CLI-first and return concise operational errors via `SystemExit(str(exc))` at the boundary.
- Quality thresholds are centralized and aligned:
  - review target: IoU > 0.70
  - automated IAA escalation: box IoU < 0.65 or kappa < 0.60
- Visual guidance assets live under `docs/diagrams/` so documentation stays self-contained in the repo.

## Gotchas

- `setup_cvat_projects.py` was only validated for syntax and CLI help in this environment. I could not run it against a live CVAT server because no CVAT endpoint or credentials were provided here.
- The setup script checks `/api/server/about` and `/api/server/annotation/formats`, and expects the deployed CVAT to report a `2.62.x` version prefix unless `--skip-version-check` is used.
- `compute_iaa.py` assumes one normalized JSON file per annotator. There is still no companion export-normalization script in the repo, so the next data task may want to add one if direct CVAT export ingestion is needed.
- `split_dataset.py` behaves well when there are enough temporal sequences to satisfy 70/15/15, but on very small manifests it will preserve chronology first and approximate the ratios second.

## Open questions / limitations

- I did not attempt to create CVAT tasks or import data automatically. This task only established the project schemas and the labeling/QC baseline.
- There is no DVC wiring yet for the split outputs. The script writes manifests and summaries, but dataset versioning still needs a later task.
- The tracking project can encode MOT-compatible labels and attributes, but CVAT “track mode” is still a task-level operator behavior rather than a persistent project-level flag the API can enforce directly.
