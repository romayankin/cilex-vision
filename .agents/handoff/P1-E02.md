## Key decisions

- I treated P1-E02 the same way P1-E01 handled the missing runtime stack: the final recommendation is based on published benchmark data, not on a faked local execution.
- The published benchmark source of truth is the official MOT17 private-detection MOTChallenge leaderboard:
  - ByteTrack: `80.3 MOTA`, `77.3 IDF1`, `2196` ID switches, `2277` fragmentations, `29.6 Hz`
  - BoT-SORT: `80.5 MOTA`, `80.2 IDF1`, `1212` ID switches, `1803` fragmentations, `6.8 Hz`
- Under the protocol’s ordered decision rule, BoT-SORT is a clear proxy winner because it beats ByteTrack on all four ranked metrics. The safe-default fallback (`ByteTrack`) is therefore not triggered for the published proxy report.
- I still kept the report explicit that this is only a proxy:
  - the published tracker submissions use their own private detectors
  - the live bake-off still needs to be rerun later on pilot MOT ground truth using the chosen `YOLOv8-L` detector outputs
- The harness does not fake a local BoT-SORT implementation. Instead, it gives two honest paths for future live runs:
  - feed precomputed MOT txt predictions from a BoT-SORT run
  - supply an external command that writes those predictions

## Patterns established

- Tracker evaluation now follows the same two-mode pattern as detector evaluation:
  - real MLflow-backed harness for future live runs
  - offline JSON-backed proxy comparison for thin environments
- `run_tracker_bakeoff.py` keeps optional dependencies lazy:
  - `motmetrics` is only imported when the script gets past manifest validation
  - the inference-worker ByteTrack modules are imported only if the local ByteTrack path is actually used
- MOT evaluation is class-aware even though the format is MOT-like:
  - GT and prediction rows may carry `class_id` in column 8
  - cross-class matches are blocked by writing `NaN` distances into the MOT accumulator
- The comparison tool is now phase-dispatched rather than detector-hardcoded. Detector behavior stayed intact, and tracker-specific charts/reporting live behind `--phase tracker`.

## Gotchas for the next agent

- `data/eval/mot/manifest.json` still does not exist. `run_tracker_bakeoff.py --tracker bytetrack` fails fast by design with the missing-manifest error.
- The harness expects a manifest, not just raw MOT folders. The intended contract is:
  - top-level `manifest.json`
  - `sequences[]`
  - per sequence:
    - `sequence_id`
    - `sequence_dir`
    - `gt_path`
    - `detections_path` for local ByteTrack runs
    - `image_width`, `image_height`, `frame_rate` if they are not in `seqinfo.ini`
- Local ByteTrack runs assume the detector outputs file contains normalized `[0,1]` `x_min/y_min/x_max/y_max` boxes in the `RawDetection` shape from `services/inference-worker/detector_client.py`.
- BoT-SORT is still not implemented in this repo’s production code. The eval harness therefore cannot execute a real local BoT-SORT run by itself.
- The official tracker leaderboard numbers are pedestrian MOT17 results, not direct pilot-taxonomy metrics. The report calls this out, but the live bake-off still needs pilot data before the recommendation should drive production changes.
- `.claude-task-context.md` is stale in the same way P1-E01 already noted. I still used the prompt file `.agents/prompts/P1-E02.md` as the more precise deliverable check.
