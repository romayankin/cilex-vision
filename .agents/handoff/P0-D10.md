# Handoff: P0-D10 — Triton Model Placement & VRAM Budget

## What was built

- `docs/triton-placement.md` — full spec replacing the stub (13 sections: model inventory, GPU classes, VRAM budget, co-location rules, dynamic batching, preprocessing, server config, lifecycle/shadow deployment, scaling, monitoring, pipeline diagram, acceptance criteria)
- `infra/triton/models/yolov8l/config.pbtxt` — YOLOv8-L detector Triton config (tensorrt_plan, batch≤8, 50ms queue delay)
- `infra/triton/models/osnet/config.pbtxt` — OSNet Re-ID Triton config (tensorrt_plan, batch≤32, 100ms queue delay)
- `infra/triton/models/color_classifier/config.pbtxt` — ResNet-18 color classifier Triton config (tensorrt_plan, batch≤32, 100ms queue delay)
- `infra/prometheus/rules/triton-alerts.yml` — 6 alert rules (VRAM warn/crit, queue delay warn/crit, error rate, model availability)

## Key decisions and why

- **3 models, not 4**: The stub had a separate "clothing classifier" but the taxonomy (P0-D01) defines only 3 color attributes that share the same 10-color vocabulary. A single `color_classifier` model serves all three by receiving different crops. Adding a 4th model later is trivial given the VRAM headroom.
- **FP32 I/O with FP16 compute**: TensorRT engines run FP16 internally but accept/return FP32 tensors. This avoids forcing inference clients to handle FP16 packing while keeping 2x throughput and ~50% VRAM savings.
- **50ms vs 100ms queue delay**: yolov8l gets a tighter 50ms max queue delay because it gates the entire pipeline (directly affects e2e latency NFR). osnet and color_classifier use 100ms because they are downstream and benefit more from larger batches.
- **version_policy latest 2**: Supports shadow deployment (ADR-005) by keeping two versions loadable simultaneously. Even 2x of the largest model (yolov8l) only adds ~185 MB — still <4% of 24 GB.
- **Output tensor shape [11, 8400] for yolov8l**: 4 bbox coordinates + 7 class scores × 8400 anchor points (YOLOv8 grid at 640×640). NMS is the client's responsibility, not built into the TensorRT engine.

## Patterns established that future agents should follow

- **Triton config.pbtxt**: use `platform: "tensorrt_plan"`, `KIND_GPU`, `version_policy { latest { num_versions: 2 } }` for all TensorRT models.
- **Dynamic batching**: latency-critical models (detectors) get lower `max_queue_delay_microseconds` (50ms); batch-efficient models (classifiers, Re-ID) get higher (100ms).
- **Preprocessing is the client's responsibility**: Triton configs define tensor shapes and types; the inference client handles resize, normalization, color space conversion.
- **Alert naming**: `Triton` prefix, `Warn`/`Critical` suffix, `service: triton` label. Matches the `Clock` prefix pattern from P0-D07.
- **Prometheus metric names**: use Triton native metrics (`nv_gpu_*`, `nv_inference_*`) — don't create custom exporters for data Triton already exposes.
- **TensorRT build commands**: always use `--fp16` and dynamic shapes (`--minShapes`, `--optShapes`, `--maxShapes`) matching the config.pbtxt batch range.

## Gotchas

- **ByteTrack is NOT a Triton model.** It runs on CPU (Kalman filter + Hungarian algorithm). Don't try to serve it via Triton.
- **YOLOv8 output format is transposed**: shape is `[batch, 4+num_classes, 8400]`, NOT `[batch, 8400, 4+num_classes]`. The inference client must handle this layout.
- **OSNet version boundary (ADR-008)**: swapping OSNet requires flushing FAISS and resetting all trackers. This is NOT a standard model swap — the inference service must coordinate with the MTMC service.
- **EXPLICIT mode means no models on startup**: the inference service must explicitly call `POST /v2/repository/models/{name}/load` after Triton starts. If it doesn't, Triton serves nothing.
- **Color classifier index order**: the 10 output probabilities correspond to the color enum in docs/taxonomy.md order (red=0, blue=1, ..., unknown=9). The inference client must map indices to enum values — this mapping is not in the Triton config.
- **VRAM numbers are estimates**: actual VRAM depends on the TensorRT engine built for the specific GPU (CUDA compute capability, available kernels). Profile with `nvidia-smi` after deploying real engines.

## Open questions or known limitations

- Exact GPU model is not specified — the spec targets "24 GB GPU" generically. Throughput estimates vary significantly between A5000 (~200 FPS) and RTX 4090 (~350 FPS).
- The color classifier output index-to-enum mapping should be formalized somewhere (either in the ONNX model metadata or a config file). Currently it's only documented in the spec.
- No config.pbtxt for a future pose estimation or license plate model, but the VRAM headroom analysis shows they would fit easily.
- TensorRT engine builds are GPU-specific — an engine built on A5000 won't run on RTX 4090. The ops agent needs a build pipeline per target GPU.
