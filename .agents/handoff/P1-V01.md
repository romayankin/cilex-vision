# Handoff: P1-V01 — Edge Agent

## What was built

- `services/edge-agent/main.py` — entry point: loads YAML config, starts Prometheus metrics server, connects to NATS + MinIO, launches per-camera pipelines, handles SIGINT/SIGTERM graceful shutdown
- `services/edge-agent/camera_pipeline.py` — per-camera orchestrator: RTSP → edge_receive_ts stamp → motion filter → JPEG encode → MinIO upload → FrameRef protobuf → NATS publish (or local buffer on failure)
- `services/edge-agent/motion_detector.py` — frame-differencing motion detector with configurable pixel_threshold, motion_threshold, scene_change_threshold, periodic reference update; BT.601 grayscale conversion
- `services/edge-agent/rtsp_client.py` — GStreamer-backed RTSP client with decodebin, appsink, exponential backoff reconnection (1s → 60s cap), per-camera uptime health scoring
- `services/edge-agent/nats_publisher.py` — NATS JetStream publisher with mTLS support; includes `serialize_frame_ref()` for FrameRef protobuf construction (core_ingest_ts intentionally left unset)
- `services/edge-agent/local_buffer.py` — disk-backed ring buffer using file-per-message format with sequence numbering, FIFO drain with rate limit, crash recovery via directory scan, eviction of oldest messages when over budget
- `services/edge-agent/config.py` — Pydantic Settings v2 with `from_yaml()` class method and env-var overrides (prefix `EDGE_`)
- `services/edge-agent/metrics.py` — 6 Prometheus metrics: camera_uptime_ratio, decode_errors_total, motion_frames_total, static_frames_filtered_total, nats_publish_latency_ms, buffer_fill_bytes
- `services/edge-agent/gen_proto.sh` — generates Python protobuf code from repo's .proto files into `proto_gen/`
- `services/edge-agent/Dockerfile` — python:3.11-slim with GStreamer system packages, PyGObject build deps, proto generation at build time
- `services/edge-agent/requirements.txt` — 10 pinned dependencies (nats-py, protobuf, grpcio-tools, pydantic, pyyaml, prometheus-client, numpy, minio, Pillow, pydantic-settings)
- `services/edge-agent/tests/test_motion_detector.py` — 10 tests: first frame, identical, small/moderate/large change, scene change reset, pixel threshold, grayscale conversion
- `services/edge-agent/tests/test_rtsp_client.py` — 6 tests: backoff doubles, backoff caps at 60s, backoff resets, uptime starts zero, uptime after connect, start failure raises
- `services/edge-agent/tests/test_local_buffer.py` — 7 tests: enqueue creates files, drain FIFO order, drain stops on failure, empty drain, eviction, crash recovery, message codec roundtrip
- `services/edge-agent/README.md` — usage, config reference, metrics table, test instructions

## Key decisions and why

- **edge_receive_ts stamped IMMEDIATELY after `read_frame()`, BEFORE motion detection** — per time-sync-policy.md and P0-D07 handoff, this is the trust anchor for cross-camera ordering. Any processing delay after stamping is acceptable.
- **core_ingest_ts left unset in FrameRef** — per time-sync-policy.md §2, this field is "set when the core tier first accepts the message" (i.e., the ingress bridge). The edge agent MUST NOT populate it.
- **source_capture_ts falls back to edge_receive_ts** — GStreamer RTSP PTS values are stream-relative, not wall-clock. Without RTCP SR correlation, we can't derive an absolute camera timestamp. Marked as CLOCK_QUALITY_ESTIMATED (value 3).
- **NATS subjects follow security-design.md**: `frames.live.{site_id}.{camera_id}` and `frames.replay.{site_id}.{camera_id}` — NOT the draft format `site.{site_id}.frames` from the task prompt.
- **Backoff NOT reset in `start()`** — moved to explicit `reset_backoff()` called by CameraPipeline after a successful direct connect. This preserves backoff accumulation across reconnect_with_backoff() cycles while still resetting when the connection becomes stable.
- **File-per-message buffer format** — simpler than segment-file WAL for the pilot scale (4 cameras × ~6 frames/s pass-through). Crash recovery via `_scan_existing()` is trivial. The format uses a compact binary header (6 bytes) + subject + payload.
- **GStreamer imports guarded by lazy import in rtsp_client.py** — allows unit tests to run without system GStreamer packages by stubbing `gi` at the module level.
- **Pillow for JPEG encoding** — lighter than OpenCV, sufficient for the encode-and-upload path. The JPEG quality is 85 (good compression ratio for surveillance frames).
- **Flat module layout (no Python package)** — files live directly in `services/edge-agent/` with plain imports, matching the CONVENTIONS.md service structure template. Tests use `conftest.py` to add the service root to `sys.path`.

## Patterns established that future agents should follow

- **Flat service layout**: `services/{name}/main.py`, `config.py`, `metrics.py`, etc. — no nested `__init__.py` packages. Tests in `tests/` with `conftest.py` adding parent to `sys.path`.
- **Config pattern**: `Settings.from_yaml(path)` class method on Pydantic BaseSettings, with `env_prefix="EDGE_"` for environment overrides. Nested models for subsystems (NatsConfig, MinioConfig, etc.).
- **Metrics pattern**: dedicated `metrics.py` file defining all Prometheus instruments. Service prefix `edge_` on all metric names. Histogram buckets for latency, counters for errors, gauges for state.
- **Proto usage**: `gen_proto.sh` generates into `proto_gen/`; `sys.path.insert(0, "proto_gen")` in the consuming module. Lazy imports inside functions to avoid import-time failure if protos aren't generated.
- **GStreamer mocking in tests**: stub `gi` module tree with `types.ModuleType` and `MagicMock` before importing the module under test. This avoids requiring system GStreamer packages in CI.
- **Async + threading bridge**: GStreamer blocking calls wrapped in `asyncio.to_thread()` (e.g., `appsink.try_pull_sample`). MinIO synchronous client also wrapped in `to_thread()`.

## Gotchas

- **`proto_gen/` is generated, not committed** — developers must run `bash gen_proto.sh` before running the service locally. The Dockerfile handles this automatically during build. Tests mock the serialization layer so they don't need generated protos.
- **GStreamer system packages are required at runtime** — `python3-gi`, `gir1.2-gstreamer-1.0`, and plugin packages must be installed. They come from apt, not pip. The Dockerfile installs them along with build deps for PyGObject.
- **Pillow not in the task's dependency list** — added as a practical necessity for JPEG encoding. If a lighter alternative is preferred, the `_upload_frame` method is the only consumer.
- **`decodebin` handles codec negotiation** — the pipeline uses `decodebin` instead of explicit `rtph264depay ! avdec_h264` for codec-agnostic support (H.264, H.265, MJPEG). This introduces a dynamic pad linking step that GStreamer handles internally.
- **MinIO is a runtime dependency** — if MinIO is unreachable, frames are dropped (logged as warning). The local buffer only handles NATS outages, not MinIO outages. This is acceptable for the pilot but may need enhancement for production.
- **NATS reconnection is handled by nats-py library** — the client has `max_reconnect_attempts=-1` (infinite retries) with 2-second wait between attempts. The NatsPublisher.publish() catches exceptions and returns False, triggering local buffering.

## Open questions or known limitations

- **Frame deletion from MinIO on buffer replay**: if a FrameRef was buffered and later replayed, the corresponding JPEG in MinIO is already uploaded. This means frames uploaded during NATS outages accumulate without downstream processing until replay completes.
- **No per-camera motion sensitivity**: all cameras share the same MotionConfig. A future enhancement could allow per-camera motion tuning in the YAML config.
- **source_capture_ts always equals edge_receive_ts**: until RTCP SR timestamp extraction is implemented, the camera clock is effectively unused. Clock quality is always ESTIMATED.
- **No MinIO connection pool or retry**: the synchronous Minio client is used via asyncio.to_thread. For higher throughput, consider aiobotocore or connection pooling.
- **Buffer shared across cameras**: all cameras write to the same local buffer directory. Under extreme load, file I/O contention could be a bottleneck. For the pilot (4 cameras), this is a non-issue.
