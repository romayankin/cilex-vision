# Handoff: P1-V03 — Central Decode & Frame Sampling Service

## What was built

- `services/decode-service/main.py` — Kafka consumer pipeline: consume FrameRef from `frames.sampled.refs`, download encoded frame from MinIO, decode → color-normalize → resize → sample → re-encode as JPEG → upload to `decoded-frames` bucket → publish FrameRef to `frames.decoded.refs`. Graceful shutdown via SIGINT/SIGTERM.
- `services/decode-service/decoder.py` — Codec-agnostic frame decoder: Pillow for JPEG (direct, no GStreamer needed), GStreamer pipeline (`appsrc → decodebin → videoconvert → videoscale → appsink`) for H.264/H.265, raw RGB passthrough. Lazy GStreamer imports following edge-agent pattern.
- `services/decode-service/color_space.py` — BT.601/BT.709 YCbCr→RGB conversion with numpy matrices. Detection priority: explicit GStreamer colorimetry metadata → codec hint (JPEG = BT.601) → resolution heuristic (≤720px width = BT.601, else BT.709) → configurable default. `ensure_rgb()` is the main entry point.
- `services/decode-service/sampler.py` — Per-camera FPS-based frame sampler. Tracks last-accepted timestamp per camera, skips frames arriving faster than `target_fps` (default 5.0). Cameras are sampled independently.
- `services/decode-service/publisher.py` — Kafka producer for decoded FrameRef messages to `frames.decoded.refs`. Uses `acks=all`, `enable_idempotence=True`, `compression_type=zstd`. Includes `x-proto-schema` header.
- `services/decode-service/config.py` — Pydantic Settings v2 with `from_yaml()` and `DECODE_` env prefix; nested configs for Kafka, MinIO, Decode (output_width/height, jpeg_quality, default_color_space), Sampler (target_fps).
- `services/decode-service/metrics.py` — 9 Prometheus metrics: decode_errors_total{codec}, frames_decoded_total, frames_sampled_total, frames_skipped_total, decode_latency_ms, frames_consumed_total, publish_errors_total, consumer_lag{topic,partition}, color_space_conversions_total{from_space,to_space}.
- `services/decode-service/gen_proto.sh` — Proto generation script (Docker + local paths).
- `services/decode-service/Dockerfile` — python:3.11-slim with GStreamer system packages (python3-gi, gir1.2-gstreamer-1.0, gstreamer1.0-plugins-*).
- `services/decode-service/requirements.txt` — 10 dependencies (no scipy or tritonclient — this service doesn't run inference).
- `services/decode-service/README.md` — Usage, config table, running, testing.
- `services/decode-service/tests/conftest.py` — sys.path setup.
- `services/decode-service/tests/test_color_space.py` — 22 tests: detection heuristic (metadata override, codec hint, resolution boundary, RGB/raw passthrough), YCbCr→RGB conversion (white/black/coloured, BT.601 vs BT.709 divergence, clipping, shape), ensure_rgb integration.
- `services/decode-service/tests/test_decoder.py` — 16 tests: JPEG decode (shape, resize, color preservation), FrameDecoder dispatch (jpeg/mjpeg/raw/h264/h265), GStreamer mock verification, error metrics, caps string generation.
- `services/decode-service/tests/test_sampler.py` — 13 tests: first-frame acceptance, interval enforcement, camera independence, reset, edge cases (1 FPS, 30 FPS, invalid), steady-state 30fps→5fps simulation.

## Key decisions and why

- **Pillow for JPEG, GStreamer for H.264/H.265**: JPEG single-frame decode doesn't benefit from GStreamer's pipeline model and Pillow is universally available. GStreamer is only needed for codec-stream decode (H.264/H.265 NAL units). This also means JPEG decode works in tests without GStreamer stubs.
- **Separate output topic `frames.decoded.refs`**: avoids consumer loop risk on `frames.sampled.refs` and cleanly separates raw-from-edge vs decoded-for-inference frames. The inference worker's `input_topic` should be reconfigured to consume from `frames.decoded.refs` when this service is deployed.
- **Color space detection before conversion**: the priority-based heuristic (metadata → codec → resolution → default) matches ITU-R conventions. Applying the wrong BT.601/BT.709 matrix shifts hue by 3–8% in R/B channels — this is the single most common source of color-based model regression.
- **FPS sampling in the decode service, not the consumer**: sampling happens AFTER FrameRef deserialization but BEFORE MinIO download, to avoid wasting bandwidth downloading frames that will be skipped.
- **Re-encode as JPEG after decode**: the downstream inference worker expects JPEG-encoded frames in MinIO (per P1-V04 pattern). Re-encoding at configurable quality (default 90) ensures consistent input.
- **GStreamer lazy imports**: same pattern as `edge-agent/rtsp_client.py`. Imports `gi` at function-call time, not module level. This lets tests run without system GObject-Introspection packages.
- **One-shot GStreamer pipeline for single frames**: unlike the RTSP client which maintains a persistent pipeline, the decoder creates and tears down a pipeline per frame. This is appropriate because frames arrive as discrete encoded blobs from MinIO, not as a continuous stream.

## Patterns established

- **GStreamer test stub**: `types.ModuleType` + `MagicMock` for `gi`, `gi.repository`, `gi.repository.Gst`, `gi.repository.GstApp` — same pattern as edge-agent tests.
- **Codec dispatch in FrameDecoder**: `decode()` routes by codec string to the appropriate decoder function. New codecs can be added by extending the if/elif chain.
- **Color space safety net**: even when GStreamer's `videoconvert` handles YCbCr→RGB, `ensure_rgb()` is called as a secondary check. For JPEG decoded by Pillow, this is a no-op (Pillow decodes directly to RGB).
- **Sampler state per camera**: `_last_accepted` dict keyed by `camera_id`. Each camera's FPS budget is independent. `reset()` clears state for a single camera on reconnect.

## Gotchas

- **GStreamer system packages required at runtime**: the Dockerfile installs `python3-gi`, `gir1.2-gstreamer-1.0`, and `gstreamer1.0-plugins-*`. These are NOT pip packages — they must be installed via apt. Without them, H.264/H.265 decode will fail (JPEG still works via Pillow).
- **`frames.decoded.refs` topic not in `infra/kafka/topics.yaml`**: this new topic needs to be added to the topic definitions. Partition key is `camera_id`, retention should match `frames.sampled.refs`.
- **Inference worker reconfiguration needed**: when the decode service is deployed, the inference worker's `input_topic` must change from `frames.sampled.refs` to `frames.decoded.refs`. Otherwise it will still consume raw frames and decode them internally (redundant work).
- **BT.601 is the safe default**: when metadata and resolution heuristics are ambiguous, BT.601 is used because JPEG (the most common input codec from edge agents) is BT.601 by JFIF convention. Setting a wrong default silently degrades model accuracy.
- **GStreamer pipeline lifetime**: each `decode_gstreamer()` call creates and destroys a pipeline. For high-throughput scenarios, a persistent pipeline with `appsrc` feeding multiple frames would be more efficient, but adds complexity for state management and error recovery.
- **No dead-letter queue**: failed messages are logged but not routed to a DLQ. Per kafka-contract.md, DLQ after 3 retries should be added when infrastructure is available.

## Open questions or known limitations

- **No H.264/H.265 integration tests**: GStreamer is fully mocked in tests. Real decode testing requires GStreamer system packages and test fixtures (NAL-unit encoded frames). Consider adding a `tests/integration/` directory with real encoded frames for CI environments that have GStreamer installed.
- **No batched decode**: frames are decoded one at a time. For GPU-accelerated decode (e.g., `nvh264dec`), batching multiple frames through a persistent pipeline would be more efficient.
- **core_ingest_ts set in decode service**: this timestamp is stamped after decode completes. If the inference worker is expected to stamp `core_ingest_ts`, there may be ambiguity about which service "ingested" first. The decode service is the first core service to touch the data, so stamping here is appropriate.
- **Memory**: decoded frames (1280×720×3 = ~2.7 MB each) are held in memory during the upload step. With `max_poll_records=10` and 4 workers, peak memory could reach ~108 MB of frame data. This is within reasonable bounds.
