# P4-O01 — Multi-Site Infrastructure

## What I built

- `infra/terraform/modules/site/*`
  - per-site infrastructure module for edge gateways, site-local storage volumes, site-local network controls, and a WireGuard/VPN endpoint placeholder
  - outputs edge hostnames/IPs, site storage endpoint, and site VPN endpoint
- `infra/terraform/modules/central/*`
  - central multi-site module that composes the existing compute/network/storage modules for Kafka, TimescaleDB, MinIO, GPU inference, monitoring, and central service nodes
  - outputs core network IDs/CIDRs, role hostnames/IPs, storage IDs, and a central VPN endpoint placeholder
- `infra/terraform/environments/multi-site/main.tf`
  - multi-site environment wiring that instantiates one central stack and one site module per site via `for_each`
  - includes a `null_resource` peering map to model site-to-central WireGuard wiring
- `infra/ansible/playbooks/add-site.yml`
  - idempotent site onboarding playbook
  - bootstraps per-site PKI when needed, deploys site-local NATS, deploys edge hosts via the new `site-edge` role, provisions site-specific Kafka SCRAM + ACLs, upserts site/camera rows in PostgreSQL, reloads Prometheus, and runs site-scoped smoke checks
- `infra/ansible/playbooks/remove-site.yml`
  - site decommission playbook
  - marks cameras as `maintenance`, waits for active local tracks to drain, removes topology edges, removes Kafka ACLs, archives site-keyed object prefixes into `archive-warm`, reloads Prometheus with site-host exclusions, attempts certificate revocation with `step`, and removes edge/NATS containers
- `infra/ansible/roles/site-edge/tasks/main.yml`
  - new per-site edge role
  - installs GStreamer and WireGuard tools, optionally configures NVIDIA runtime, renders site-specific edge-agent config, deploys local MinIO for frame buffering, deploys node-exporter, builds the edge-agent image from repo root, and smoke-tests the edge node
- `infra/ansible/inventory/multi-site.yml`
  - multi-site inventory template with:
    - central groups (`kafka`, `timescaledb`, `minio`, `triton`, `monitoring`, `services`, `mlflow`, `cvat`)
    - per-site umbrella groups (`site_alpha`, `site_beta`)
    - `edge_gateways` and `nats` groups that stay compatible with the current playbooks
- `scripts/deploy/site-onboarding.sh`
  - end-to-end site onboarding wrapper
  - validates inputs, generates a site inventory fragment, bootstraps PKI, runs `add-site.yml`, and prints an onboarding summary

## Key decisions

- **Kept multi-site additive to the existing pilot and production paths**
  - I did not rewrite `deploy-all.yml`, `deploy-multi-node.yml`, or the existing inventories.
  - The new site lifecycle automation sits beside the existing single-site and multi-node tooling.

- **Separated operational site IDs from database site IDs**
  - Infra, PKI, hostnames, and metrics still use a human-readable site slug such as `alpha`.
  - PostgreSQL topology registration uses a separate `site_db_id` UUID because `sites.site_id` is a UUID column.
  - This avoids repeating the `site_id` type mismatch that already shows up elsewhere in the repo.

- **Made site onboarding fragment-based instead of mutating the canonical inventory**
  - `site-onboarding.sh` generates a separate inventory fragment and passes it together with `inventory/multi-site.yml`.
  - That keeps the base inventory template stable and lets operators review site-specific data before merging it into a long-lived inventory source.

- **Used the same edge host for NATS when the operator does not provide a separate NATS host**
  - The task requires only `--edge-host`.
  - The onboarding wrapper defaults `--nats-host` to the same machine so the flow stays usable without extra inputs.

- **Provisioned site-specific Kafka credentials and exact ACLs**
  - The add-site playbook creates or reuses a per-site SCRAM password on the first Kafka host and grants exact-topic `Write`/`Describe` access to `frames.sampled.refs`.
  - This stays aligned with the repo’s “exact ACL, no wildcard topic grants” rule.

- **Used Prometheus target exclusion instead of editing the inventory during decommission**
  - `remove-site.yml` re-renders Prometheus with `monitoring_excluded_hosts` so the site can disappear from monitoring immediately even before an operator cleans up inventory files.

## Patterns followed

- reused the existing Ansible structure:
  - `community.docker`
  - `group_vars/all.yml`
  - `repo_root`
  - `common` role
  - host-networked Docker containers
  - post-start smoke checks
- kept Terraform composition consistent with `P3-O01` by building new higher-level modules on top of the existing `compute`, `network`, and `storage` modules instead of duplicating provider resources everywhere
- kept site-local monitoring additive by extending the shared Prometheus template with an optional exclusion list rather than introducing a second monitoring template

## Gotchas

- **`site_id` vs `site_db_id` matters**
  - `site_id` is the short slug used for PKI/NATS/hostnames.
  - `site_db_id` must be a UUID string for the `sites` / `cameras` tables.
  - Do not feed the slug into topology SQL.

- **`site-onboarding.sh` generates placeholder cameras disabled by default**
  - The task inputs do not include real camera RTSP URLs or zone layouts.
  - The generated fragment uses `enabled: false` and placeholder RTSP values so onboarding does not immediately create broken live streams.

- **Monitoring exclusion is runtime-only**
  - `remove-site.yml` removes scrape targets from the deployed Prometheus config, but it does not delete the site from inventory source files.
  - Operators still need to remove or archive the inventory fragment as part of permanent decommissioning.

- **Site archival is best-effort against current object-key conventions**
  - The remove playbook mirrors camera/site prefixes from buckets where the repo already uses camera- or site-based paths.
  - Buckets or objects that are not keyed by camera/site cannot be perfectly archived by prefix today.

- **Certificate revocation assumes `step` is installed and already bootstrapped**
  - The remove playbook attempts `step ca revoke --cert ... --key ...` only when `step` is present.
  - If the operator workstation or control node lacks `step`, decommission still continues and cert revocation becomes a manual follow-up.
