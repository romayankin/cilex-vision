# P2-V02 — MTMC Re-ID Association Service

## What Was Built

A stateful Python service at `services/mtmc-service/` that performs cross-camera Re-ID matching using FAISS, topology-constrained candidate filtering, and checkpoint/restore for fault tolerance.

### Files Created
- `main.py` — Kafka consumer (confluent-kafka, cooperative-sticky assignor), lifecycle management, periodic FAISS cleanup
- `matcher.py` — 5-stage scoring pipeline: topology filter → FAISS search → version/class filter → transit-time likelihood → combined scoring with colour attribute check. Greedy assignment (<10 candidates) or Hungarian (>=10)
- `faiss_index.py` — Thread-safe FAISS IndexIDMap wrapping IndexFlatIP (inner product on L2-normalised = cosine). Metadata dict, reverse maps for embedding_id and local_track_id. Active horizon cleanup.
- `checkpoint.py` — Pickle-based snapshot/restore. Local (60s) + MinIO (5min) intervals. Atomic local writes via tmp+rename. Restore: MinIO → local → empty.
- `topology_client.py` — Loads TopologyGraph from DB (cameras + topology_edges tables), caches with configurable refresh. Candidate camera set with 30s TTL cache. Edge distribution lookup for transit-time scoring.
- `publisher.py` — asyncpg DB writer: create_global_track, create_global_track_link, update_global_track_last_seen, batch_create_links (COPY), get_local_track_info, find_existing_global_track, get_track_colors.
- `config.py` — Pydantic Settings v2 with YAML loading and MTMC_ env prefix.
- `metrics.py` — 8 Prometheus instruments (matches, rejects, score histogram, index size, checkpoint size/lag, rebalance duration, embeddings consumed).
- `gen_proto.sh` — Proto generation following inference-worker pattern.
- `Dockerfile` — python:3.11-slim, proto gen at build time, EXPOSE 8080.
- `requirements.txt` — confluent-kafka, faiss-cpu, scipy, asyncpg, minio, prometheus-client, pydantic, pyyaml, protobuf, grpcio-tools, numpy.
- `tests/conftest.py` + `tests/helpers.py` — Shared fixtures and vector generation helpers.
- `tests/test_matcher.py` — 8 tests: identity matching, version boundary, class consistency, topology rejection, best-of-many, colour bonus, 10-embedding pair test.
- `tests/test_checkpoint.py` — 5 tests: snapshot/restore cycle, empty restore, removal persistence, metadata fields, tombstone persistence.

## Key Decisions

1. **confluent-kafka over aiokafka**: Conventions specify confluent-kafka. cooperative-sticky partition assignor is natively supported. Blocking `poll()` wrapped in `asyncio.to_thread()` to integrate with the async event loop.

2. **Track info from DB**: The Embedding proto doesn't carry camera_id or object_class. These are resolved from the local_tracks table and cached in the matcher. This avoids modifying the upstream inference-worker.

3. **Topology fallback models**: topology_client.py includes inline Pydantic models that mirror `services/topology/models.py` as a fallback. This avoids a hard import dependency on the topology service while preserving identical BFS logic. A `topology_models.py` symlink/copy is the intended production path.

4. **Thread-safe FAISS**: All index operations protected by a threading.Lock since confluent-kafka's poll runs in a thread via to_thread. Metadata maps (embedding_id→faiss_id, track_id→faiss_id) maintained alongside the index for O(1) removal.

5. **Transit-time scoring**: Gaussian approximation around p50 with sigma derived from p90. Sharp exponential penalty beyond p99. Returns 0.5 (neutral) for unknown topology edges.

6. **Assignment strategy**: Greedy (max score) for < 10 candidates above threshold. scipy Hungarian for >= 10. In practice with K=20 and filtering, most calls will be greedy.

## Patterns Established

- confluent-kafka consumer with cooperative-sticky in async context (to_thread wrapping)
- FAISS IndexIDMap + metadata dict pattern for filtered vector search
- Pickle checkpoint with atomic local write (tmp + rename) + MinIO remote backup
- Topology graph loaded from DB with TTL cache for candidate sets

## Gotchas

1. **Topology models**: The service includes fallback inline topology models. If `services/topology/models.py` changes its API, the fallback code must be updated. Consider publishing topology models as a shared package.

2. **UUID parsing**: publisher.py's `get_local_track_info` passes the local_track_id as a string to asyncpg. The DB column is UUID type — asyncpg handles the conversion, but the caller must pass valid UUID strings.

3. **Checkpoint pickle safety**: We use pickle for FAISS index serialization (faiss's own serialize is less convenient with metadata). The `# noqa: S301` suppresses the security warning — this is acceptable since checkpoints are written and read by the same service.

4. **FAISS remove_ids**: Calling remove_ids on IndexIDMap with an ID that doesn't exist is a no-op (no error), but the metadata maps must be cleaned up independently.

5. **Active horizon cleanup**: Runs every 60s in a background asyncio task. Embeddings older than `active_horizon_minutes` are purged from FAISS. The cleanup and consumer share the same lock.

## Open Questions

1. **site_id sharding**: Currently site_id is a config value. For multi-site deployments, each instance should consume only partitions for its site. This requires Kafka partition assignment coordination (e.g., partition-to-site mapping in config).

2. **Topology refresh under load**: Refreshing topology requires DB queries that could add latency during heavy matching. The 5min default is conservative; may need tuning per deployment.

3. **pgvector handoff**: After the 30-minute active horizon, embeddings are dropped from FAISS. The spec mentions pgvector for 90-day historical search (ADR-004) — that integration is not yet built.
