# Handoff: P1-V05 — Metadata Bulk Collector

## What was built

- `services/bulk-collector/main.py` implements the service runtime:
  - one Kafka consumer loop per configured topic/group binding
  - protobuf decoding with optional Schema Registry deserializer and raw-protobuf fallback
  - batch buffering by message count / age
  - offset commits only after the corresponding COPY flush succeeds
  - `/health`, `/ready`, and `/metrics`
- `services/bulk-collector/collector.py` batches whole Kafka envelopes instead of raw rows so commit safety is preserved.
- `services/bulk-collector/writer.py` provides the asyncpg COPY writer for:
  - `detections`
  - `track_observations`
  with bounded TTL dedup caches keyed by `(camera_id, frame_seq, local_track_id)`.
- `services/bulk-collector/config.py` and `services/bulk-collector/metrics.py` follow the same YAML + env override / Prometheus layout used by the ingress bridge.
- `services/bulk-collector/gen_proto.sh`, `requirements.txt`, and `Dockerfile` complete the runtime packaging.
- Tests:
  - `test_collector.py` — batch size / max-age / requeue behavior
  - `test_writer.py` — COPY path + dedup TTL behavior
  - `test_main.py` — detection message mapping, bbox `x_min/y_min/x_max/y_max -> x/y/w/h`, threshold rejection
- `scripts/load-test/load-test-collector.py` generates synthetic 100-camera-equivalent detection traffic and asserts no row loss through the batch/writer path.

## Key decisions

- **Batching is commit-safe.**
  - I initially staged rows by table, but that would have allowed Kafka offsets to advance before a time-based flush wrote the rows.
  - The final collector batches full Kafka envelopes and only commits offsets after both COPY targets for that envelope succeed.
- **Storage time uses `edge_receive_ts` first.**
  - This matches `docs/time-sync-policy.md`, where `edge_receive_ts` is the authoritative cross-camera time axis.
  - Fallback order is `core_ingest_ts`, then `source_capture_ts`.
- **Detection -> DB mapping relies on Kafka headers for the fields missing from `Detection` proto.**
  - Required header: `x-frame-seq`
  - Optional headers: `x-local-track-id`, `x-embedding-ref`
  - `x-proto-schema` is used for schema routing and validation.
- **COPY only on the write path.**
  - The writer calls `copy_records_to_table()` directly for the hypertables and does not issue row-by-row inserts.

## Patterns established

- Flat service layout matching `edge-agent` and `ingress-bridge`.
- Lazy imports for optional runtime deps (`aiokafka`, `asyncpg`, `confluent-kafka`, `fastapi`, `uvicorn`) so CLI help and unit tests work in a thin environment.
- Metrics use the `bulk_` prefix:
  - `bulk_rows_written_total`
  - `bulk_batch_size_histogram`
  - `bulk_write_latency_ms`
  - `bulk_write_errors_total`
  - `bulk_consumer_lag`

## Gotchas

- There is still a repo-level contract gap:
  - `docs/adr/ADR-003-database-schema.md` and the P1-V05 prompt clearly expect a high-volume detection stream.
  - `infra/kafka/topics.yaml` does **not** define a canonical detections topic yet.
  - `P1-V04` says the inference worker will publish “bulk detections to bulk-collector”, but the topic / schema for that stream is not formalized.
  - I left the service configurable and defaulted it to `bulk.detections` so the collector can be wired immediately once that producer exists.
- `tracklets.local` alone is not sufficient to populate `track_observations` accurately:
  - the current `Tracklet` proto does not carry `frame_seq` or `bbox_area`
  - those fields are required for the dedup key and DB row shape
- The requested handoff file `.agents/handoff/P0-D03.md` does not exist in this repo. I used `docs/kafka-contract.md` directly instead.
- Because the dedup key is enforced in-memory, duplicate protection is bounded by TTL / cache size and does not survive a process restart. A persistent idempotency key path will need either:
  - a canonical detections topic + unique DB key strategy, or
  - a staging / merge design once the upstream contract is finalized.
