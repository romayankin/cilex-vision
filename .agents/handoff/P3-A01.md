# P3-A01 — Continuous Annotation Pipeline

## What I built

- `scripts/annotation/hard_example_miner.py`
  - queries low-confidence detections from TimescaleDB with `asyncpg`
  - scans recent MinIO debug traces and extracts exportable low-confidence / optional shadow-disagreement examples
  - downloads frame assets from MinIO, writes optional crops when Pillow is available, and emits `data/hard-examples/manifest.json`
- `scripts/annotation/auto_create_cvat_tasks.py`
  - reuses the existing CVAT REST helpers from `setup_cvat_projects.py`
  - ensures a `hard-examples` project exists with the canonical 7 detector labels
  - groups mined examples into tasks, uploads frames, and seeds task annotations with model-predicted rectangles
- `scripts/annotation/annotation_feedback_loop.py`
  - exports completed `hard-examples` tasks in `CVAT for images 1.1`
  - compares final annotations against the manifest predictions with IoU matching
  - writes JSON + Markdown feedback reports and a retraining-ready manifest at `data/training/raw/feedback-additions.json`
- `infra/cron/hard-example-mining-cron.yml`
  - daily timer / cron artifact for the mining step

## Key decisions

- **Trace-driven assets, DB-driven accounting.**
  The miner always queries the `detections` table so operators can see the true low-confidence volume in TimescaleDB, but actual export examples come from debug traces because those are the only records that carry recoverable frame URIs today.

- **Manifest is the contract between all three scripts.**
  The miner writes frame paths, timestamps, predicted boxes, and selection reasons; the CVAT task creator and feedback exporter both consume that exact structure instead of inventing separate task-side state.

- **Pre-annotations go straight into task annotations.**
  Rather than relying on a separate dataset-import format for model suggestions, `auto_create_cvat_tasks.py` uploads media and then replaces task annotations through `/api/tasks/{id}/annotations/` with predicted rectangles. That keeps the workflow simple and auditable.

- **Feedback output is image-centric.**
  The retraining manifest groups final annotations per exported image, which matches the loose `train.py` manifest contract from `P3-V01` and preserves hard-negative images where the final annotation set is empty.

## Patterns followed

- CLI-first scripts with concise `SystemExit(str(exc))` boundaries
- `urllib.request` for all CVAT HTTP interactions
- `asyncpg` for database access in the miner
- lazy / optional runtime dependencies where appropriate (`asyncpg`, `minio`, `Pillow`)
- repo-local JSON artifacts under `data/` and operator cron artifacts under `infra/cron/`

## Gotchas

- The miner can only export examples for detections that also have a recoverable debug trace with a usable `frame_uri`. Low-confidence detections without a sampled trace are counted in the manifest summary but cannot yet be pushed into CVAT automatically.
- `auto_create_cvat_tasks.py` was validated for lint / compile / CLI help only in this environment. The multipart media upload and annotation replacement paths target the documented CVAT task APIs, but they were not exercised against a live server here.
- The feedback loop only processes tasks whose CVAT task status is exactly `completed`. If operators rely on another terminal workflow state, the filter will need adjustment.
- The new cron file is only a committed operator artifact. No playbook installs or enables it yet.

## Open questions
