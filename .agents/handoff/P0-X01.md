# Handoff: P0-X01 — Parametric Cost Model

## What was built

- `scripts/cost-model/params.yaml`
  - Extended the existing calibration file instead of replacing it.
  - The pre-existing top-level calibration keys remain intact:
    - `version`
    - `updated_at`
    - `defaults`
    - `cameras`
  - Added a new `cost_model` section with:
    - camera count defaults (`4/10/100`)
    - motion duty cycle scenarios (`P25/P50/P90`)
    - workload assumptions (bitrate, inference FPS, detections/frame, active tracks, event clips)
    - GPU assumptions (`gpu_cost_monthly_usd`, `cameras_per_gpu`, headroom factor)
    - retention periods
    - storage $/GB-month assumptions
    - TimescaleDB storage assumptions
    - Kafka average message sizes
    - fixed infrastructure service unit costs
  - Every assumed parameter is explicitly marked with `REPLACE WITH MEASURED` in its `note` field.

- `scripts/cost-model/cost_model.py`
  - Standalone CLI runnable as:
    - `python3 scripts/cost-model/cost_model.py`
  - Reads:
    - `scripts/cost-model/params.yaml`
    - `infra/kafka/topics.yaml`
    - `infra/docker-compose.yml`
  - Computes per-scenario outputs for `4`, `10`, and `100` cameras across `P25`, `P50`, and `P90` motion duty cycles.
  - Prints two formatted stdout tables per scenario:
    - throughput/storage summary
    - monthly cost breakdown
  - Generates Excel workbook output at:
    - `artifacts/cost-model/cost-model.xlsx`

- Tests:
  - `scripts/cost-model/tests/test_cost_model.py`
    - verifies the YAML merge contract
    - verifies topic/partition inventory from `infra/kafka/topics.yaml`
    - verifies the `P25/P50/P90` × `4/10/100` matrix
    - verifies stdout report rendering
    - verifies XLSX workbook creation

## Modeling choices

- **Calibration contract preserved**
  - The existing `params.yaml` layout from P1-E03 is treated as authoritative.
  - The new script only reads the calibration keys and the appended `cost_model` block; it does not require any schema change to the calibration code.

- **Motion duty cycle is the main scenario lever**
  - `raw/day GB` is calculated from full source bitrate.
  - `central/day GB` and central inference FPS are reduced by the motion duty cycle.
  - This keeps the output aligned with the edge-filter work from P1-V01 and P1-E03.

- **GPU sizing is camera-equivalent based**
  - `cameras_per_gpu = 32` is the default assumption, chosen to stay consistent with the Triton placement doc’s 24 GB GPU scaling guidance and to remain conservative for the pilot.
  - GPU count is:
    - `ceil(cameras * duty_cycle * gpu_headroom_factor / cameras_per_gpu)`
    - with a floor of `1` when cameras > 0

- **Kafka storage references the canonical topic catalog**
  - Topic retention, partition count, and replication factor are taken from `infra/kafka/topics.yaml`, not duplicated in Python constants.
  - The script requires an average message-size assumption for every topic and fails fast if the YAML is missing one.

- **TimescaleDB storage is modeled as hot + compressed**
  - Daily detection and track-observation volume is estimated from:
    - `central_inference_fps`
    - `detections_per_frame`
    - `active_tracks_per_camera`
  - Retention/compression math follows the ADR-003 pattern:
    - hot window before compression
    - compressed tail after `timeseries_compress_after`
    - multiplied by an index/metadata overhead factor

- **Infrastructure fixed cost mirrors docker-compose services**
  - The script parses `infra/docker-compose.yml` and maps service counts into billing groups:
    - `kafka_broker`
    - `kafka_ui`
    - `nats`
    - `timescaledb`
    - `minio`
    - `redis`
    - `prometheus`
    - `grafana`
    - `mlflow`
  - Kafka brokers are counted dynamically from `kafka-0/1/2`.

## Gotchas for the next agent

- **The current numbers are placeholders by design**
  - All numeric defaults in `cost_model` are assumptions and explicitly marked `REPLACE WITH MEASURED`.
  - The script validates those markers so future edits cannot silently convert assumed defaults into undocumented values.

- **The storage model is intentionally approximate**
  - `bitrate_mbps` is used as a proxy for source footage/day and for event clip size.
  - The current pipeline actually stores JPEG frame blobs rather than continuous raw video segments, so the hot object-store estimate is a planning approximation, not a measured byte-accurate forecast.

- **Kafka traffic formulas are topic-specific but still assumed**
  - Topic retention comes from `infra/kafka/topics.yaml`, but per-topic message volume still depends on assumed workload conversion factors:
    - `attribute_messages_per_detection`
    - `embedding_updates_per_active_track_hour`
    - `events_per_active_track_hour`

- **Excel generation depends on openpyxl**
  - The script imports `openpyxl` lazily and supports `--skip-xlsx` for thin environments.
  - During this task I installed `openpyxl` into the local user environment with:
    - `python3 -m pip install --user --break-system-packages openpyxl`
  - If a future environment lacks it, the script will fail with a clear install hint unless `--skip-xlsx` is used.

- **`artifacts/cost-model/` is generated output**
  - The workbook path is under `artifacts/` and is not required by the prompt as a committed deliverable.
  - Re-running the script overwrites the workbook in place.
